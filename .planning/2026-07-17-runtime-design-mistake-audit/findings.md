# Findings: Remaining Runtime Design-Mistake Audit

## Baseline
- The legacy root plan shows Phase 71's prior architecture audit complete and Phase 72 remediation still in progress.
- The remediation has already replaced the prior event-bus lifecycle, manual scene injection, required-service fallbacks, fixed building-state persistence, FacilityFunction authority, closed work/need/role/stock taxonomies, free-text character activity routing, fixed run/meta effects, concrete AI-action dispatch, defense-effect dispatch, blueprint unlock arrays, facility anchor flags, crime/species tuning, and fixed character stats.
- The current audit must not repeat the prior P0/P1/P2 document or the user's explicitly named exclusions.
- Runtime inventory currently contains 335 C# files outside Editor folders.
- Prior-audit findings excluded from this pass: building ability/FacilityFunction authority, building-state persistence, stock snapshots, manual scene injection and required-service no-op fallbacks, EventObserver lifecycle, work/need catalogs, character log text parsing, run/meta effect bags, concrete AI-action orchestration, room/facility role duplication, feature-surface presenter and tab-text routing, defense effects, blueprint unlock arrays, facility anchors, crime/species tuning, StatChange hunger targeting, and fixed character stats.
- Explicit user exclusions are tracked separately even if a similarly shaped search hit appears: static event payloads, Grid removal identity, BehaviorSource owner, camera fallback, backdrop scene-name protocol, and CharacterSO basicBehaviorPatterns.
- Prior-audit non-findings remain presumptively excluded unless current consumer evidence proves a different defect: polymorphic considerations/conditions/purchase effects, generic building factory dispatch, coherent evolution/synthesis recipe records, finite discontent state transitions, small mood/plan state enums, UI pointer blocker, and SocialReputationRuntime teardown.

## Confirmed Findings (Working)
- LLM queue abort is not completion-safe: `OnDisable`/debug abort stop all coroutines and clear queued requests without invoking callbacks. `CachedLocalLlmFacilityEvolutionProposalProvider` removes `pendingSignatures` only on rejection or callback, so an accepted request aborted by queue disable remains permanently pending and cannot be retried. Trace other consumers to quantify blast radius.
- The callback-loss blast radius is confirmed beyond evolution: `CustomerPersonaRuntime` sets `personaRequestInProgress` before enqueue and clears it only on immediate rejection or callback, so queue disable permanently blocks that actor's retry. `CharacterLogNarrativeService` similarly decrements per-log pending counts only on callback, while `CharacterLog` hides entries until narrative completion, leaving accepted-aborted records private and eventually saturating the three-request limit.
- `BackgroundManager` leaks shared-data subscriptions and tween ownership: `Awake` attaches two anonymous lambdas to the ScriptableObject-backed `gameData.day.OnValueChange`, provides no unsubscription, and creates untracked DOTween sequences per day. Destruction/scene replacement leaves closures in the shared `Data<int>` action and can target destroyed render objects or duplicate day transitions.
- `AbilitySchedule` is disable/reenable-unsafe because it subscribes only in `Awake` and unsubscribes in `OnDisable`, but the actual pooled customer prefab does not contain that component. Keep this as secondary lifecycle evidence only; do not claim the pool currently triggers it.
- Global GameData fixed persistence bag: `GameData` owns six `Data<T>` slots. `ModularFacilityGameDataSaveData` mirrors each with `hasX + value`; capture and restore manually enumerate the same six fields. In addition, `hour` and `timeOfDay` are derived from `curTime` every frame but all three are independently serialized, while restored `gameSpeed` does not update the separate `Time.timeScale` authority. Adding or restoring global state therefore crosses aggregate, initialization, save DTO, capture/restore, and engine synchronization. This is distinct from the prior building-module persistence finding.
- Offense reward closed aggregate: seven `OffenseRewardCategory` values dispatch through one switch; `OffenseRewardContext` carries every category's dependencies and `OffenseRewardState` carries category-specific counters/sets. Stock subtype, blueprint intent, faction target, and special-prisoner behavior are inferred from display labels. Adding or renaming content can change grants, and a new reward category expands the enum, switch, state, context, and often selector.
- Local LLM request taxonomy is closed in several parallel authorities: seven enum values, seven interface methods, seven timeout fields, seven wrappers, a priority switch, and special-type queue policies. Combine with the generic parser's separate raw-validator type list as one infrastructure finding unless remediation directions differ materially.
- Facility shop closed union / fixed optional bag: `FacilityShopOffer` stores `FacilityShopOfferType` beside mutually exclusive `BuildingSO` and `FacilityBlueprintSO`; `FacilityShopPurchaseResult` then duplicates snapshot, discriminator, data ID, and both references. Runtime purchase application branches by type, blueprint research consumes only the blueprint field, and UI branches again for labels. A third offer kind requires coordinated edits across model, constructors, validity/display/snapshot, purchase application, result projection, event consumers, and UI.
- Facility evolution species-name routing: visits call `IsCombatVisitor`/`IsNobleVisitor`; these inspect `SpeciesTag` substrings (`Orc`, `Mercenary`, `Vampire`, `Noble`) and feed `CombatVisitorRatio`, `NobleVisitorRatio`, `MercenaryHangout`, and `NoblePatronage`. Those metrics/tokens participate in identity signals and recipe eligibility, so renaming/localizing a tag or adding a semantically equivalent species changes progression.
- Generic LLM parser closed subtype list: `TryParse<T>` performs central `typeof(T)` checks for four DTOs before `JsonUtility`, while seven DTO types currently use the API. Per-type semantic `Validate` is polymorphic, but required-field raw JSON validation is a separate central authority. A new DTO can compile and parse while silently skipping required-field-presence checks unless the parser is edited.
- Facility evolution LLM status text protocol: status messages are stored as strings, then `StartsWith("LLM failed/unavailable/disabled")` determines the structured proposal source propagated into evolution result and Codex. Rewording a diagnostic can misclassify completed failure as pending.
- Facility evolution resource fallback hides a missing production integration: recipes expose `requiredMaterials`, validation and mutation call `IFacilityEvolutionResourceProvider`, but the runtime constructor does not inject that port and production composition does not register it. `CreateEngine` silently installs `EmptyFacilityEvolutionResourceProvider`, which rejects every positive named requirement. Current recipe assets happen to leave the array empty; the first authored material requirement becomes permanently unavailable instead of exposing a composition error.
- CharacterActor is a fixed runtime feature aggregate: it requires, caches, validates, binds, and exposes `CustomerPersonaRuntime` for every actor even though that module immediately rejects every non-customer. The single prefab/actor initialization path calls persona generation unconditionally, and facility scoring treats its default state as a real neutral contribution. Role-specific additions are therefore pushed toward another mandatory component/property/check rather than optional capability composition.
- Facility semantic routing uses presentation and enum text as identity in multiple real behaviors: LLM macro/mood targets are matched by substring against `BuildingSO.objectName` and `FacilityRole.ToString()`, the same matching controls mood interruption/cooldown, and generated persona preference tags feed facility candidate scores through the same display strings. Stable IDs are available for exact targets, but the accepted tag fallback remains a behavior protocol vulnerable to renames, localization, and substring collisions.
- Typed AI failures are reconstructed from diagnostic text: `AIActionFailure.FromReason` maps Korean/English substrings such as `ņłśņÜ® ņØĖņøÉ`, `ļ╣äļ▓ł`, `Ļ▓ĮļĪ£ ņŚåņØī`, and `ņ¦ĆņøÉĒĢśņ¦Ć` into `AIActionFailureKind`. Reservation and work-selection paths feed their free-form messages through it, then `AIBrain` uses the inferred kind to decide candidate cooldown and failure priority. Rewording a producer message therefore changes retry behavior despite the typed failure model already existing.
- Defense observation text protocol is technically confirmed: concrete effect assets emit designer-editable `logTag`; invasion formatting and Codex independently recover effect semantics by Korean substring. Because this borders the prior P2 defense-effect finding, keep it pending overlap review before final inclusion.

## Rejected Candidates
- Defense observation `logTag` parsing overlaps the excluded prior P2 defense-effect authority finding and will not be repeated.
- Facility-evolution provider-to-component-service casts end in explicit failure and reflect one implementation serving two related ports; they are not independently severe enough to report.
- Large classes such as `CharacterBlackboard` are cohesive runtime state/diagnostic aggregates; line count alone is not evidence of an optional-field bag.
- `AbilitySchedule` lifecycle asymmetry is real but no current pool/reenable consumer was proven, so it is below the final reporting threshold.
- Facility-evolution status-prefix routing has a real structured-source consequence, but it is reporting-oriented and materially weaker than the retained behavior-routing findings.
- CharacterActor's mandatory persona module is an overly broad aggregate signal, but current scoring intentionally defines a neutral persona contribution; without a proven role-extension failure it remains below the reporting threshold.
- Facility-evolution species-name routing is confirmed, but it is lower impact than the retained facility-name and AI-failure text protocols and is omitted to keep the final report focused.

## Final Report Set
- High: Local LLM disable/abort loses accepted callbacks and leaves consumer-owned pending state permanently uncleared.
- High: Offense rewards combine a closed payload, central dispatch, text-derived behavior, duplicated state, and a broad scene-located context.
- Medium-high: Typed AI failure kinds are reconstructed from localized diagnostics and control cooldown/priority.
- Medium: Global GameData/save state duplicates derived clock values and engine speed authority.
- Medium: Facility behavior targets and persona preferences use display/enum text as semantic identity.
- Medium: Facility shop offers are a closed discriminated optional-field bag duplicated into results and event consumers.
- Medium: Local LLM request kinds are repeated across enum, interface, settings, wrappers, policy, and raw-parser subtype dispatch.
- Medium: Facility evolution silently replaces an absent material provider with a rejecting empty implementation.
- Medium: BackgroundManager leaves ScriptableObject subscriptions and DOTween sequences without lifecycle ownership.
