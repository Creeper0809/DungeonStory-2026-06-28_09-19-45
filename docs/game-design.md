# DungeonStory 게임 설계

이 문서는 전체 게임 설계의 허브다.

세부 설계는 `docs/game-design/` 아래 주제별 문서로 분리한다.

## 한 줄 콘셉트

몬스터 사장이 몬스터 고객을 위한 던전을 경영하고, 시설/몬스터/설계도를 덱처럼 조합해 모험가 침입을 최대한 오래 버티며, 성장한 몬스터와 시설로 다른 던전에 원정을 보내는 로그라이크식 경영 + 던전 공방 게임.

## 핵심 루프

```text
경영
-> 침입
-> 연구/합성
-> 오펜스
-> 보상으로 던전 재설계
```

## 현재 설계 상태

큰 게임 구성은 정해진 상태다.

정해진 핵심:

- 몬스터 고객 중심 경영
- 장기 목표는 로그라이크식 덱빌딩 생존이며, 플레이어 캐릭터가 죽으면 런 종료
- 런 종료 후 얼마나 버텼는지에 따라 계승 화폐를 받고, 이 화폐로 계승 트리를 강화
- 런 시작 시 사장 종족을 선택하고, 초반에는 사장이 직접 근무/보충/수리/연구/방어에 참여
- 런 시작 변수, 운영 중 이벤트, 침입/보상 변수로 매 런의 운영 흐름을 흔듦
- 고객 AI는 욕구 기반 Utility AI로 행동을 선택하고, 도달 가능 시설 중 선호도/재고/가격/혼잡/거리로 목적지를 점수화
- 초기 종족 컨셉 5개: 슬라임, 오크, 뱀파이어, 수인, 데몬
- 능력/개인 특성/종족 특성을 가진 캐릭터 공통 모델
- 만족한 손님이 단골이 되고 직원/방어/원정 인력으로 영입되는 확보 루프
- 영입된 몬스터가 자동 비번 중 시설을 이용하며 피로와 만족도를 관리하는 생활 루프
- 직원 유지비를 월급이 아니라 시설 점유, 재고/식사 소비, 피로/만족도 관리로 표현
- 직원 불만이 효율 저하, 태업/결근/사고, 영구 이탈, 국지 반란과 사장방 확산으로 악화되는 내부 위협
- 반란 대응은 자동 제압, 제압 명령, 격리, 협상/진정을 모두 지원
- 침입/반란 피해 시설은 사라지지 않고 파손 상태가 되며, 초기에는 완전 비활성화 후 수리로 복구
- 림월드식 직원 작업 우선순위
- 모든 작업은 자동으로 진행하되, 선택 캐릭터의 대상 우선 지정 명령을 지원
- 2D 횡스크롤 방/복도/계단 동선
- 방은 벽/문/가구/운영 기록 조건을 정체성 압력으로 압축하고, 현재 시설은 그 조건을 바탕으로 시설 계보 진화를 진행
- 방어 시설도 복도 부착물이 아니라 Building 레이어의 주 시설로 처리
- 고정 하루 턴이 아닌 운영일 정산 이벤트
- 모험가 침입과 자동 전투
- 전투 중 발동 표시와 결과 요약
- 몬스터/침략/시설 도감 정보 해금
- 운영일 정산 기준 일일 상점 리셋
- 경영 시설군: 식당, 상점, 휴식, 훈련, 연구/마력, 오락, 창고/보충, 직원 생활/유지보수
- 초기 재고 계열 4개: 식재료, 잡화, 무기, 마력
- 창고에서 시설 내부 재고로 옮기는 직원 보충 작업
- 원재료 수급: 운영일 납품, 내부 생산, 침입 방어 보상, 오펜스 원정 보상
- 재고 성장 흐름: 초반은 돈으로 재고를 사서 운영하고, 중후반은 내부 생산/보상으로 재고 안정성과 고급 운영을 강화
- 경영/방어/지원 시설 역할 분리와 합성 연결
- 초기 시설 공격 컨셉 5개: 물리/가시, 독/부식, 화염/연소, 번개/축전, 냉기/속박
- 초기 방어 시설 세트: 가시 함정, 독 웅덩이, 화염 분사구, 번개 기둥, 냉기 분사구, 경비실
- 방어 시설은 시설별로 진입 발동, 체류 주기 발동, 쿨타임 발동, 경비 합류를 다르게 사용
- 전투 효과 구현은 Visitor가 아니라 Strategy + ScriptableObject Effect 구조를 사용
- 모든 공격 컨셉은 피해를 가지며, 초기에는 번개만 축전 누적 구조를 사용
- 직원 연구 작업으로 연구시설에서 설계도 분석 후 시설/진화식 해금
- 일반 진화식은 공개하고, 특수 진화식은 설계도 연구로 해금
- 연구로 해금되는 분기형 시설 계보 진화와, 방의 운영 기록을 촉매로 쓰는 LLM 해석형 진화 후보
- 월드맵 기반 오펜스 원정은 희귀 시설/설계도 획득, 인간 세력 약화, 손님/직원/포로 확보를 담당
- 톤은 하나로 고정하지 않고 플레이어 운영 방식에 따라 몬스터 상점, 방어형 던전, 블랙 코미디, 마왕 던전 느낌이 달라지게 둠

다음 단계는 세부 수치와 첫 프로토타입 범위를 확정하는 것이다.
