## 우선순위 기준

```text
P0: 없으면 현재 플레이가 멈추거나 다음 시스템 구현이 어려운 기반 작업
P1: 첫 번째 플레이 가능한 경영/침입/합성 루프에 필요한 작업
P2: 첫 루프가 읽힌 뒤 확장할 작업
P3: 장기 확장, 오펜스, 메타 심화 작업
```

완료 기준은 다음 세 가지를 같이 본다.

- 플레이어가 화면에서 원인과 결과를 읽을 수 있다.
- AI가 목적지를 잃거나 null 행동 루프에 빠지지 않는다.
- 설계 문서의 다음 시스템이 이 작업 위에 자연스럽게 얹힌다.

## 구현 원칙

- 경영, 침입, 합성, 직원 운영, 피드백은 모두 필요하다. 다만 처음에는 각 시스템의 콘텐츠 수를 줄인다.
- 모든 자동 행동은 실패 원인을 로그로 남기고 재계획할 수 있어야 한다.
- 전투 결과 화면은 추천 대응을 직접 제시하지 않고, 관측된 사실만 보여준다.
- 방어 시설은 초기에는 복도 부착물이 아니라 `Building` 레이어의 주 시설로 둔다.
- 합성은 재료 아이템 조합이 아니라 시설물끼리의 조합이다.
- 일반 조합식은 공개하고, 특수 조합식은 설계도 연구로 해금한다.
- 사장 캐릭터가 죽으면 런이 끝난다. 사장방에 도달하는 것만으로 즉시 게임 오버가 아니다.

## 0. 현재 기반 안정화

우선순위: P0

### 0.1 Grid/Building 구조 정리

- [x] `Grid`가 이동, 점유, 연결성만 책임하는지 점검한다.
- [x] `GridSystemManager`가 특정 게임 규칙보다 범용 그리드 런타임에 가까운지 다시 분류한다.
- [x] `BuildingSO`가 던전스토리 전용 데이터와 범용 배치 데이터를 섞고 있는지 분리 기준을 정한다.
- [x] `BuildableObject`가 그리드 점유, 비주얼, 상호작용, 시설 기능을 과하게 동시에 들고 있지 않은지 정리한다.
- [x] `GridBuildingRuntime`과 `DungeonStoryGridBuildingController`의 역할을 다시 나눈다.
- [x] 범용 배치/검증/고스트/매니저는 `Assets/Scripts/Grid` 아래에 둔다.
- [x] 던전스토리 전용 규칙은 이름에서 명확히 드러나게 둔다.
- [x] 스크립트 수를 늘려 분리하기보다, 한 파일 안에 있어도 되는 작은 helper는 무리하게 쪼개지 않는다.

완료 기준:

- 새 시설을 배치할 때 상위 API가 GameObject와 Tile 비주얼을 조립할 수 있다.
- Grid 하위 API는 특정 던전스토리 컨셉 없이 다른 Unity 프로젝트에서도 그리드 에셋처럼 재사용 가능하다.
- Unity 의존을 제거하는 것이 목표가 아니라, 게임 특화 의존을 낮추는 것이 목표다.

### 0.2 이동과 경로 안정화

- [x] 캐릭터 생명주기 상태를 분리한다.
- [x] 외부 스폰, 던전 입장, 활성 상태, 퇴장, despawn 흐름을 분리한다.
- [x] AI가 던전 입장 전이나 퇴장 중에 행동 판단을 하지 않게 한다.
- [x] 방문할 곳이 없을 때 손님이 멈추지 않고 배회/퇴장 fallback을 갖게 한다.
- [x] 직원은 할 일이 없을 때 퇴장하지 않고 대기 fallback을 사용하게 한다.
- [x] 계단 이동 애니메이션이 BuildingSO의 이동 앵커/경로 정보를 기준으로 중앙을 지나가는지 검증한다.
- [x] 복도, 계단, 시설 내부 이동이 모두 같은 `GridMoveStep` 기준으로 처리되는지 확인한다.
- [x] 이동 불가 영역이 배회 목적지 후보에 섞이지 않는지 Play Mode에서 반복 검증한다.
- [x] 입장/퇴장/계단/일반 이동을 각각 테스트 씬 또는 디버그 시나리오로 만든다.

완료 기준:

- 손님 1명과 직원 1명이 5분 이상 멈춤 없이 입장, 행동, fallback, 퇴장을 반복한다.
- 계단 이동 시 캐릭터가 계단 끝이나 잘못된 앵커로 튀지 않는다.
- 콘솔에 null 행동, 경로 없음 무한 반복, lifecycle 관련 에러가 없다.

### 0.3 AI Action/Consideration 정리

- [x] 행동 선택과 목적지 선택을 명확히 분리한다.
- [x] 하나의 판단 주기에서 BFS 결과를 캐시하고 여러 consideration이 공유하게 한다.
- [x] `AIAction`이 목적지 없는 행동, 목적지 있는 행동, 이동 경로 있는 행동을 모두 안전하게 표현하게 한다.
- [x] 행동 실패 시 캐릭터 로그에 짧은 원인 태그를 남긴다.
- [x] 손님 전용 행동과 직원 전용 행동이 prefab/action list에서 섞이지 않게 점검한다.

완료 기준:

- `bestAction == null` 상태가 정상 fallback으로 처리된다.
- 손님은 퇴장 가능하고, 직원은 퇴장하지 않는다.
- 같은 행동 실패가 무한 반복될 때 로그와 쿨다운으로 재시도 폭주를 막는다.

우선순위: P1

초기 경영 시설:

- [ ] 휴식방
- [ ] 훈련장
- [ ] 연구실
- [ ] 마력 저장소
- [ ] 창고

각 시설에 필요한 공통 데이터:

- [ ] 시설 역할 태그: 식사, 구매, 휴식, 훈련, 연구, 마력, 물류
- [ ] 수용 인원
- [ ] 이용 시간

- [ ] 선호 종족 보정

- [ ] 파손 시 비활성화 여부

완료 기준:

- 손님이 시설을 실제 목적지로 선택한다.
- 직원이 시설을 근무 대상으로 인식한다.
- 재고가 없거나 파손되면 후보에서 제외되거나 실패 로그가 남는다.

### 1.2 손님 AI

- [ ] 손님의 행동 후보를 식사, 구매, 휴식, 구경, 퇴장으로 구성한다.
- [ ] 현재 욕구를 행동 점수의 중심으로 사용한다.
- [ ] 행동별 목적지 후보를 도달 가능한 시설 목록에서 수집한다.
- [ ] 후보 제거 조건을 구현한다: 도달 불가, 파손, 재고 0, 돈 부족, 이용 불가.
- [ ] 후보 점수 요소를 구현한다: 욕구 적합도, 종족 선호, 개인 특성, 재고, 가격, 혼잡, 거리, 신규성.
- [ ] 최근 방문 시설은 재방문 점수를 낮춘다.
- [ ] 목적지를 못 잡으면 배회 1회 후 퇴장한다.
- [ ] 비정상 상황에서는 짧게 대기 후 재판단한다.

완료 기준:

- 오크는 고기 식당과 무기점을 더 자주 고른다.
- 슬라임은 저가 음식점과 휴식방을 더 자주 고른다.
- 뱀파이어는 마력/어둠/연구 관련 시설을 더 자주 고른다.
- 가까운 시설만 무조건 선택하지 않는다.

### 1.3 캐릭터 모델

- [ ] `CharacterStats`를 만든다: 공격, 영업, 연구, 이동속도, 힘을 1차 구현으로 둔다.
- [ ] 2차 능력 슬롯을 남긴다: 맷집, 손재주, 청소, 지구력.
- [ ] 개인 특성 8개 이하를 데이터로 만든다.
- [ ] 종족 특성 데이터 구조를 만든다.
- [ ] 캐릭터는 능력, 개인 특성, 종족 특성을 모두 가진다.
- [ ] 손님, 직원, 방어 인력, 원정 인력은 같은 캐릭터 모델을 공유한다.
- [ ] 역할 전환 시 캐릭터의 특성과 능력이 유지된다.

초기 개인 특성 후보:

- [ ] 대식가
- [ ] 절약가
- [ ] 부자
- [ ] 성급함
- [ ] 소심함
- [ ] 싸움꾼
- [ ] 깔끔함
- [ ] 연구가 또는 호기심 많음

완료 기준:

- 같은 오크라도 개인 특성에 따라 소비, 작업, 사고 확률이 달라진다.
- 능력/특성/종족 보정이 한 곳에서 최종 효율로 계산된다.

- [ ] 런 시작 시 사장 종족 선택 UI를 만든다.

- 사장을 위험 지역에 오래 두는 것이 리스크로 작동한다.

## 2. 직원과 작업 시스템

우선순위: P1

### 2.1 작업 우선순위

- [ ] 직원별 작업 우선순위 UI를 만든다.
- [ ] 우선순위 값은 1, 2, 3, 꺼짐으로 둔다.
- [ ] 작업 카테고리를 구현한다: 근무, 보충, 수리, 청소, 연구, 경비, 구조/응급, 휴식.
- [ ] 같은 우선순위에서는 거리, 긴급도, 능력, 종족 적성을 고려한다.
- [ ] 침입, 반란, 파손 같은 긴급 상황은 별도 긴급도 보정을 받는다.
- [ ] 피로가 너무 높으면 휴식이 AI 보호 규칙으로 올라온다.

완료 기준:

- 오크 직원은 경비/수리에 강하고, 뱀파이어 직원은 연구에 강한 식으로 차이가 난다.
- 할 일이 없는 직원은 대기하거나 비번 행동을 한다.
- 낮은 우선순위 작업이 영원히 굶는 상황은 로그/정산에서 보인다.

### 2.2 우선 지정

- [ ] 캐릭터 선택 상태를 만든다.
- [ ] 대상 시설 우클릭 시 가능한 작업을 판정한다.
- [ ] 파손 시설 우클릭은 수리 우선 지정이 된다.
- [ ] 재고 부족 시설 우클릭은 보충 우선 지정이 된다.
- [ ] 연구실 또는 설계도 우클릭은 연구 우선 지정이 된다.
- [ ] 침입자/반란 몬스터 우클릭은 제압 우선 지정이 된다.
- [ ] 도달 불가나 수행 불가 시 실패 로그를 남긴다.

완료 기준:

- 우선 지정은 직접 조작이 아니라 AI 목표 변경으로 작동한다.
- 캐릭터가 이동부터 작업 완료까지 자동으로 수행한다.

### 2.3 근무와 비번

- [ ] 직원 피로 값을 만든다.
- [ ] 직원 만족도 값을 만든다.
- [ ] 근무 중 피로가 오른다.
- [ ] 비번 중 식사/휴식/선호 시설을 이용한다.
- [ ] 비번 직원은 매출을 만들지 않는다.
- [ ] 비번 직원은 시설 수용량과 동선을 차지한다.
- [ ] 긴급 상황에서는 상태가 허용하면 비번 중에도 대응에 합류한다.

완료 기준:

- 직원 수가 늘어나면 휴식방, 식당, 물류 동선이 실제 압박이 된다.
- 직원 유지비가 월급이 아니라 내부 운영 부담으로 느껴진다.

## 3. 재고와 물류

우선순위: P1

### 3.1 재고 계열

초기 재고 계열:

- [ ] 식재료
- [ ] 잡화
- [ ] 무기
- [ ] 마력

공통 구현:

- [ ] 직원 보충 작업이 창고에서 시설로 재고를 옮긴다.
- [ ] 재고 0이면 해당 이용 후보를 제외하거나 실패 로그를 남긴다.
- [ ] 시설 UI에 현재 내부 재고와 보충 요청 상태를 표시한다.

완료 기준:

- 식당 재고가 없으면 손님/직원 식사 문제가 발생한다.
- 무기점 재고가 없으면 오크 구매/만족 루프가 흔들린다.
- 마력 재고가 부족하면 연구/마력 시설 운영이 늦어진다.

### 3.2 초반 수급

- [ ] 운영일 납품을 만든다.
- [ ] 돈으로 창고 재고를 구매할 수 있게 한다.
- [ ] 운영일 정산 후 재고 구매/납품 결과를 보여준다.
- [ ] 침입 방어 보상으로 일부 재고를 지급할 수 있게 한다.
- [ ] 내부 생산은 첫 버전에서 최소화하고, 중후반 확장 슬롯만 남긴다.

완료 기준:

- 초반 경제는 돈으로 재고를 사서 시설을 돌리고 다시 돈을 버는 구조가 된다.
- 후반 내부 생산과 오펜스 보상이 들어갈 데이터 자리가 남아 있다.

## 5. 침입과 방어

우선순위: P1

### 5.1 위협도 시스템

- [ ] 내부 위협도 값을 만든다.
- [ ] 위협도는 플레이어에게 게이지로 직접 보여주지 않는다.
- [ ] 던전 가치, 평판, 시간, 난이도, 위험도를 상승 요인으로 둔다.
- [ ] 70퍼센트 이상에서 경고 이벤트가 가능하게 한다.
- [ ] 100퍼센트 이상에서 랜덤 딜레이 후 침입 후보가 되게 한다.
- [ ] 침입 직후 위협도를 초기화하고 안전 시간을 둔다.

완료 기준:

- 침입은 고정 날짜가 아니라 던전 성장과 시간에 반응한다.
- 플레이어는 소문/정찰/경고로 침입 가능성을 간접적으로 파악한다.

### 5.2 침입자 1종 구현

- [ ] 사장방 돌파형 침입자 1종을 만든다.
- [ ] 침입자도 그리드/복도/계단 이동 로직을 사용한다.
- [ ] 침입 직후에는 탐색성이 있고, 시간이 지날수록 사장방 방향 보정이 강해진다.
- [ ] 첫 버전 보조 목표는 일부 시설 파손 정도로 둔다.
- [ ] 사장방 도달 시 최종 교전 단계로 들어간다.
- [ ] 사장이 사망하면 런 종료로 연결한다.

완료 기준:

- 침입자가 너무 오래 헤매지 않는다.
- 던전 배치가 침입자 노출 시간과 피해량에 영향을 준다.
- 사장방 도달이 즉시 게임 오버가 아니라 최종 교전 진입으로 보인다.

### 5.3 초기 방어 시설

초기 1성 방어 시설:

- [ ] 가시 함정: 물리/가시, 즉발 피해
- [ ] 독 웅덩이: 독/부식, 피해와 방어력 감소
- [ ] 화염 분사구: 화염/연소, 높은 피해와 지속 피해
- [ ] 번개 기둥: 번개/축전, 피해와 연계
- [ ] 냉기 분사구: 냉기/속박, 피해와 감속
- [ ] 경비실: 직원 대기와 전투 합류

공통 구현:

- [ ] 방어 시설도 주 시설로 배치한다.
- [ ] 정상/파손 2상태를 가진다.
- [ ] 파손 시 완전 비활성화된다.
- [ ] 직원 수리 작업으로 복구된다.
- [ ] 시설별 발동 조건, 대상 규칙, 쿨타임/주기를 데이터화한다.

완료 기준:

- 각 방어 시설은 피해 외에 서로 다른 의미가 있다.
- 방어 시설을 배치한 위치가 침입 결과에 영향을 준다.

### 5.4 전투 효과 구조

- [ ] `DefenseFacilitySO` 또는 동등한 방어 시설 데이터 구조를 만든다.
- [ ] 발동 조건 데이터를 만든다.
- [ ] 대상 선택 데이터를 만든다.
- [ ] Effect ScriptableObject 기반으로 피해/상태이상을 처리한다.
- [ ] 번개만 축전 스택을 가진다.
- [ ] 부식, 연소, 감속은 중첩보다 지속 시간 갱신으로 시작한다.
- [ ] 전투 로그를 효과 적용 결과에서 생성한다.

완료 기준:

- 새 방어 효과를 추가할 때 기존 효과 코드를 크게 수정하지 않는다.
- Visitor 패턴 중심이 아니라 Strategy + SO Effect 조합으로 확장된다.

### 5.5 자동 전투 피드백

- [ ] 침입 결과 요약 패널을 만든다.
- [ ] 전투 결과는 추천 대응을 직접 제시하지 않는다.
- [ ] 시설, 직원, 함정 시너지가 어떻게 작동했는지 짧게 보여준다.

결과 요약 항목:

- [ ] 가장 많은 피해를 준 시설
- [ ] 가장 오래 지연시킨 시설
- [ ] 결정적 방어
- [ ] 피해를 입은 시설
- [ ] 파손 시설
- [ ] 획득한 관찰 정보

완료 기준:

- 리플레이 없이도 플레이어가 어떤 방어가 의미 있었는지 읽을 수 있다.

## 6. 시설 획득, 연구, 합성

우선순위: P1

### 6.1 상점과 기본 구매

- [ ] 운영일 정산 후 상점 목록이 갱신된다.
- [ ] 상점에는 시설과 설계도가 나온다.
- [ ] 희귀 상품은 랜덤 등장한다.
- [ ] 연구된 1성 시설은 기본 구매 목록에 들어간다.
- [ ] 일부 2성 시설도 연구 후 기본 구매 가능하게 둔다.
- [ ] 3성 이상은 합성, 침입 보상, 오펜스 보상 중심으로 둔다.

완료 기준:

- 필요한 기초 시설이 운 때문에 너무 오래 막히지 않는다.
- 상점 랜덤성은 희귀 시설과 특수 설계도 쪽에 남는다.

### 6.2 연구시설과 설계도

- [ ] 설계도 아이템 또는 데이터 구조를 만든다.
- [ ] 연구실에서 설계도를 분석할 수 있게 한다.
- [ ] 연구 작업은 직원 작업 우선순위에 포함된다.
- [ ] 연구 속도는 연구 능력, 개인 특성, 종족 보정, 시설 보정의 영향을 받는다.
- [ ] 연구 완료 시 특수 조합식 또는 기본 구매 목록이 해금된다.
- [ ] 연구 중단/재개가 가능하게 상태를 저장한다.

완료 기준:

- 뱀파이어나 연구가 특성이 연구 속도 차이를 실제로 만든다.
- 설계도는 그냥 소모품이 아니라 새 빌드 방향을 여는 정보가 된다.

### 6.3 시설물끼리의 합성

- [ ] 합성 UI를 만든다.
- [ ] 합성 대상은 시설물이다.
- [ ] 합성에 사용한 시설은 사라진다.
- [ ] 합성 결과는 새 시설이 되거나 기존 시설을 상위 시설로 대체한다.
- [ ] 배치 중인 시설을 합성 재료로 쓸지 여부를 명확히 정한다.
- [ ] 합성 결과에 재료 시설 레벨 일부를 계승한다.
- [ ] 파손 시설은 수리 전까지 합성 재료로 사용하지 못한다.
- [ ] 일반 조합식은 처음부터 보여준다.
- [ ] 특수 조합식은 설계도 연구 후 보여준다.

완료 기준:

- 고기 식당이 여러 갈래로 성장할 수 있다.
- 시설 강화가 단순 수치 상승이 아니라 기능 변화로 느껴진다.

### 6.4 대표 합성 트리

첫 프로토타입은 모든 시설군을 5성까지 만들지 않는다.

- [ ] 식당 계열 1성에서 3성까지 최소 2갈래 구현
- [ ] 함정 계열 1성에서 3성까지 최소 2갈래 구현
- [ ] 경비/훈련 계열 1성에서 3성까지 최소 1갈래 구현

예시 트리:

- [ ] 고기 식당 + 훈련장 -> 전투 식당
- [ ] 고기 식당 + 장식/휴식 계열 -> 고급 고기 식당
- [ ] 전투 식당 + 경비실 -> 전장의 식당
- [ ] 가시 함정 + 독 웅덩이 -> 맹독 가시 함정
- [ ] 번개 기둥 + 경비실 -> 경보 코일
- [ ] 경비실 + 훈련장 -> 병영

완료 기준:

- 첫 침입 이후 플레이어가 합성을 연구하고 다음 침입을 대비하는 흐름이 생긴다.

## 8. 손님 영입과 내부 위협

- [ ] 단골 중 일부가 영입 후보가 된다.
- [ ] 영입하면 소비 고객 하나를 잃고 직원/방어/원정 인력을 얻는다.
- [ ] 영입 조건은 초기에는 방문 횟수와 평균 만족도만 쓴다.

완료 기준:

- [ ] 1단계 효율 저하를 구현한다.

- [ ] 이탈과 반란은 기본적으로 영구 손실로 둔다.
- [ ] 반란은 바로 사장방을 노리지 않고 주변 피해에서 시작한다.
- [ ] 장기 방치 시 사장방/사장 위협으로 확산된다.

완료 기준:

- 직원 관리 실패가 외부 침입과 다른 내부 위협으로 느껴진다.
- 영구 손실 전 충분한 경고가 있다.

### 8.3 반란 대응

- [ ] 자동 제압
- [ ] 제압 명령
- [ ] 격리
- [ ] 협상/진정

완료 기준:

- 반란 대응도 직접 전투 조작이 아니라 자동 AI 목표 지정으로 처리된다.
- 협상/진정은 반란 직전 또는 초기 단계에서만 의미가 있다.

- [ ] 사장 종족
- [ ] 시작 시설 후보
- [ ] 시작 손님층 후보
- [ ] 시작 설계도 후보
- [ ] 초기 상점 상품
- [ ] 초기 던전 구조
- [ ] 난이도와 위협 계수

완료 기준:

- 시작 선택이 이번 런의 방향을 바꾼다.

### 9.2 운영 중 이벤트

초기 변수 세트:

- [ ] 슬라임 단체 방문
- [ ] 오크 회식
- [ ] 뱀파이어 야간 방문
- [ ] 식재료 납품 지연
- [ ] 잡화 할인
- [ ] 마력 재고 과잉
- [ ] 방문 상인
- [ ] 방어 시설 할인
- [ ] 설계도 소문

완료 기준:

- 이벤트는 즉시 큰 피해보다 우선순위를 바꾸는 압력으로 작동한다.
- 최소 2개 이상의 대응 방법이 있다.

### 9.3 침입 변수

- [ ] 정찰 흔적
- [ ] 급습
- [ ] 무장한 침입자
- [ ] 약탈 우선
- [ ] 피로한 침입자

완료 기준:

- 침입자 종류가 1종이어도 침입 상황이 매번 완전히 같지는 않다.

## 10. 런 종료와 계승

우선순위: P2

### 10.1 런 종료

- [ ] 사장방 도달 후 최종 교전 상태를 만든다.
- [ ] 사장 체력 또는 생존 상태를 런 종료 조건으로 연결한다.
- [ ] 사장 사망 시 런 결과 정산 화면으로 이동한다.
- [ ] 현재 런의 돈, 재고, 배치 시설은 기본적으로 계승하지 않는다.

완료 기준:

- 플레이어는 사장방이 뚫린 순간이 아니라 사장이 죽는 순간 런 종료를 경험한다.

### 10.2 계승 화폐

- [ ] 생존 시간
- [ ] 운영일 정산 횟수
- [ ] 막아낸 침입 수
- [ ] 도달한 최대 위협 단계
- [ ] 최종 침입 강도
- [ ] 최초 발견 시설
- [ ] 최초 해금 조합식
- [ ] 오펜스 성과
- [ ] 난이도 배율

완료 기준:

- 오래 버틴 보상이 가장 크다.
- 짧게 죽고 발견 보너스만 챙기는 플레이가 최적이 되지 않는다.

### 10.3 초기 계승 트리

- [ ] 운영 지식
- [ ] 설계 보존
- [ ] 사장 생존

초기 강화 예시:

- [ ] 시작 시설 후보 +1
- [ ] 시작 사장 특성 후보 +1
- [ ] 1성 기본 시설 구매 목록 확장
- [ ] 특수 조합식 기록 일부 보존
- [ ] 사장 생존력 소폭 증가
- [ ] 침입 전 경고 정확도 증가

완료 기준:

- 계승은 다음 런의 선택지를 늘리고 초반 반복 피로를 줄인다.
- 계승만으로 방어 설계가 무의미해지지 않는다.

## 11. 오펜스

우선순위: P3

오펜스는 초반 핵심 루프가 안정된 뒤 구현한다.

### 11.1 월드맵과 정찰

- [ ] 월드맵 화면을 만든다.
- [ ] 가까운 원정 대상만 처음에 보이게 한다.
- [ ] 정찰/스캔 업그레이드로 더 먼 대상이 열린다.
- [ ] 대상은 위험도, 예상 보상, 소요 시간, 필요 인력을 가진다.

### 11.2 원정 편성

- [ ] 직원/방어 몬스터를 원정대에 편성한다.
- [ ] 원정 중인 몬스터는 던전 운영/방어에서 빠진다.
- [ ] 원정 전투는 자동 처리한다.
- [ ] 원정 결과로 부상, 사망, 피로, 보상 획득이 가능하다.

### 11.3 오펜스 보상

- [ ] 희귀 시설
- [ ] 특수 설계도
- [ ] 인간 세력 또는 경쟁 세력 약화
- [ ] 새 손님/직원 후보
- [ ] 포로 또는 특수 몬스터
- [ ] 특정 재고 계열 보급

완료 기준:

- 오펜스는 단순 보상 버튼이 아니라, 내 던전 바깥 세계를 건드리는 선택이다.

## 첫 플레이 가능 버전 순서

### 1차 목표: 살아 움직이는 경영

- [ ] 그리드 이동, 입장, 퇴장, 배회 안정화
- [ ] 저가 음식점, 잡화점, 휴식방, 창고 구현
- [ ] 식재료, 잡화 재고 구현
- [ ] 손님 욕구 기반 시설 선택 구현
- [ ] 직원 근무, 보충, 대기 구현
- [ ] 운영일 정산 알림 구현
- [ ] 캐릭터 이모티콘과 클릭 로그 구현

성공 판정:

```text
손님이 들어온다
-> 시설을 고른다
-> 돈과 재고가 변한다
-> 직원이 보충/근무한다
-> 문제 발생 시 로그가 남는다
-> 운영일 정산에서 결과를 본다
```

### 2차 목표: 첫 침입

- [ ] 위협도 누적 구현
- [ ] 침입자 1종 구현
- [ ] 가시 함정, 독 웅덩이, 경비실 구현
- [ ] 침입자 이동과 시설 파손 구현
- [ ] 방어 시설 발동 로그 구현
- [ ] 침입 결과 알림과 상세 패널 구현
- [ ] 파손 시설 수리 작업 구현

성공 판정:

```text
침입 경고가 뜬다
-> 침입자가 들어온다
-> 방어 시설과 직원이 자동 대응한다
-> 일부 시설이 파손될 수 있다
-> 결과 요약에서 무엇이 작동했는지 본다
-> 직원이 수리한다
```

### 3차 목표: 연구와 합성

- [ ] 연구실 구현
- [ ] 설계도 분석 구현
- [ ] 1성 기본 구매 해금 구현
- [ ] 시설물끼리 합성 구현
- [ ] 식당/함정 대표 트리 1성에서 3성까지 구현
- [ ] 합성 결과 레벨 계승 구현
- [ ] 시설 도감에 합성 정보 표시

성공 판정:

```text
침입으로 약점이 보인다
-> 설계도/연구로 조합식이 열린다
-> 상점과 기본 구매로 필요한 시설을 모은다
-> 시설물끼리 합성한다
-> 다음 침입에서 결과 차이를 본다
```

### 4차 목표: 로그라이트 런

- [ ] 사장 종족 선택 구현
- [ ] 사장 사망 런 종료 구현
- [ ] 런 결과 정산 구현
- [ ] 계승 화폐 지급 구현
- [ ] 초기 계승 트리 구현
- [ ] 시작 선택지 확장 구현

성공 판정:

```text
런을 시작한다
-> 던전을 운영한다
-> 침입과 내부 문제를 버틴다
-> 사장이 죽으면 런이 끝난다
-> 생존 성과로 계승 화폐를 얻는다
-> 다음 런의 시작 선택지가 넓어진다
```

## 반복 검증 체크리스트

새 시스템을 붙일 때마다 다음을 확인한다.

- [ ] 손님이 멈추지 않고 행동한다.
- [ ] 직원이 멈추지 않고 작업하거나 대기한다.
- [ ] 사장이 AI와 우선 지정을 정상 수행한다.
- [ ] 경로가 끊기면 캐릭터 로그에 이유가 남는다.
- [ ] 재고 부족이 시설 이용 실패와 정산에 표시된다.
- [ ] 파손 시설은 이용/발동하지 않는다.
- [ ] 수리 후 시설이 다시 후보와 발동 대상에 들어온다.
- [ ] 이벤트 결과는 우하단/오른쪽 알림으로 남는다.
- [ ] 결과 화면은 추천 대응 대신 관측 사실을 보여준다.
- [ ] 새 데이터 추가가 코드 수정 없이 ScriptableObject 조립으로 가능한지 확인한다.
