# AI 구조 설계 노트

## 현재 방향

현재 게임 구조에서는 다음 구성을 기본 방향으로 둔다.

```text
Utility AI가 무엇을 할지 결정한다.
AIBrain은 점수 계산, 목적지 선정, 경로 탐색 캐시, 액션 상태를 조율한다.
Ability 클래스는 선택된 행동을 실제로 실행한다.
Behavior Tree는 행동 실행 분기가 너무 복잡해질 때까지 보류한다.
```

경영 루프가 아직 형성되는 단계에서는 이 구성이 가장 단순하고 추적하기 쉽다.

전투 효과 구현도 Visitor 패턴을 핵심으로 쓰지 않는다. 방어 시설 효과는 `Strategy + ScriptableObject Effect` 구조로 두고, 시설 데이터가 발동 조건, 대상 선택, 효과 목록을 조합하게 한다.

## 책임 분리

### Utility AI

Utility AI가 답해야 하는 질문:

```text
이 캐릭터는 지금 무엇을 해야 하는가?
```

예시:

- 쇼핑
- 식사
- 일하기
- 휴식
- 구경
- 던전 퇴장
- 방어
- 침입자 공격

DungeonStory의 NPC는 배고픔, 돈, 선호 상점, 스케줄, 만족도, 위험도, 거리, 이용 가능 여부처럼 부드러운 우선순위를 많이 가질 수 있다. 그래서 Utility AI가 잘 맞는다.

현재 세계관 기준으로는 몬스터 손님, 직원, 방어 몬스터, 모험가 침입자가 서로 다른 목적을 가진다. 같은 이동 시스템을 사용하더라도 점수 계산 기준은 다르게 둔다.

고객 AI의 구체적인 행동/목적지 선택 규칙은 [고객 AI 선택 규칙](game-design/customer-ai.md) 문서에서 관리한다.

### AIBrain

AIBrain이 답해야 하는 질문:

```text
어떤 행동이 선택되었고, 어디로 가야 하며, 어떤 경로 데이터를 재사용할 수 있는가?
```

현재 책임:

- 가능한 액션 점수 계산
- 선택된 액션 저장
- 플레이어가 지정한 우선 작업/대상 반영
- 목적지 결정
- 행동 점수와 목적지 후보 점수의 분리
- 한 번의 의사결정 안에서 BFS 탐색 결과 재사용
- 액션 실행 상태 보관
- 이후 Success, Failure, Replan 처리

### Ability

Ability 클래스가 답해야 하는 질문:

```text
선택된 행동을 게임 안에서 어떻게 수행할 것인가?
```

예시:

- 경로를 따라 이동
- 상점과 상호작용
- 아이템 구매
- 배정된 상점에서 근무
- 던전 퇴장

Ability는 장기적으로 다음과 같은 실행 결과를 반환하는 구조가 좋다.

```text
Success
Failure
Running
Cancelled
```

### Behavior Tree

Behavior Tree는 Utility AI와 동시에 고수준 행동을 결정하면 안 된다. 그러면 의사결정의 주인이 둘이 된다.

BT는 하나의 행동 내부 실행 분기가 깊어질 때만 유용하다.

좋은 BT 사용 예:

```text
Utility AI가 Shopping을 선택한다.
BT는 Shopping을 실행한다.
상점 찾기 -> 이동 -> 재고 확인 -> 구매 -> 실패하면 다른 상점 시도 -> 모두 실패하면 Failure 반환
```

위험한 BT 사용 예:

```text
Utility AI가 Shopping을 선택한다.
BT도 Shopping, Eating, Working, Leaving 중 무엇을 할지 다시 고른다.
```

이렇게 되면 두 시스템이 서로 다른 의도를 가질 수 있어 디버깅이 어려워진다.

## 단기 권장 흐름

```text
Character.Update
-> AIBrain.DecideAction
-> AIAction.CalculateScore
-> AIActionSet.GetDestination
-> AIBrain 경로 캐시 / GridPathSearchResult
-> AIActionSet.Execute
-> Ability 코루틴 실행
-> Ability가 Success 또는 Failure 보고
-> AIBrain이 완료 처리 또는 재계획
```

## 실패 처리 우선

BT를 추가하기 전에 명확한 실패 처리를 먼저 만든다.

대표 실패 상황:

- 목적지가 파괴됨
- 목적지에 도달할 수 없음
- 상점에 재고가 없음
- 캐릭터가 아이템을 살 돈이 없음
- 직원 배정이 유효하지 않음
- 배치 변경으로 경로가 무효화됨
- 액션이 너무 오래 걸림

권장 처리:

```text
실패 발생
-> 실패한 대상을 임시 제외
-> 경로 캐시 제거
-> DecideAction 다시 실행
```

예시:

```text
손님이 음식을 원함
-> 도달 가능한 음식점을 선택
-> 음식점으로 이동
-> 재고가 비어 있음
-> 해당 음식점을 이번 의사결정 구간에서 실패 대상으로 표시
-> 재계획
-> 다른 음식점을 시도하거나 퇴장
```

## Behavior Tree 도입 기준

다음 조건이 대부분 참이 될 때 BT 도입을 고려한다.

- 하나의 행동 안에 내부 단계가 5-7개 이상 있다.
- 하나의 행동 안에 fallback 분기가 여러 개 있다.
- 디자이너가 실행 흐름을 시각적으로 편집해야 한다.
- 코루틴이 중첩 조건 때문에 읽기 어려워진다.
- 여러 행동이 같은 실행 조각을 반복해서 공유한다.

그 전까지는 Utility AI와 Ability 코루틴 조합이 더 단순하다.

## 길찾기 전략

현재 그리드는 이동 비용이 균일하므로 BFS가 잘 맞는다.

캐시된 BFS 결과를 사용하기 좋은 경우:

- 여러 consideration이 도달 가능한 건물을 물어본다.
- 목적지 선택과 경로 생성이 같은 의사결정 안에서 일어난다.
- 그리드가 바뀌지 않았고 캐릭터가 다른 셀로 이동하지 않았다.

나중에 Dijkstra를 고려할 상황:

- 계단, 문, 함정, 군중, 방 종류에 따라 이동 비용이 달라진다.

나중에 A*를 고려할 상황:

- 고정된 목적지가 하나로 정해져 있다.
- 맵이 커져서 전체 BFS가 부담된다.

## 침입자 이동 AI

침입자는 모험가이며, 최종적으로 던전의 사장방과 플레이어 캐릭터를 노린다.

기본 원칙:

- 침입자도 손님과 같은 그리드, 복도, 계단 이동 규칙을 사용한다.
- 던전 구조 때문에 우회하거나 헤매는 장면은 허용한다.
- 하지만 너무 오래 헤매면 이벤트 템포가 나빠지므로 시간이 지날수록 사장방을 더 잘 찾게 한다.
- 침입자는 유형에 따라 사장방 돌파, 시설 파괴, 몬스터 처치, 자원 약탈 중 다른 목표 가중치를 가진다.
- 첫 구현은 사장방 돌파형 침입자 1종류로 시작하고, 다른 유형은 목표 가중치 데이터로 확장 가능하게 둔다.

추천 구조:

```text
침입자 최종 목표: 사장방 / 플레이어 캐릭터
침입자 보조 목표: 시설 파괴 / 몬스터 처치 / 자원 약탈
초기 이동: 탐색 성향 높음
시간 경과: 목표 방향/최단 경로 가중치 증가
장시간 경과: 거의 확정적으로 목표 경로 선택
```

구현 후보:

1. 같은 BFS 결과를 쓰되, 후보 이동 칸 중 목표까지의 거리가 짧은 칸에 점점 높은 점수를 준다.
2. 침입자마다 `pathKnowledge` 값을 두고 시간이 지날수록 증가시킨다.
3. `pathKnowledge`가 낮으면 탐색/랜덤성이 높고, 높으면 사장방 방향성이 강해진다.
4. 일정 시간 이상 지나면 완전한 최단 경로를 사용한다.
5. 침입자 유형별로 `bossRoomWeight`, `facilityDestroyWeight`, `monsterKillWeight`, `lootWeight`를 다르게 둔다.

첫 구현 기본값:

```text
bossRoomWeight: 높음
facilityDestroyWeight: 낮음 또는 제한적
monsterKillWeight: 낮음 또는 비활성
lootWeight: 비활성
```

예시:

```text
score = randomWeight + bossRoomWeight + facilityDestroyWeight + monsterKillWeight + lootWeight + dangerWeight

초반:
randomWeight 높음
bossRoomWeight 낮음

후반:
randomWeight 낮음
bossRoomWeight 높음
```

이 방식은 손님과 침입자가 같은 길찾기 기반을 공유하면서도, 침입 이벤트가 지나치게 늘어지는 것을 막는다.

## 침입 이벤트 스케일링

침입 이벤트는 고정 날짜/웨이브가 아니라 내부 위협도 누적 방식으로 발생한다.

림월드식 난이도/시간/가치 스케일링을 참고한다.

위협도는 시스템 내부 값이며, 플레이어에게 게이지로 그대로 표시하지 않는다.

AI 관점에서 필요한 값:

- 던전 가치
- 경과 시간
- 난이도 계수
- 최근 침입 결과
- 현재 방어력 또는 취약점

위협도 상승량과 침입 강도는 대략 다음 방향으로 계산한다.

```text
위협도 상승량 = 기본 상승량 + 던전 가치 보정 + 시간 보정 + 난이도 보정 + 랜덤 변동폭
침입 강도 = 현재 위협도 + 던전 가치 보정 + 시간 보정 + 난이도 보정 + 랜덤 변동폭
```

난이도는 던전 가치와 시간의 반영 비율을 조절한다. 쉬운 난이도는 성장하더라도 위협 증가가 느리고, 어려운 난이도는 부유하거나 오래 살아남은 던전에 더 강하게 반응한다.

권장 흐름:

```text
내부 위협도 70% 이상: 경고 상태
내부 위협도 100% 이상: 침입 후보 상태
100% 이후 짧은 랜덤 딜레이: 실제 침입 발생
침입 후: 위협도 초기화 + 안전 시간 부여
```

## 오펜스 AI

오펜스는 플레이어가 직원/몬스터를 데리고 다른 던전이나 세력을 공격하는 방향으로 둔다.

모든 전투는 자동으로 진행된다. 플레이어는 전투 중 직접 조작하지 않고, 편성/배치/성장/대상 선택으로 결과에 개입한다.

초기에는 핵심 루프에 넣지 않고, 경영과 디펜스가 안정된 뒤 설계한다.

오펜스 AI에서 필요할 수 있는 판단:

- 어떤 몬스터를 원정에 보낼 것인가?
- 원정 중 몬스터가 어떤 행동을 우선할 것인가?
- 전리품, 생존, 빠른 돌파 중 무엇을 우선할 것인가?
- 원정에 보낸 몬스터가 던전 방어에서 빠질 때 생기는 리스크를 어떻게 표시할 것인가?
- 월드맵에서 어떤 원정 대상이 현재 스캔 범위 안에 있는가?
- 스킬트리 업그레이드로 어떤 원정 대상 정보가 추가 공개되는가?

오펜스가 들어오면 Utility AI의 대상은 내 던전 내부 NPC뿐 아니라 원정 중인 몬스터까지 확장될 수 있다.

## 현재 권장안

아직 BT를 핵심 루프에 넣지 않는다.

우선순위:

1. Utility AI 액션 점수 계산
2. 목적지 선정
3. 캐시된 경로 탐색
4. Ability 실행
5. Success / Failure / Replan

BT는 이후 전투, 침입, 다단계 쇼핑 같은 복잡한 행동을 실행하는 도구로 도입한다.
