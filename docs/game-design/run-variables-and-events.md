# 런 변수와 이벤트

DungeonStory는 로그라이트식 생존 게임이므로, 매 런이 같은 순서로 흘러가면 안 된다.

플레이어가 같은 시설을 짓고, 같은 손님을 받고, 같은 침입을 막는 구조가 반복되면 덱빌딩과 계승의 재미가 약해진다.

따라서 중간중간 플레이 흐름을 흔드는 변수가 필요하다.

다만 변수는 랜덤 억까가 아니라, 플레이어가 읽고 대응할 수 있는 사건이어야 한다.

## 변수의 역할

변수는 다음 역할을 가진다.

- 같은 빌드가 매번 똑같이 굴러가지 않게 한다.
- 플레이어가 현재 던전 상태를 보고 우선순위를 바꾸게 한다.
- 사장 종족, 고객층, 시설 합성 선택이 런마다 달라지게 한다.
- 경영, 침입, 합성, 오펜스 사이에 작은 의사결정을 만든다.
- 결과 화면에 `이번 런만의 이야기`를 남긴다.

좋은 변수는 다음 질문을 만든다.

```text
지금은 돈을 아껴야 하나, 재고를 사야 하나?
이 손님층을 받아야 하나, 다른 손님층을 노려야 하나?
다음 침입이 올 것 같은데 합성을 먼저 해야 하나?
이 설계도는 지금 연구할 가치가 있나?
오펜스를 보내서 보상을 노릴까, 방어 인력을 남길까?
```

## 변수 계층

변수는 크게 세 계층으로 나눈다.

```text
런 시작 변수
운영 중 이벤트 변수
침입/보상 변수
```

### 런 시작 변수

런 시작 변수는 이번 런의 방향을 만든다.

예시:

```text
사장 종족
시작 시설 후보
시작 손님층 후보
시작 설계도 후보
초기 상점 상품
초기 던전 구조
난이도/위협 계수
```

이 변수는 덱빌딩의 시작 패에 해당한다.

사장 종족과 시작 시설이 다르면 같은 플레이어라도 초반 운영 방식이 달라진다.

예시:

```text
슬라임 사장 + 저가 음식점 + 휴식방
-> 대량 손님/재고 순환 중심

오크 사장 + 고기 식당 + 훈련장
-> 전투/경비/고소비 손님 중심

뱀파이어 사장 + 연구실 + 마력 저장소
-> 연구/설계도/특수 조합식 중심
```

### 운영 중 이벤트 변수

운영 중 이벤트는 던전 경영 중간에 발생하는 작은 흔들림이다.

종류:

- 손님층 변화
- 재고/납품 변화
- 상점 변동
- 시설 문제
- 직원 상태 변화
- 연구/설계도 기회
- 특수 손님 방문
- 내부 사고 전조

예시:

```text
슬라임 단체 방문:
슬라임 손님이 잠시 많이 온다. 저가 음식점과 휴식방 수요가 증가한다.

오크 회식:
오크 손님이 늘고 고기 식당/훈련장 수요가 증가한다. 혼잡하면 난동 위험이 오른다.

마력 재고 과잉:
마력 재고 가격이 내려간다. 마력 시설 운영과 연구가 유리하다.

식재료 납품 지연:
다음 운영일까지 식재료 구매 비용이 오른다. 음식점 재고 관리가 중요해진다.

방문 상인:
일부 시설이나 설계도를 할인 판매한다.

설계도 소문:
특정 시설군 설계도가 상점/침입/오펜스 보상에 등장할 확률이 오른다.
```

운영 중 이벤트는 즉시 큰 피해를 주기보다, 플레이어가 우선순위를 바꾸게 만드는 정도가 좋다.

### 침입/보상 변수

침입 이벤트도 매번 같은 형태면 안 된다.

초기에는 침입자 종류를 많이 늘리지 않더라도, 침입에 작은 변수를 붙일 수 있다.

예시:

```text
정찰 흔적:
침입 전 경고가 빨리 뜬다. 대신 침입 강도는 조금 높다.

무장한 침입자:
침입자의 방어력이 높다. 독/부식이나 화염 피해가 중요해진다.

급습:
경고 시간이 짧다. 침입 강도는 낮다.

약탈 우선:
사장방보다 가까운 시설 파손을 더 자주 시도한다.

피로한 침입자:
침입자 체력은 낮지만 수가 많다.
```

보상도 변수로 작동한다.

예시:

```text
침입 방어 보상:
시설, 설계도, 재고, 계승 정보 중 일부가 나온다.

오펜스 보상:
희귀 시설, 특수 설계도, 손님/직원 후보, 세력 약화 보상이 나온다.
```

보상 변수는 다음 빌드 방향을 바꾸는 역할을 해야 한다.

## 이벤트의 좋은 조건

이벤트는 다음 조건을 만족해야 한다.

```text
1. 발생 이유나 전조가 보인다.
2. 플레이어가 대응할 시간이 있다.
3. 최소 2개 이상의 대응 방법이 있다.
4. 결과가 로그나 정산에서 보인다.
5. 완전한 랜덤 패배를 만들지 않는다.
```

나쁜 이벤트:

```text
아무 경고 없이 핵심 시설 파손
돈이 없는데 강제 대형 비용 발생
초반에 대응 불가능한 강한 침입
특정 시설이 없으면 무조건 실패
```

좋은 이벤트:

```text
식재료 납품 지연
-> 재고를 아껴 쓰거나, 음식점 운영을 줄이거나, 비싼 재고를 살 수 있다.

오크 회식
-> 고기 식당을 보강하거나, 훈련장으로 분산하거나, 경비를 배치할 수 있다.

정찰 흔적
-> 방어 시설을 수리하거나, 경비를 배치하거나, 합성을 서두를 수 있다.
```

## 이벤트 표시 방식

이벤트는 플레이를 강제로 멈추는 모달보다, 오른쪽/우하단 알림 버튼으로 표시한다.

중요도에 따라 표현을 다르게 한다.

```text
낮음:
상점 변동, 손님층 소문, 작은 재고 가격 변화

중간:
특수 손님 방문, 설계도 소문, 납품 지연, 시설 이상

높음:
침입 경고, 반란 전조, 핵심 재고 부족, 사장 부상 위험
```

선택 이벤트는 플레이어가 눌렀을 때 2-3개의 선택지를 보여줄 수 있다.

예시:

```text
방문 상인 도착

선택 1: 가시 함정 구매
선택 2: 설계도 구매
선택 3: 그냥 보냄
```

이벤트 결과는 운영일 정산과 이벤트 로그에 남긴다.

## 변수 빈도

변수는 너무 자주 발생하면 피곤하다.

초기 방향:

```text
초반:
작은 운영 변수 위주. 큰 패널티 이벤트는 거의 없음.

중반:
손님층, 재고, 상점, 침입 변수가 함께 등장.

후반:
오펜스, 세력, 특수 침입, 고급 설계도 이벤트 등장.
```

운영일마다 반드시 큰 사건이 생길 필요는 없다.

운영일 정산은 리듬 장치이고, 이벤트는 그 사이에 플레이어의 우선순위를 흔드는 장치다.

## 초기 구현 추천

첫 구현에서는 변수 종류를 많이 만들지 않는다.

초기 변수 세트:

```text
손님층 변화:
슬라임 단체 방문, 오크 회식, 뱀파이어 야간 방문

재고 변화:
식재료 납품 지연, 잡화 할인, 마력 재고 과잉

상점 변화:
방문 상인, 방어 시설 할인, 설계도 소문

침입 변화:
정찰 흔적, 급습, 무장한 침입자
```

초기에는 선택 이벤트보다 상태 변화 이벤트를 먼저 만든다.

이유:

- UI가 단순하다.
- 자동 정산/로그와 연결하기 쉽다.
- 플레이어가 변수를 관찰하고 대응하는 감각을 먼저 검증할 수 있다.

선택 이벤트는 이후에 추가한다.

첫 검증 목표:

```text
같은 시작 조건이어도 운영 중 변수 때문에 다음 선택이 달라지는가?
변수가 억울한 랜덤 패널티가 아니라 대응 가능한 사건으로 느껴지는가?
정산/로그에서 이번 런만의 이야기가 보이는가?
```
