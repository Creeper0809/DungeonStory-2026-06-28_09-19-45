# 고객 AI 선택 규칙

고객 AI는 Behavior Tree가 아니라 Utility AI를 기본으로 한다.

현재 코드의 `AIBrain`, `AIAction`, `Consideration` 구조를 유지하되, 손님의 행동 선택과 목적지 선택을 더 명확하게 나눈다.

```text
1. 지금 무엇을 할지 고른다.
2. 그 행동을 어디서 할지 고른다.
3. 이동/상호작용 중 실패하면 로그를 남기고 재계획한다.
```

이 구조를 쓰면 손님이 식사, 구매, 휴식, 구경, 퇴장 중 무엇을 고르는지와 어떤 시설을 선택하는지가 한 덩어리로 꼬이지 않는다.

## 기본 방향

손님은 완벽하게 최적화된 로봇처럼 움직이지 않는다.

대신 다음 요소를 종합해 그럴듯한 선택을 한다.

- 현재 욕구
- 종족 선호
- 개인 특성
- 보유 금액
- 도달 가능 여부
- 재고 상태
- 가격
- 혼잡도
- 거리
- 최근 방문/실패 기록

중요한 점은 고정된 if/else 우선순위가 아니라, 사용할 수 없는 선택지를 먼저 걸러내고 남은 후보를 점수화하는 방식이라는 것이다.

```text
불가능한 후보 제거
-> 행동 점수 계산
-> 목적지 후보 점수 계산
-> 최고 점수 선택
-> 실행 중 실패하면 재계획
```

## 행동 선택

행동 선택은 `무엇을 할 것인가`를 정한다.

초기 고객 행동:

- 식사
- 구매
- 휴식
- 구경
- 퇴장

행동 점수는 캐릭터의 욕구와 상황을 중심으로 계산한다.

예시:

```text
식사 점수 = 배고픔 욕구 + 음식점 도달 가능 여부 + 음식 구매 가능성
구매 점수 = 쇼핑 욕구 + 선호 상점 도달 가능 여부 + 보유 금액
휴식 점수 = 피로/기분 욕구 + 휴식 시설 도달 가능 여부
구경 점수 = 특별히 강한 욕구가 없을 때의 기본 행동
퇴장 점수 = 만족 완료, 돈 부족, 반복 실패, 기분 저하
```

현재 코드의 `Character.Condition`은 초기 욕구 축으로 사용할 수 있다.

```text
HUNGER: 식사 욕구
SLEEP: 휴식 욕구
FUN: 구매, 오락, 구경 욕구
MOOD: 체류 의지, 불만, 퇴장 판단
```

욕구 점수는 선형 수치보다 커브를 쓰는 편이 좋다. 예를 들어 배고픔이 조금 낮다고 바로 식당에 가지 않고, 일정 구간부터 식사 점수가 급격히 올라가게 만들 수 있다.

현재 `ConsiderationStat`처럼 `AnimationCurve`로 욕구 점수를 계산하는 방식이 이 구조와 잘 맞는다.

## 목적지 선택

목적지 선택은 `선택한 행동을 어디서 할 것인가`를 정한다.

행동 선택이 `식사`라면 목적지 후보는 음식점 계열 시설이 된다. 행동 선택이 `구매`라면 상점 계열 시설이 후보가 된다.

목적지 선택은 다음 순서로 처리한다.

```text
1. 후보 수집
2. 불가능한 후보 제거
3. 후보별 점수 계산
4. 최고 점수 후보 선택
```

### 1. 후보 수집

후보는 기본적으로 현재 캐릭터 위치에서 도달 가능한 시설 목록에서 찾는다.

현재 구조에서는 한 번의 BFS 결과를 캐시하고, 그 결과에서 방문 가능한 시설을 가져오는 방식이 적합하다.

```text
AIBrain
-> GridPathSearchResult 캐시
-> GetAllVisitableBuilding
-> 행동에 맞는 시설만 후보로 사용
```

이 방식은 같은 판단 안에서 여러 consideration이 도달 가능 시설을 다시 계산하지 않게 해준다.

### 2. 불가능한 후보 제거

다음 조건은 점수 보정이 아니라 후보 제거에 가깝다.

- 도달할 수 없음
- 시설이 파손되어 비활성화됨
- 선택한 행동을 지원하지 않음
- 필수 재고가 0임
- 구매 행동인데 가격을 감당할 수 없음
- 현재 이용할 수 없는 상태임
- 같은 판단 안에서 이미 실패한 목적지임

예시:

```text
식사를 선택했는데 음식점 재고가 0이면 후보에서 제외한다.
구매를 선택했는데 어떤 상품도 살 돈이 없으면 후보에서 제외한다.
휴식을 선택했는데 휴식방이 파손 상태면 후보에서 제외한다.
```

다만 향후에는 `재고가 부족하지만 아직 이용 가능`, `비싸지만 살 수는 있음`, `혼잡하지만 기다릴 수 있음`처럼 0/1이 아닌 상태를 점수화할 수 있다.

초기 구현은 단순하게 시작한다.

```text
재고 0 = 후보 제외
돈 부족 = 후보 제외
파손 = 후보 제외
```

### 3. 후보별 점수 계산

후보가 남으면 다음 요소를 곱하거나 가중합해 점수를 만든다.

권장 기본 순서:

```text
욕구 적합도
-> 종족/개인 선호도
-> 재고 안정성
-> 가격 적합도
-> 혼잡도
-> 거리
-> 신규성/랜덤성
```

이 순서는 `항상 먼저 본다`는 의미가 아니라, 점수 영향의 기본 중요도를 뜻한다.

#### 욕구 적합도

현재 행동과 시설의 기능이 얼마나 잘 맞는지 본다.

예시:

```text
배고픈 손님에게 고기 식당은 높은 점수
피곤한 손님에게 휴식방은 높은 점수
FUN이 높은 손님에게 잡화점, 오락 시설은 높은 점수
```

#### 종족/개인 선호도

종족 특성과 개인 특성은 목적지 점수를 크게 바꾼다.

예시:

```text
오크: 고기 식당, 무기점, 훈련장 선호
슬라임: 저가 음식점, 휴식방(습기) 선호
뱀파이어: 휴식방(어둠), 마력 저장소 선호
대식가: 식당 계열 점수 증가
절약가: 비싼 시설 점수 감소
호기심 많음: 아직 방문하지 않은 시설 점수 증가
```

선호도는 살아있는 던전을 만드는 핵심이므로 거리보다 강하게 작동해야 한다.

즉 오크가 조금 더 멀리 있는 고기 식당을 택하는 장면은 허용한다.

#### 재고 안정성

재고는 시설 선택의 신뢰도를 만든다.

```text
재고 충분: 점수 높음
재고 낮음: 점수 약간 감소
재고 부족: 점수 크게 감소
재고 없음: 후보 제외
```

재고가 부족한 시설을 계속 고르면 손님이 헛걸음하고 로그가 지저분해진다. 따라서 손님은 재고가 안정적인 시설을 선호해야 한다.

#### 가격 적합도

가격은 보유 금액과 개인 성향을 같이 본다.

```text
살 수 없음: 후보 제외
살 수 있지만 비쌈: 점수 감소
예산에 적당함: 점수 유지
선호 시설이고 예산 충분: 점수 증가
```

특성 예시:

```text
부자: 비싼 시설 감점 감소
절약가: 비싼 시설 감점 증가
데몬: 고가 시설 선호 보정
슬라임: 저가 시설 선호 보정
```

#### 혼잡도

혼잡도는 대기 시간과 불만으로 이어진다.

```text
한산함: 점수 높음
보통: 점수 유지
혼잡: 점수 감소
만석/대기열 과다: 후보 제외 또는 큰 감점
```

혼잡도 민감도는 개인 특성으로 보정한다.

```text
성급함: 대기열 감점 증가
소심함: 혼잡 감점 증가
오크: 약간의 혼잡 감점 감소
데몬: 혼잡 감점 증가
```

#### 거리

거리는 마지막에 조정값으로 사용한다.

거리가 너무 강하면 모든 손님이 가까운 시설만 쓰게 되고, 종족 선호와 던전 구성이 흐려진다.

권장 방향:

```text
가까운 시설은 약간 유리하다.
하지만 선호도, 재고, 가격 조건이 좋으면 먼 시설도 선택한다.
너무 멀면 체류 시간과 불만에 영향을 준다.
```

#### 신규성/랜덤성

손님이 매번 같은 시설만 고르면 던전이 기계적으로 보인다.

아주 작은 랜덤성과 신규성 보정을 둔다.

```text
아직 방문하지 않은 시설: 소폭 증가
최근 실패한 시설: 감소 또는 임시 제외
무작위 변동: 소폭 적용
```

랜덤성은 선택을 흔들 정도로만 쓰고, 욕구와 선호를 뒤집을 정도로 크게 두지 않는다.

## 행동별 판단 차이

모든 행동이 같은 비중을 쓰면 안 된다.

### 식사

식사는 욕구 긴급도가 높다.

```text
중요도:
재고/가격
-> 욕구 적합도
-> 종족/개인 선호
-> 혼잡도
-> 거리
```

배고픈 손님은 완벽히 선호하는 식당이 아니어도 먹을 수 있는 곳을 찾아간다.

다만 선호 식당이 도달 가능하고 재고가 있다면 선호 식당을 고른다.

### 구매

구매는 선호와 가격이 중요하다.

```text
중요도:
종족/개인 선호
-> 가격 적합도
-> 재고 안정성
-> 혼잡도
-> 거리
```

쇼핑은 필수 생존 욕구보다 취향이 강하게 작동한다.

### 휴식

휴식은 시설 태그와 혼잡이 중요하다.

```text
중요도:
환경 태그 적합도
-> 혼잡도
-> 종족/개인 선호
-> 거리
```

예시:

```text
슬라임은 휴식방(습기)을 선호한다.
뱀파이어는 휴식방(어둠)을 선호한다.
직원은 휴식방(직원 우선)을 선호한다.
```

### 구경

구경은 강한 욕구가 없거나, 다른 행동 후보가 애매할 때 선택된다.

```text
중요도:
신규성
-> 종족/개인 선호
-> 거리
-> 혼잡도
```

구경은 던전이 살아 보이게 만드는 완충 행동이다. 하지만 너무 자주 발생하면 손님이 돈을 쓰지 않는 것처럼 보이므로, 초반에는 낮은 비중으로 둔다.

### 퇴장

퇴장은 실패 누적과 목표 완료의 결과다.

퇴장 점수가 올라가는 조건:

- 돈을 거의 다 씀
- 만족도가 충분히 높고 방문 목적을 달성함
- 반복해서 목적지 선택에 실패함
- 원하는 시설이 없음
- 재고 부족을 여러 번 겪음
- 길 막힘을 겪음
- 기분이 너무 낮음

퇴장은 단순 실패가 아니라 자연스러운 방문 종료로도 사용해야 한다.

## 실패와 재계획

실행 중 실패가 발생하면 캐릭터 로그에 짧은 원인 태그를 남기고 재계획한다.

실패 예시:

```text
재고 없음
돈 부족
길 막힘
목적지 파손
혼잡함
대기 길음
선호 시설 없음
```

처리 흐름:

```text
실패 발생
-> 로그 기록
-> 만족도 또는 기분 보정
-> 실패한 목적지를 임시 제외
-> 경로 캐시 무효화
-> 행동 재선택
```

같은 실패가 반복되면 퇴장 점수와 사고 위험이 올라간다.

## 로그와 감정 표시

선택 규칙은 플레이어에게 전부 수치로 공개하지 않는다.

대신 캐릭터 클릭 로그와 이모티콘 풍선으로 원인을 보여준다.

화면 위 표시:

```text
기쁨: 원하는 시설 이용
불만: 대기 길음, 비쌈, 선호 시설 없음
당황: 길 막힘, 재고 없음
분노: 불만 누적, 사고 직전
```

캐릭터 로그:

```text
고기 식당 선택: 오크 선호
잡화점 포기: 돈 부족
저가 음식점 포기: 재고 없음
휴식방 선택: 피로 높음
퇴장 결정: 방문 목표 완료
```

로그는 플레이어가 “왜 저 손님이 저기로 갔는지”를 이해할 수 있을 정도면 충분하다.

## 현재 구현과 확장 방향

현재 구조에서 이미 맞는 부분:

- `AIBrain`이 액션 점수를 계산한다.
- `AIAction`이 `Consideration` 점수를 종합한다.
- `ConsiderationStat`이 욕구 기반 점수를 만들 수 있다.
- BFS 결과를 캐시해 도달 가능한 시설을 재사용할 수 있다.
- `AIEat`, `AIShopping`, `AIExitDungeon` 같은 행동 단위가 이미 분리되어 있다.

보완이 필요한 부분:

- 행동 점수와 목적지 점수를 분리한다.
- 목적지 후보를 `첫 번째 도달 가능 시설`이 아니라 점수로 고른다.
- 재고, 가격, 혼잡, 거리, 선호도 consideration을 추가한다.
- 실패한 목적지를 임시 제외하는 재계획 데이터를 둔다.
- 캐릭터 로그에 선택/실패 원인을 남긴다.

초기 구현 추천:

```text
행동 선택:
HUNGER, FUN, SLEEP, MOOD 기반 Utility AI 유지

목적지 선택:
도달 가능
재고 있음
돈 있음
종족 선호
거리
작은 랜덤
```

혼잡도와 가격 세부 평가는 첫 구현 후 손님 흐름이 보이기 시작할 때 추가해도 된다.

## 구현 우선순위

1. 행동 점수와 목적지 점수 분리
2. 도달 가능한 시설 후보 수집 공통화
3. 시설 후보 점수 계산기 추가
4. 재고 없음/돈 부족/파손/길 막힘 실패 처리
5. 캐릭터 로그 태그 추가
6. 혼잡도와 대기열 점수 추가
7. 종족/개인 특성별 세부 보정 추가
