# 캐릭터 모델

이 게임의 몬스터는 단순한 역할 단위가 아니라 하나의 캐릭터다.

같은 개체가 상황에 따라 손님, 직원, 방어 인력, 원정 인력으로 바뀔 수 있으므로 캐릭터 데이터는 역할보다 위에 있어야 한다.

## 기본 구조

각 캐릭터는 세 층의 데이터를 가진다.

```text
캐릭터 = 능력 + 개인 특성 + 종족 특성
```

- 능력: 캐릭터 개인이 가진 기본 수치다.
- 개인 특성: 같은 종족 안에서도 개체별 차이를 만드는 변주다.
- 종족 특성: 종족 공통의 성향, 선호, 약점, 작업 적성이다.

이 구조의 핵심은 종족이 모든 것을 결정하지 않는다는 점이다.

예를 들어 뱀파이어는 연구에 유리한 종족 보정을 가질 수 있지만, 개인 연구 능력이 낮고 산만함 특성이 있으면 뛰어난 연구원이 아닐 수 있다. 반대로 오크는 연구 종족 보정이 낮아도 개인 연구 능력이 높고 연구가 특성이 있으면 쓸 만한 연구원이 될 수 있다.

## 사장 캐릭터

플레이어 캐릭터, 즉 사장도 몬스터 종족 안에서 선택한다.

사장은 고정된 인간형 주인공이 아니라, 던전을 직접 운영하는 몬스터 사장이다.

런 시작 시 플레이어는 사장의 종족과 일부 시작 특성을 고른다.

```text
사장 = 능력 + 개인 특성 + 종족 특성 + 사장 역할
```

사장은 일반 직원과 같은 캐릭터 모델을 사용하지만, 다음 점이 다르다.

- 플레이어 캐릭터가 사망하면 런이 종료된다.
- 사장은 이탈하거나 반란을 일으키지 않는다.
- 초반에는 직접 노동력을 제공한다.
- 중후반에는 직원과 시설에 일을 위임하고, 최종 방어 코어 역할이 커진다.

### 사장 종족 선택

사장 종족은 시작 빌드를 정하는 첫 번째 선택이다.

초기에는 실제 구현 종족에 맞춰 3종을 먼저 제공한다.

```text
슬라임 사장:
청소, 보충, 유지보수, 대량 손님 운영에 유리

오크 사장:
수리, 경비, 전투, 고기 식당/훈련 계열 운영에 유리

뱀파이어 사장:
연구, 마력, 장기 운영, 특수 조합식 해금에 유리
```

후반 계승 또는 해금으로 다음 사장 종족을 열 수 있다.

```text
수인 사장:
이동, 보충, 상점 회전, 정찰에 유리

데몬 사장:
마력, 고급 시설, 강한 최종 교전, 높은 위험/보상 운영에 유리
```

사장 종족은 단순 스킨이 아니라 다음 값에 영향을 준다.

- 시작 능력치
- 선호 시설
- 초반 작업 효율
- 사장방/최종 교전 성향
- 시작 손님층 또는 상점 후보
- 계승 트리에서 열리는 특화 강화

### 초반 사장 작업

초반에는 직원이 부족하므로 사장이 직접 움직여야 한다.

가능한 초반 작업:

- 상점 또는 식당 근무
- 재고 보충
- 파손 시설 수리
- 연구시설에서 설계도 분석
- 침입 시 전투 참여
- 불만 캐릭터 진정 또는 협상
- 긴급 작업 우선 지정 수행

이 구조는 초반 플레이어가 사장을 통해 던전 운영을 직접 느끼게 만든다.

```text
초반:
사장이 직접 뛰며 던전을 버틴다.

중반:
직원을 영입하고 작업을 분산한다.

후반:
사장은 직접 노동보다 사장방 방어, 명령, 최종 생존의 의미가 커진다.
```

사장도 피로와 부상 영향을 받는다.

다만 일반 직원처럼 불만으로 이탈하지는 않는다.

방향:

```text
사장이 과로하면 작업 효율이 떨어진다.
사장이 부상당하면 최종 교전 위험이 커진다.
사장이 사망하면 런이 끝난다.
```

### 사장 조작 방식

사장도 기본적으로 AI가 자동 작업을 수행한다.

플레이어는 직원과 마찬가지로 사장을 선택해 우선 작업을 지정할 수 있다.

예시:

```text
사장 선택
-> 파손된 가시 함정 우클릭
-> 사장이 수리 우선 작업 수행
```

사장은 초반의 만능 노동력이지만, 모든 일을 혼자 해결할 수 있으면 직원 시스템이 죽는다.

따라서 사장 작업은 강하되 다음 한계를 둔다.

- 피로 누적
- 이동 시간
- 부상 위험
- 동시에 한 작업만 수행
- 최종 방어 코어이므로 위험 지역에 오래 두는 리스크

이 한계 덕분에 플레이어는 결국 직원 확보와 작업 우선순위를 필요로 하게 된다.

## 캐릭터 확보 루프

몬스터 직원과 방어 인력은 별도의 고용 풀에서 갑자기 생기지 않는다.

기본 확보 루프는 몬스터 손님이 던전에 만족하고, 단골이 되고, 그중 일부를 영입하는 구조다.

```text
몬스터 손님 방문
-> 시설 이용
-> 만족도/단골도 상승
-> 단골 등록
-> 영입 후보 등장
-> 직원/방어 인력/원정 인력으로 전환
```

이 구조의 장점:

- 경영이 캐릭터 확보로 이어진다.
- 자주 보던 손님을 영입하므로 캐릭터 애착이 생긴다.
- 좋은 고객층을 노리는 것이 곧 좋은 인력 풀을 만드는 일이 된다.
- 손님을 직원으로 영입하면 소비 고객 하나를 잃는 대신 운영/방어 인력을 얻는 선택이 된다.
- 종족별 고객 전략이 방어와 오펜스 전략으로 자연스럽게 연결된다.

영입 가능 조건 예시:

```text
단골도 일정 이상
최근 만족도 일정 이상
해당 종족이 선호하는 시설 보유
직원 숙소 또는 계약 슬롯 여유
영입 제안 조건 충족
```

초기 구현에서는 조건을 단순하게 둔다.

```text
최근 방문 횟수 + 평균 만족도
```

이 두 값만으로도 `좋은 서비스를 제공한 손님이 우리 편이 된다`는 핵심 감각을 만들 수 있다.

## 근무와 생활

영입된 몬스터는 완전히 일만 하는 유닛으로 고정되지 않는다.

직원, 방어 인력, 원정 인력도 비번이 되면 던전 안에서 생활 캐릭터처럼 행동한다.

근무와 비번 전환은 기본적으로 AI가 자동으로 처리한다.

```text
근무 상태
-> 피로/욕구 누적
-> 비번 상태
-> 시설 이용
-> 만족도/피로 회복
-> 다시 근무 가능
```

비번 캐릭터의 행동:

- 식사
- 휴식
- 선호 시설 방문
- 훈련
- 치료
- 사교 또는 구경
- 불만 누적 시 사고 위험 증가

비번 캐릭터는 손님처럼 시설을 이용하지만 일반 손님처럼 매출을 만들지는 않는다.

직원/방어 인력은 월급처럼 돈이 직접 빠지는 유지비를 기본으로 두지 않는다.

대신 다음 비용과 압박이 사실상의 유지비가 된다.

- 시설 수용량을 차지한다.
- 재고나 식사를 소비할 수 있다.
- 동선을 혼잡하게 만들 수 있다.
- 만족도가 낮으면 근무 효율이 떨어질 수 있다.
- 불만이 계속 쌓이면 직원 사고나 이탈 위험이 생긴다.

이 구조의 의미:

- 직원 수가 늘어날수록 숙소, 식당, 휴식 시설이 중요해진다.
- 좋은 직원을 많이 뽑는 것만으로는 충분하지 않고, 유지할 생활 기반이 필요하다.
- 경영 시설은 손님 수익뿐 아니라 직원 유지에도 쓰인다.
- 방어 인력을 늘리면 방어력은 오르지만 내부 소비와 혼잡도도 커진다.
- 돈이 빠지는 월급보다 던전 내부 운영 압박으로 유지비를 느끼게 한다.
- 플레이어가 시간표를 직접 짜기보다, 작업 우선순위와 시설 배치로 직원 생활을 간접 제어한다.

자동 비번 판단 기준:

```text
피로 높음
직원 만족도 낮음
식사/휴식 욕구 높음
부상 또는 회복 필요
현재 작업 긴급도 낮음
```

긴급 상황에서는 자동 비번보다 대응 작업이 우선될 수 있다.

예를 들어 침입, 화재, 난동이 발생하면 경비 우선순위와 상태가 맞는 직원은 비번 중이라도 대응에 합류할 수 있다.

## 직원 불만 단계

영입된 몬스터도 만족도가 낮아지면 문제를 일으킨다.

직원 불만은 단순 효율 저하로 끝나지 않고, 방치하면 결근, 사고, 이탈, 반란까지 이어질 수 있다.

```text
만족도 낮음
-> 효율 저하
-> 태업/결근/사고
-> 이탈
-> 반란/내부 전투
```

단계별 의미:

```text
1단계 - 효율 저하:
작업 속도 감소, 이동 지연, 작업 실패 확률 증가

2단계 - 태업/결근/사고:
작업 우선순위 무시, 근무 거부, 시설 파손, 동료와 충돌

3단계 - 이탈:
던전을 떠나며 영구 인력 손실 발생

4단계 - 반란/내부 전투:
일부 몬스터가 적대 상태가 되거나 내부 전투 이벤트 발생
```

이탈한 몬스터는 기본적으로 재영입할 수 없다.

반란을 일으킨 몬스터도 진압 이후 다시 회유하는 것을 기본 루프로 두지 않는다.

즉 이탈과 반란은 관리 실패가 만든 영구 손실이다.

반란은 자주 터지는 일반 사고가 아니라, 여러 경고를 무시했을 때 발생하는 고위험 내부 이벤트다.

반란이 발생하면 외부 침입과 다른 성격의 위협이 된다.

```text
외부 침입: 모험가가 던전 밖에서 들어온다.
직원 반란: 던전 내부 인력이 적대화된다.
```

반란은 발생 즉시 사장방/플레이어 캐릭터를 노리는 구조로 두지 않는다.

처음에는 반란이 발생한 위치 주변에서 국지적인 피해를 만들고, 방치되면 점점 위험한 목표로 확산된다.

반란 진행 흐름:

```text
불만 폭발
-> 주변 시설 파손
-> 가까운 동료 공격 또는 충돌
-> 던전 탈출 시도
-> 제압 실패/장기 방치
-> 사장방 또는 플레이어 캐릭터 위협
```

이 구조의 목적:

- 반란이 외부 침입과 다른 내부 사고처럼 느껴진다.
- 플레이어가 초기에 격리하거나 피해를 줄일 기회가 생긴다.
- 시설 배치와 구역 분리가 내부 위협 관리에도 의미를 가진다.
- 사장방 위협은 최종 확산 단계로 남겨 긴장감을 만든다.

반란 목표 우선순위 예시:

```text
초기: 가까운 시설 파손, 근처 몬스터와 충돌
중기: 출구 또는 안전 구역으로 탈출 시도
후기: 제압되지 않으면 사장방/플레이어 캐릭터 쪽으로 확산
```

### 반란 대응 방식

반란 대응은 여러 방식이 가능해야 한다.

핵심은 플레이어가 직접 전투를 조작하는 것이 아니라, 긴급 명령과 던전 구조로 자동 행동을 유도하는 것이다.

대응 방식:

```text
자동 제압: 경비/방어 몬스터가 반란 몬스터를 자동으로 제압한다.
제압 명령: 플레이어가 반란 몬스터나 구역을 지정해 경비 인력의 우선 대응을 지시한다.
격리: 문, 잠금 구역, 병목, 방 배치로 반란 몬스터를 가두거나 피해 범위를 줄인다.
협상/진정: 반란 직전 또는 초기 단계에서 비전투 방식으로 확산을 막는다.
```

제압 명령은 수동 전투 조작이 아니다.

플레이어는 `저 대상을 제압하라`, `저 구역을 봉쇄하라` 같은 목표를 지정하고, 실제 이동과 전투는 AI가 자동으로 처리한다.

협상/진정은 영구 손실 원칙과 충돌하지 않게 제한한다.

```text
반란 직전: 달래기/협상 가능
반란 초기: 진정 시도 가능
완전 적대화/진압 이후: 재영입 불가
```

협상/진정에 영향을 주는 값:

- 영업 능력
- 같은 종족 여부
- 개인 특성
- 반란 원인의 해결 여부
- 주변 위협도
- 해당 캐릭터와의 단골/근무 관계

예시:

```text
오크 직원이 과로와 식사 부족으로 폭발 직전
-> 플레이어가 진정 명령
-> 영업 능력이 높은 직원이 접근
-> 고기 식당 재고가 충분하면 진정 성공률 증가
```

직원 불만 대응 수단:

- 비번과 휴식 보장
- 종족 선호 시설 제공
- 식사/재고 부족 해결
- 혼잡 완화
- 사고를 일으키기 쉬운 종족/특성 관리
- 위험한 직원을 방어 핵심 구역에서 분리
- 반란 발생 위치 주변을 격리하거나 경비 인력을 투입
- 반란 직전/초기 단계에서 협상이나 진정 시도

영구 손실이 억울하게 느껴지지 않도록, 이탈과 반란 전에는 충분한 경고가 필요하다.

경고 예시:

```text
불만 이모티콘 반복
캐릭터 로그에 같은 불만 원인 누적
근무 효율 저하 표시
태업/결근 선행 발생
오른쪽/우하단 위험 알림
운영일 정산의 직원 불만 요약
```

초기 구현에서는 네 단계를 모두 데이터 상태로 둔다.

다만 첫 버전에서는 반란을 자주 발생시키기보다, 매우 낮은 만족도와 반복 경고 이후에만 발생하도록 한다.

초기 구현에서는 생활 시뮬레이션을 깊게 만들지 않는다.

우선 다음 값만 관리한다.

```text
피로
직원 만족도
시설 점유
간단한 욕구: 식사/휴식
```

이 정도만 있어도 직원이 던전 안에서 살아가는 느낌을 줄 수 있다.

## 능력

능력은 캐릭터가 실제 행동을 수행할 때 쓰는 기본 수치다.

초기 능력 후보:

```text
공격: 침입자와 전투할 때 주는 피해
맷집: 전투 지속력, 부상 저항, 쓰러지기까지 버티는 힘
영업: 상점 근무, 손님 응대, 판매 효율
연구: 설계도 분석, 조합식 해금, 연구 작업 속도
손재주: 수리, 시설 조작, 함정 관리
청소: 오염, 사고 흔적, 더러움 처리 속도
이동속도: 던전 안 이동, 작업 전환, 침입 대응 속도
힘: 한 번에 들 수 있는 물품량, 재고 보충 효율
지구력: 피로 누적 저항, 장시간 근무/전투 지속력
```

초기 구현에서는 모든 능력을 깊게 쓰지 않는다.

우선순위:

```text
1차 구현: 공격, 영업, 연구, 이동속도, 힘
2차 구현: 맷집, 손재주, 청소, 지구력
```

이렇게 하면 경영과 디펜스에 필요한 차이는 만들면서도, 첫 버전의 수치 관리 부담을 줄일 수 있다.

## 개인 특성

개인 특성은 캐릭터의 수치와 행동 경향을 살짝 바꾼다.

역할:

- 특정 능력 보정
- 특정 작업 선호
- 만족도 변화 보정
- 사고 확률 보정
- 연구/원정/전투 적성 보정
- 손님일 때 소비 성향 보정

예시:

```text
대식가: 음식 소비 증가
절약가: 소비 감소, 만족도 하락 속도 감소
부자: 소비 금액 증가
성급함: 대기 시간 불만 증가, 긴 작업 효율 감소
소심함: 혼잡도 불만 증가
싸움꾼: 전투/훈련 선호, 공격 보정, 사고 확률 증가
깔끔함: 오염 시설 기피, 청소 작업 보정
호기심 많음: 새 시설 방문 확률 증가, 연구/조사 선호 증가
연구가: 연구 작업 속도 증가
산만함: 연구 작업 속도 감소, 목적 변경 빈도 증가
```

초기에는 한 캐릭터가 개인 특성을 0-2개 가진다.

특성은 복잡한 예외 규칙보다 수치 보정과 행동 선호 중심으로 둔다.

## 종족 특성

종족 특성은 같은 종족이 공유하는 기본 성향이다.

역할:

- 선호 시설
- 기피 시설
- 소비 성향
- 체류 성향
- 충돌 대상
- 시너지 대상
- 사고 위험
- 작업 적성 보정
- 전투/방어 성향

예시:

```text
오크
- 선호: 무기점, 고기 식당, 훈련장
- 작업 보정: 공격, 힘, 수리, 경비 높음
- 작업 약점: 연구 낮음
- 위험성: 불만이 쌓이면 난동

뱀파이어
- 선호: 휴식방(어둠), 어둠 시설, 마력 저장소
- 작업 보정: 연구, 마력 분석, 장시간 근무 높음
- 작업 약점: 청소, 운반 낮음
- 위험성: 특정 손님에게 공포

슬라임
- 선호: 휴식방(습기), 저가 음식점
- 작업 보정: 청소, 오염 처리 높음
- 작업 약점: 공격, 힘 낮음
- 위험성: 방치하면 시설 오염
```

종족 특성은 캐릭터의 방향성을 만들지만 정답을 고정하지 않는다.

## 최종 행동 계산

캐릭터가 실제 행동을 할 때는 다음 값들이 함께 적용된다.

```text
최종 효율 = 개인 능력 + 개인 특성 보정 + 종족 특성 보정 + 시설/환경 보정 + 상태 보정
```

상태 보정 예시:

```text
피로
부상
만족도
공포
분노
혼잡
버프/디버프
```

예시:

```text
뱀파이어 연구원
- 연구 능력 높음
- 연구가 특성
- 뱀파이어 연구 보정
- 마력 연구실 보정
=> 설계도 분석 속도 매우 빠름
```

```text
오크 직원
- 공격 능력 높음
- 싸움꾼 특성
- 오크 경비 보정
- 훈련장 시너지
=> 침입 시 빠르게 전투에 합류하면 강함
```

```text
슬라임 직원
- 이동속도 보통
- 힘 낮음
- 슬라임 청소 보정
- 습한 구역 만족도 높음
=> 운반은 느리지만 오염 처리에는 좋음
```

## 역할별 사용

같은 캐릭터 데이터라도 현재 역할에 따라 사용하는 값이 다르다.

손님일 때:

- 종족 선호 시설
- 개인 특성
- 소비 성향
- 만족도 변화
- 이동속도
- 사고 위험

직원일 때:

- 작업 우선순위
- 근무/비번 상태
- 영업
- 연구
- 손재주
- 청소
- 이동속도
- 힘
- 지구력
- 피로
- 직원 만족도

방어 인력일 때:

- 공격
- 맷집
- 이동속도
- 지구력
- 전투 관련 개인 특성
- 종족 전투 보정

원정 인력일 때:

- 공격
- 맷집
- 이동속도
- 힘
- 지구력
- 원정 관련 개인 특성
- 종족 전투/탐색 보정

## UI 원칙

캐릭터 정보 UI는 세 층을 분리해서 보여줘야 한다.

```text
능력: 이 캐릭터가 실제로 잘하는 일
개인 특성: 같은 종족 안에서 다른 점
종족 특성: 종족 공통 성향
```

플레이어가 봐야 하는 정보:

- 이 캐릭터가 어떤 작업에 강한지
- 어떤 작업을 싫어하거나 못 하는지
- 손님일 때 어떤 시설을 좋아하는지
- 직원일 때 어떤 우선순위가 어울리는지
- 방어/원정에 보내도 되는지

다만 UI가 정답을 직접 추천하지는 않는다.

예를 들어 `이 캐릭터를 연구원으로 쓰세요`가 아니라, 연구 능력과 연구 관련 보정이 보여서 플레이어가 직접 판단하게 한다.

## 초기 구현 원칙

- 능력은 1차 구현에서 5개 정도만 실제 시스템에 연결한다.
- 개인 특성은 0-2개만 부여한다.
- 종족 특성은 보정과 선호 중심으로 시작한다.
- 모든 수치를 한 번에 깊게 시뮬레이션하지 않는다.
- 캐릭터 클릭 UI에서 능력, 개인 특성, 종족 특성을 분리해 확인할 수 있게 한다.
- 손님, 직원, 방어 인력, 원정 인력은 같은 캐릭터 데이터를 공유한다.
