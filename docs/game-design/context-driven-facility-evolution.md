# 컨텍스트 기반 시설 계보 진화 최종안

## 목적

이 문서는 시설 합성, 방 파츠, 운영 기록, LLM 해석을 하나의 성장 시스템으로 정리한 최종 방향 문서다.

기존 논의에서 나온 문제는 다음 네 가지였다.

- `시설 + 시설 = 상위 시설`은 던전메이커식 합성 감각은 좋지만, 림월드식 방/가구 파츠 구조와 충돌한다.
- `방 + 방 = 상위 방`은 방이 결과물이 되어버려, 벽/문/가구/동선으로 방을 만드는 감각과 맞지 않는다.
- 모든 가구마다 조합식을 만들면 콘텐츠 수가 폭발하고, 무엇은 합성 가능하고 무엇은 아닌지 감각이 어색해진다.
- LLM이 결과 시설을 마음대로 만들면 안전하지 않고, 반대로 flavor text만 쓰면 차세대적인 의미가 약하다.

최종 방향은 다음이다.

```text
현재 배치된 시설
+ 방의 물리 구조
+ 가구와 기물의 밀도/품질
+ 고객 이용 기록
+ 직원 이용 기록
+ 사건/범죄/소문/평판
+ 위생/배변/청결 기록
+ 재고/매출/운영 성과
+ 연구/설계도/재료
+ LLM 맥락 해석
-> 같은 위치의 다음 계보 시설로 진화
```

한 줄로 말하면:

```text
방을 합성하는 게임이 아니라, 방의 배치와 운영사가 현재 시설의 계보를 밀어 올리는 게임.
```

가장 중요한 판정 감각은 `수량이 만든 업태`다.

```text
테이블과 의자가 많음
-> 대중 식당 확정
```

이렇게 보지 않는다.

```text
테이블과 의자가 많음
+ 높은 좌석 밀도
+ 빠른 회전율
+ 낮은/중간 객단가
+ 불만이 관리된 많은 손님
-> 대중 식당 후보 강화
```

반대로 다음처럼 본다.

```text
테이블과 의자가 적음
+ 넓은 좌석 간격
+ 높은 좌석당 장식/서비스
+ 높은 평균 지불액
+ 고소비 손님 반복 이용
-> 고급 식당 후보 강화
```

즉 가구 수량은 출발점이고, 최종 판정은 방의 물리 구조, 이용 기록, 고객층, 사건, 위생, 보안, 평판이 함께 만든 `업태 정체성`으로 한다.

상세 판정 규칙은 [방 맥락 기반 시설 계보 진화](room-context-lineage-evolution.md)를 기준으로 한다.

## 이번 결정: 방 맥락이 시설을 분화시킨다

최종 형태는 `방 등급 업그레이드`가 아니다.

방은 계속 벽, 문, 가구, 동선, 청결, 보안, 손님 흐름을 담는 공간이고, 성장하는 대상은 그 방 안에서 운영 중인 시설이다.

```text
방의 구조와 운영 방식이
현재 시설의 업태를 증명하고
그 업태가 다음 시설 계보 후보를 연다.
```

따라서 식당 예시는 다음처럼 읽는다.

```text
테이블과 의자가 많음
+ 작은/중간 방
+ 좌석 밀도 높음
+ 회전율 높음
+ 객단가 낮음/중간
+ 대기열과 위생 불만이 통제됨
-> 대중 식당 계보
```

```text
테이블과 의자가 적음
+ 넓은 좌석 간격
+ 좌석당 장식/서비스 높음
+ 평균 지불액 높음
+ 고소비 손님 반복 방문
+ 조용하고 깨끗한 평판
-> 고급 식당 계보
```

```text
테이블과 의자가 많음
+ 큰 방
+ 좌석 간격이 답답하지 않음
+ 단체 손님 기록
+ 체류 시간이 길고 행사/회식 기록이 있음
-> 연회장 계보
```

즉 `의자 수`는 결과가 아니라 신호다.

같은 의자 수라도 방 크기, 좌석 밀도, 회전율, 객단가, 손님층, 불만, 위생, 직원 스트레스, 소문에 따라 전혀 다른 업태가 된다.

### 컨텍스트 반영 우선순위

모든 컨텍스트를 반영하되, 모두 같은 권한을 갖지는 않는다.

| 계층 | 역할 | 예시 | 결과 권한 |
| --- | --- | --- | --- |
| Hard Gate | 후보가 성립 가능한 최소 조건 | 현재 시설 계보, 성급, 연구 해금, 닫힌 방, 도달 가능성 | 미달 시 후보 탈락 |
| Physical Evidence | 방의 물리적 증거 | 면적, 문, 동선, 좌석 밀도, 가구 배치 | 압력 계산 |
| Operation Evidence | 실제 운영 증거 | 방문 수, 회전율, 체류 시간, 매출, 재고 소비 | 압력 계산 |
| Social Evidence | 손님/직원/소문 증거 | 용병 단골, 귀족 손님, 직원 이용, 부정 소문 | 기록 토큰 |
| Risk Evidence | 사건/위생/범죄 증거 | 절도, 소란, 악취, 배변 실패, 경비 사각 | 후보 강화 또는 반증 |
| Execution Cost | 실행 비용과 해금 | 재료, 마나, 설계도, 특수 촉매 | 승인 후 소비 |
| LLM Interpretation | 납득 가능한 해석 | 이유 문장, 흐릿한 영감, 도감 기록 | 최종 승인 불가 |

이 우선순위가 중요하다.

예를 들어 전투 기물이 있어도 전투형 손님 기록이나 전투 문화가 없으면 `전투 식당`은 승인되지 않는다.

반대로 테이블이 적어도 고급 손님, 높은 객단가, 청결, 서비스 기록이 없으면 `고급 식당`이 아니라 `미성숙 식당`으로 읽힌다.

### 업태 정체성 판정

엔진은 원본 로그를 그대로 조합식에 넣지 않고, 먼저 정체성 압력으로 바꾼다.

```text
RoomProfile
+ UsageProfile
+ EconomyProfile
+ SanitationProfile
+ SecurityProfile
+ ReputationProfile
+ StaffProfile
-> IdentityPressure
-> CandidateScore
-> EngineValidation
```

초기 식당 계보는 다음 감각으로 시작한다.

| 후보 | 강한 압력 | 필요한 증거 | 반증 |
| --- | --- | --- | --- |
| 대중 식당 | Crowd, Service | 높은 좌석 밀도, 높은 회전율, 많은 방문, 불만 통제 | 위생 붕괴, 대기열 과다, 직원 스트레스 폭증 |
| 고급 식당 | Luxury, Service, Sanitation | 넓은 좌석 간격, 좌석당 장식/서비스, 높은 객단가, 고소비 손님 | 낮은 매출, 낮은 장식, 낮은 방문, 소란 |
| 연회장 | Crowd, Luxury, Social | 큰 방, 단체 손님, 긴 체류, 중상 이상 객단가 | 작은 방의 혼잡, 낮은 이용률 |
| 전투 식당 | Combat, Crowd | 용병/오크 손님, 고기 소비, 훈련 기물, 소란/무용담 | 전투 손님 기록 없음 |
| 무법자 주점 | Outlaw, Reputation | 절도/밀거래/장기 체류, 악명, 낮은 보안 | 경비와 통제가 너무 강함 |
| 직원 식당 | StaffCare, Service | 직원 이용률, 피로 회복, 근무 안정, 저렴한 공급 | 손님 전용 운영, 직원 이용 낮음 |
| 귀족 식당 | Luxury, Sanitation, Reputation | 귀족 방문, 고급 평판, 청결, 높은 서비스 | 악취, 소란, 낮은 객단가 |

이렇게 하면 플레이어는 같은 `일반 식당`을 놓고도 운영 방식에 따라 다른 계보를 만든다.

```text
많이 앉히고 빠르게 돌린다 -> 대중 식당
적게 받고 비싸게 대접한다 -> 고급 식당
크게 만들고 단체를 받는다 -> 연회장
용병과 전투 문화를 키운다 -> 전투 식당
범죄와 악명을 방치한다 -> 무법자 주점
직원 복지 공간으로 굳힌다 -> 직원 식당
위생과 귀족 평판을 관리한다 -> 귀족 식당
```

### LLM이 살아나는 지점

LLM은 `판정자`가 아니라 `맥락 해석자`지만, 단순 flavor 담당은 아니다.

엔진이 만든 후보와 반증까지 받은 뒤, LLM은 다음을 한다.

- 왜 이 시설이 지금 이 업태로 굳어졌는지 요약한다.
- 같은 후보라도 플레이어의 실제 운영 기록에 맞춰 다른 이유를 쓴다.
- 승인 직전 후보와 아슬아슬하게 미달한 후보를 구분해 설명한다.
- 소문, 평판, 관계, 단골, 직원 불만을 자연어로 연결한다.
- 다음 계보로 이어질 수 있는 `흐릿한 영감`을 만든다.

예:

```text
엔진 결과:
Combat=0.71, Crowd=0.62, Outlaw=0.48
approved: 전투 식당
nearMiss: 무법자 주점
rejectedReason: 악명 18/50, 절도 기록 0/2
```

```text
LLM 표현:
용병 손님들은 이 식당을 밥 먹는 곳보다 무용담을 꺼내는 장소로 기억하기 시작했습니다.
다만 아직 무법자 주점이라 부르기에는 악명도, 실제 범죄 기록도 부족합니다.
```

이 정도가 LLM이 게임성에 실제로 기여하는 지점이다.

LLM이 결과를 만들지는 않지만, 플레이어가 `왜 이 계보가 열렸는지`, `왜 저 계보는 아직 안 열렸는지`, `내 운영이 어떤 이야기로 굳어졌는지`를 납득하게 만든다.

## 최종 잠금 요약

이 시스템의 결론은 `상위 방`도 아니고 `가구 조합식`도 아니다.

```text
현재 시설의 업태가
방의 배치와 운영 기록으로 증명된 방향을 따라
다음 성급의 계보 시설로 합성 진화한다.
```

플레이어가 조작하는 것은 벽, 문, 테이블, 의자, 장식, 동선, 직원, 손님층, 재료, 연구, 위험 관리다.

게임이 최종적으로 판단하는 것은 `이 시설이 어떤 업태로 굳어졌는가`다.

```text
시설 = 성장 대상
방 = 맥락 생성기
가구 = 물리적 증거
운영 기록 = 업태가 실제로 굳어졌다는 증거
재료/연구/설계도 = 실행 비용과 해금 조건
LLM = 맥락 해석과 서사화
엔진 = 후보 생성, 점수 계산, 검증, 최종 승인
```

따라서 다음처럼 단순 판정하지 않는다.

```text
테이블 많음 = 대중 식당
테이블 적음 = 고급 식당
식당 + 훈련장 = 전투 식당
```

대신 다음처럼 판단한다.

```text
테이블 많음
+ 좌석 밀도 높음
+ 회전율 높음
+ 평균 지불액 낮음/중간
+ 대기열, 위생, 직원 스트레스가 통제됨
-> Crowd 압력
-> 대중 식당 후보 강화
```

```text
테이블 적음
+ 좌석 간격 넓음
+ 좌석당 장식/서비스 높음
+ 평균 지불액 높음
+ 고소비 손님 반복 이용
+ 조용하고 깨끗한 평판
-> Luxury 압력
-> 고급 식당 후보 강화
```

```text
식당
+ 전투 기물 또는 인접 훈련 환경
+ 용병/오크/경비 손님 이용 기록
+ 고기 소비와 소란/무용담 기록
+ 필요한 재료와 연구 조건
-> Combat 압력
-> 전투 식당 후보 강화
```

핵심은 `배치가 만든 운영 형태`다.

같은 의자 16개라도 작은 방에서 빠르게 회전하면 대중 식당, 큰 방에서 단체 손님이 오래 머물면 연회장, 고급 장식과 높은 객단가가 붙으면 고급 식당, 위생과 직원 스트레스가 무너지면 혼잡한 저가 식당 계열로 읽힌다.

이 구조가 이 게임의 차별점이다.

```text
던전메이커식 합성 성장감
+ 림월드식 공간/가구/동선 맥락
+ 운영 기록과 소문/평판
+ LLM의 해석 서사
```

다만 결과는 항상 통제된다.

LLM은 후보를 새로 발명하지 못하고, 엔진이 만든 후보 풀 안에서만 정체성 요약, 추천 순서, 이유, 흐릿한 영감, 변이 수식어 후보, 도감 문장을 만든다.

## 핵심 결정

### 1. 합성 대상은 현재 시설이다

합성/진화 버튼을 누르는 대상은 `방`도 아니고 `가구 묶음`도 아니다.

항상 현재 배치된 시설 인스턴스다.

```text
1성 일반 식당
-> 2성 대중 식당
-> 3성 군중 급식소

1성 일반 식당
-> 2성 고급 식당
-> 3성 귀족 식당

1성 일반 식당
-> 2성 전투 식당
-> 3성 전장의 식당
```

방은 결과물이 아니라, 시설이 진화할 수 있는 환경과 증거를 제공한다.

### 2. 가구는 재료가 아니라 증거다

일반 테이블, 의자, 조리대, 장식, 세면대, 화장실, 무기 거치대는 대부분 직접 소모되는 합성 재료가 아니다.

이들은 방의 성격을 만드는 입력이다.

```text
나쁜 방식:
테이블 3개 + 의자 10개 + 조리대 1개 + 훈련 더미 1개 = 전투 식당

좋은 방식:
일반 식당
+ Dining 점수 충분
+ Combat 또는 Training 압력 충분
+ 용병/오크 이용 기록
+ 고기 소비 기록
+ 필요한 재료
-> 전투 식당
```

특수한 유니크 기물은 예외적으로 직접 요구할 수 있지만, 일반 규칙은 `개별 가구 이름`보다 `압축된 방 지표`를 본다.

### 3. 상위 방은 만들지 않는다

`전투 식당 방`, `고급 식당 방` 같은 상위 방 오브젝트를 만들지 않는다.

방은 계속 방이다.

벽, 문, 크기, 동선, 가구, 청결, 소음, 접근성, 손님층을 가진 공간이고, 그 안의 시설이 계보를 가진다.

```text
방의 역할:
- 닫힌 공간인지 확인
- 도달 가능한지 확인
- 가구/기물 점수 제공
- 동선/혼잡/좌석 밀도 제공
- 운영 기록과 사건 기록 제공
- LLM 해석에 필요한 맥락 제공

시설의 역할:
- 현재 ID와 계보를 가짐
- 성급을 가짐
- 진화 이력을 가짐
- 실제 기능과 밸런스 값을 가짐
- 다음 계보 시설로 교체됨
```

### 4. LLM은 창조자가 아니라 맥락 해석자다

LLM은 새로운 시설 ID, 성능, 비용, 조건을 마음대로 만들지 않는다.

게임 엔진이 먼저 가능한 후보 풀을 만든다.

LLM은 그 후보 풀 안에서 다음을 한다.

- 현재 시설이 어떤 정체성으로 운영되어 왔는지 요약한다.
- 후보들의 추천 순서를 제안한다.
- 왜 이 후보가 어울리는지 설명한다.
- 조건이 살짝 부족한 후보를 `흐릿한 영감`으로 표현한다.
- 진화 성공 시 flavor text와 도감 기록 문장을 만든다.
- 허용된 변이 태그 안에서 후보를 제안한다.

최종 승인, 비용, 수치, 성급, 결과 시설은 항상 게임 코드가 검증한다.

## 전체 처리 구조

```text
Raw Context
-> Profile
-> IdentityPressure
-> Candidate Pool
-> Engine Validation
-> LLM Interpretation
-> Player Choice
-> Facility Evolution
```

### Raw Context

게임에서 실제로 발생한 원본 정보다.

- 방 면적
- 벽과 문
- 도달 가능성
- 테이블 수
- 의자 수
- 좌석 간격
- 조리대/카운터/장식/기물
- 화장실/세면대/청소 상태
- 방문 손님 종족과 성향
- 평균 만족도
- 평균 지불액
- 회전율
- 대기열
- 재고 소비
- 직원 근무 시간
- 직원 스트레스
- 직원 이용률
- 범죄/절도/소란
- 침입 방어 기여
- 소문과 평판

이 값은 그대로 진화식에 다 넣지 않는다.

### Profile

원본 정보를 각 도메인별로 압축한다.

```text
RoomProfile
- 면적, 닫힌 방, 문, 동선, 좌석 밀도, 기물 점수

UsageProfile
- 방문 횟수, 손님층, 직원 이용률, 체류 시간, 회전율

EconomyProfile
- 매출, 평균 지불액, 재고 소비, 품절/보충 실패

SanitationProfile
- 화장실 커버리지, 세면/청결, 악취, 위생 불만, 배변 실패

SecurityProfile
- 절도, 소란, 보안 사각지대, 경비 접근성, 침입 방어 기여

ReputationProfile
- 소문, 평판, 종족별 평가, 단골 반응, 부정적 낙인

StaffProfile
- 직원 스트레스, 피로 회복, 직원 선호, 근무 안정성
```

### IdentityPressure

Profile을 시설 진화가 읽기 쉬운 정체성 압력으로 바꾼다.

초기 압력:

- `Crowd`: 많이 받고 빠르게 처리하는 방향
- `Luxury`: 적게 받고 비싸게 오래 머물게 하는 방향
- `Combat`: 전투 손님, 훈련, 무기, 소란, 방어 기여 방향
- `Outlaw`: 범죄, 밀거래, 보안 사각, 부정적 소문 방향
- `Service`: 안정적 운영, 낮은 대기열, 높은 만족도 방향
- `Sanitation`: 위생, 세면, 배변 처리, 청결 유지 방향
- `StaffCare`: 직원 이용, 피로 회복, 직원 만족 방향
- `Security`: 경비, 침입 방어, 통제된 위험 방향
- `Rest`: 체류, 휴식, 회복 방향
- `Logistics`: 재고, 보충, 창고 접근성 방향

중요한 점:

```text
테이블 많음 -> 대중 식당
```

이렇게 바로 가지 않는다.

```text
테이블 많음
+ 좌석 밀도 높음
+ 회전율 높음
+ 객단가 낮음/중간
+ 대기열과 불만이 통제됨
-> Crowd 압력 증가
-> 대중 식당 후보 강화
```

반대로:

```text
테이블 적음
+ 좌석 간격 넓음
+ 좌석당 장식/서비스 높음
+ 평균 지불액 높음
+ 고소비 손님 반복 방문
+ 위생과 소음 관리 좋음
-> Luxury 압력 증가
-> 고급 식당 후보 강화
```

즉 단순 수량이 아니라, 수량이 만든 운영 형태를 읽는다.

## 식당 계보 판정 예시

### 대중 식당

```text
현재 시설: 일반 식당
Hard Gate:
- 닫힌 방
- 도달 가능
- Dining 점수 충분

Primary Pressure:
- Crowd 높음
- Service 중간 이상

주요 증거:
- 테이블과 의자가 많음
- 방 면적 대비 좌석 밀도 높음
- 회전율 높음
- 객단가 낮음/중간
- 일반 손님을 많이 처리
- 대기열과 위생 불만이 통제됨

결과:
-> 대중 식당 / 군중 식당
```

대중 식당은 `싼 식당`이 아니라 `많은 손님을 안정적으로 처리하는 식당`이다.

### 고급 식당

```text
현재 시설: 일반 식당
Hard Gate:
- 닫힌 방
- 도달 가능
- Dining 점수 충분
- 최소 품질 조건 충족

Primary Pressure:
- Luxury 높음
- Service 높음
- Sanitation 중간 이상

주요 증거:
- 테이블과 의자가 적음
- 좌석 간격 넓음
- 좌석당 장식/서비스 높음
- 평균 지불액 높음
- 귀족/고소비 손님 반복 방문
- 체류 시간이 길지만 불만이 낮음
- 위생과 소음 관리 좋음

결과:
-> 고급 식당 / 귀족 식당
```

고급 식당은 `의자가 적은 식당`이 아니라 `적게 받고 비싸게 오래 머물게 하는 운영이 증명된 식당`이다.

### 미성숙 식당

```text
테이블과 의자가 적음
+ 장식 낮음
+ 서비스 낮음
+ 방문 적음
+ 평균 지불액 낮음
+ 특별한 평판 없음
-> 고급 식당 후보 아님
```

의자가 적다는 사실만으로 고급화되지 않는다.

### 전투 식당

```text
현재 시설: 일반 식당 또는 고기 식당
Hard Gate:
- Dining 점수 충분
- Combat 또는 Training 환경 충분
- 전투형 손님 이용 기록 존재

Primary Pressure:
- Combat 높음
- Crowd 또는 Meat 중간 이상

주요 증거:
- 훈련 더미, 무기 거치대, 전투 장식
- 오크/용병/전투형 손님 비율 높음
- 고기 소비 높음
- 소란 또는 무용담 기록
- 인접 훈련장 이용 기록

결과:
-> 전투 식당
```

전투 식당은 `식당 + 훈련장`이 아니라, 식당의 운영 문화가 전투 손님과 결합한 결과다.

### 무법자 주점

```text
현재 시설: 식당 또는 주점 계열
Primary Pressure:
- Outlaw 높음
- Security 낮음 또는 보안 사각 존재

주요 증거:
- 범죄 성향 손님 비율 높음
- 장기 체류
- 절도/밀거래/시비 기록
- 악명 상승
- 부정적 소문
- 경비 커버리지 낮음

결과:
-> 무법자 주점 후보
```

무법자 계보는 LLM이 특히 잘 활용되는 영역이다.

수치로는 `Outlaw=0.72`라고만 보이지만, LLM은 이것을 다음처럼 플레이어가 읽을 수 있는 사건으로 바꾼다.

```text
이 식당은 식사보다 뒷거래의 약속 장소로 더 자주 언급되고 있습니다.
손님들은 음식 맛보다 "눈치 보지 않아도 되는 자리"를 기억합니다.
```

## 같은 배치의 다른 결과

같은 `테이블 4개, 의자 16개`라도 결과는 하나가 아니다.

```text
작은 방
+ 좌석 밀도 높음
+ 회전율 높음
+ 객단가 낮음
+ 큰 불만 없음
-> 대중 식당

큰 방
+ 좌석 밀도 중간
+ 좌석 간격 넉넉함
+ 단체 손님 기록 많음
+ 평균 지불액 중간 이상
-> 연회장

큰 방
+ 좌석 간격 넓음
+ 장식과 서비스 높음
+ 고소비 손님 많음
+ 체류 시간 김
-> 고급 식당

작은 방
+ 좌석 밀도 높음
+ 대기열 김
+ 위생 불만 높음
+ 직원 스트레스 높음
-> 혼잡한 식당 / 사고 위험 / 저가 식당

중간 방
+ 훈련 기물 있음
+ 용병 손님 많음
+ 고기 소비 높음
+ 소란 기록 있음
-> 전투 식당
```

이 감각이 시스템의 핵심이다.

플레이어는 물건을 배치하지만, 게임은 그 배치가 만든 운영 형태를 읽는다.

## 계보는 한 번으로 끝나지 않는다

LLM은 `일반 식당 -> 전투 식당`에서 한 번 쓰이고 끝나는 장식이 아니다.

각 계보 단계마다 현재 기록을 다시 읽는다.

```text
2성 전투 식당
+ 용병/오크 단골
+ 고기 소비 많음
+ 소란은 있지만 통제됨
+ 인접 훈련장 이용 많음
-> 3성 전장의 식당

2성 전투 식당
+ 범죄 손님 증가
+ 절도/밀거래 소문
+ 보안 낮음
+ 악명 상승
-> 3성 무법 경기 주점

2성 전투 식당
+ 경비 직원 이용 많음
+ 침입 방어 기여 높음
+ 보안 압력 높음
-> 3성 경비대 식당

2성 전투 식당
+ 좌석 밀도 낮음
+ 고급 장식과 서비스
+ 귀족 전투광 손님
-> 3성 결투 귀빈식당
```

이렇게 하면 던전메이커식 성급 상승 감각을 유지하면서도, 림월드식 운영 맥락이 매 단계 결과를 바꾼다.

## 합성 감각을 남기는 방법

시설끼리 직접 먹이지 않더라도 합성 감각은 사라지지 않는다.

합성 감각은 다음 네 가지에서 나온다.

### 1. 현재 시설이 결과 시설로 교체된다

```text
일반 식당 1성
-> 대중 식당 2성
```

플레이어는 실제로 한 시설을 다음 성급 시설로 바꾼다.

### 2. 재료와 마나를 소비한다

```text
대중 식당 진화 비용:
- 조리 재료
- 목재/석재
- 마나석
- 설계도 해금
```

### 3. 일부 기록 토큰을 촉매로 쓴다

대부분의 운영 기록은 보존되지만, 일부 의식/계약/주문형 토큰은 소모될 수 있다.

```text
Preserve:
- MercenaryHangout
- NoblePatronage
- CleanServiceStreak

ConsumeRequiredAmount:
- FestivalContract
- BloodOath
- RarePatronRequest

ConsumeAll:
- PurificationDebt
- CursedMomentum
```

### 4. 진화 이력과 변이 태그가 계승된다

```text
일반 식당
-> 전투 식당
   history: 용병 단골, 고기 소비, 소란 기록
   mutation: Tactical
```

이 이력은 다음 단계 LLM 해석과 도감 문장에 다시 쓰인다.

## LLM 사용 위치

### LLM 입력

LLM에는 원본 로그 전체를 보내지 않는다.

압축된 `FacilityIdentitySnapshot`만 보낸다.

```text
facilityInstanceId
facilityName
currentFacilityId
starGrade
lineageTags
roomSummary
topMetrics
identityPressures
topRecordTokens
dominantSignals
conflictingSignals
recentEventSummary
longTermHistorySummary
candidateEvolutionIds
nearMissEvolutionIds
```

예:

```json
{
  "facilityName": "고기 식당",
  "starGrade": 1,
  "roomSummary": "닫힌 8칸 식당. 좌석 밀도는 높고 장식은 낮다.",
  "topMetrics": {
    "seatDensity": 0.82,
    "turnoverRate": 0.76,
    "averageSpend": 0.42,
    "combatVisitorRatio": 0.61,
    "hygienePerUser": 0.48
  },
  "identityPressures": {
    "Crowd": 0.78,
    "Combat": 0.66,
    "Luxury": 0.12,
    "Sanitation": 0.38
  },
  "topRecordTokens": {
    "MercenaryHangout": 2,
    "HighMeatConsumption": 3,
    "FrequentBrawls": 1
  },
  "candidateEvolutionIds": [
    "evolve_meat_restaurant_to_combat_tavern",
    "evolve_meat_restaurant_to_mass_diner"
  ],
  "nearMissEvolutionIds": [
    "evolve_meat_restaurant_to_outlaw_pub"
  ]
}
```

### LLM 출력

LLM 출력은 제한된 스키마를 따른다.

```text
facilityIdentitySummary
rankedProposalIds
proposalReasons
nearMissHintText
mutationTagSuggestions
flavorText
codexHistoryText
confidence
```

LLM이 하면 안 되는 것:

- 존재하지 않는 시설 ID 생성
- 성능 수치 생성
- 비용 생성
- 조건 무시
- 미승인 변이 태그 확정
- 플레이어 선택 없이 자동 진화

## 소문/평판/관계의 활용

소문과 평판은 단순 flavor가 아니라 후보 방향을 바꾸는 증거다.

다만 수치 검증은 엔진이 한다.

```text
긍정 소문:
"용병들이 배부르게 먹고 무용담을 나누는 곳"
-> Combat, Crowd, MercenaryHangout 강화

고급 평판:
"비싸지만 조용하고 대접이 좋다"
-> Luxury, Service, NoblePatronage 강화

부정 소문:
"돈을 내고도 물건을 잃어버리는 곳"
-> Outlaw, TheftPressure 강화

직원 평판:
"직원들이 쉬는 시간마다 몰래 찾는 식당"
-> StaffCare, Service 강화
```

LLM은 여기서 강하다.

엔진은 `TheftPressure=2`, `NegativeRumor=3`처럼 압축하지만, LLM은 플레이어가 납득할 수 있는 문장으로 풀어준다.

## 범죄와 보안의 활용

범죄 확률을 단순히 다음처럼 계산하지 않는다.

```text
baseCrimeChance
+ unstaffedCrimeChanceBonus
- staffedCrimeChanceReduction
```

대신 범죄도 컨텍스트 프로필로 다룬다.

```text
CrimeRiskProfile:
- 범죄 성향 손님 비율
- 장기 체류 손님 비율
- 보안 사각지대
- 경비 직원 접근성
- 혼잡도
- 조명/시야
- 최근 절도/시비 기록
- 부정적 소문
- 직원 부족
- 고가 재고 노출
```

이 프로필은 두 곳에 쓰인다.

- 운영 중 범죄 발생 판정
- 시설 계보 진화의 `Outlaw`, `Security` 압력

즉 범죄는 단순 벌칙이 아니라, 무법자 계보나 경비 계보를 여는 운영 기록이 된다.

## 위생과 배변의 활용

배변과 위생은 욕구 시스템에서만 끝나지 않는다.

방의 품질, 평판, 시설 진화에도 영향을 준다.

```text
SanitationProfile:
- 화장실 수용량
- 화장실 접근 거리
- 배변 실패 횟수
- 세면대/목욕 시설
- 청소 빈도
- 악취/오염
- 위생 불만
- 직원 청소 작업 안정성
```

예:

```text
식당 + 높은 위생 + 높은 서비스 + 고소비 손님
-> 고급 식당 후보 강화

휴식방 + 세면/목욕 + 청결 유지 기록
-> 고급 휴식실 또는 회복실 후보

화장실 + 높은 이용률 + 청소 안정성
-> 공중 화장실, 직원 화장실, 고급 세면실 계보

위생 실패 + 악취 소문 + 범죄/무법 손님
-> 지저분한 주점, 오물 구덩이 같은 특수 계보의 흐릿한 영감
```

## 연구와 설계도

설계도는 진화의 재료가 아니라, 가능성을 여는 정보다.

```text
설계도 획득
-> 연구 시설에서 분석
-> 특정 계보 후보 또는 특수 진화식 해금
-> 조건을 만족한 시설에서 후보로 노출
```

일반 진화식은 공개해도 된다.

특수 진화식은 연구나 도감 힌트로 일부만 보여준다.

```text
공개:
일반 식당 -> 대중 식당
일반 식당 -> 고급 식당

연구 필요:
전투 식당 -> 전장의 식당
전투 식당 -> 무법 경기 주점
고급 식당 -> 흡혈 귀빈식당
```

## UI와 디버그

플레이어 UI는 수치 전체를 보여주지 않는다.

플레이어에게 보여줄 것:

- 현재 시설 이름과 성급
- 방 요약
- 주요 정체성 압력
- 주요 운영 기록
- 승인된 진화 후보
- 흐릿한 영감
- 필요한 재료
- LLM이 쓴 납득 가능한 이유

예:

```text
고기 식당 / 1성
방 성격: 빽빽한 좌석, 빠른 회전, 용병 단골
정체성: Crowd 높음, Combat 높음, Luxury 낮음
기록: 용병 단골 2, 고기 소비 3, 소란 1

진화 후보:
- 전투 식당
- 대중 식당

흐릿한 영감:
- 무법자 주점: 악명과 절도 기록이 더 필요합니다.
```

개발자 디버그 UI는 전부 보여준다.

```text
RoomProfile
UsageProfile
EconomyProfile
SanitationProfile
SecurityProfile
ReputationProfile
IdentityPressures
RecordTokens
CandidateScores
EngineValidationResults
LlmInputSnapshot
LlmOutput
FinalApprovedCandidates

```

## 구현 원칙

### 원칙 1. 도메인 로그는 직접 뒤지지 않는다

식당 진화 시스템이 화장실 로그, 범죄 로그, 직원 스트레스 로그를 직접 조회하면 커플링이 심해진다.

각 도메인은 자기 Profile을 만든다.

진화 판정기는 Profile과 IdentityPressure만 읽는다.

```text
ToiletRuntime -> SanitationProfile
CrimeRuntime -> SecurityProfile
StaffRuntime -> StaffProfile
FacilityRuntime -> UsageProfile
EvolutionSystem -> Profile만 읽음
```

### 원칙 2. LLM 없이도 동작해야 한다

LLM 서버가 꺼져도 공개 진화식과 엔진 검증은 동작해야 한다.

다만 LLM이 없으면 다음이 빠진다.

- 풍부한 이유 문장
- 흐릿한 영감의 자연어 표현
- 도감 기록 문장
- 변이 태그 추천

후보 생성과 최종 승인 자체는 엔진이 한다.

### 원칙 3. 후보 풀 밖의 결과는 금지한다

LLM이 아무리 그럴듯한 후보를 말해도, 게임 데이터에 없는 결과는 폐기한다.

### 원칙 4. 같은 계보라도 매 단계 다시 판정한다

`전투 식당`이 된 뒤에도 다음 단계는 고정되지 않는다.

현재 방 배치와 운영 기록이 바뀌면 다음 후보도 바뀐다.

### 원칙 5. 모든 시설을 같은 깊이로 만들지 않는다

모든 시설군이 5성까지 완성될 필요는 없다.

초기에는 핵심 시설군만 깊게 만든다.

- 식당
- 상점
- 휴식
- 방어
- 경비/훈련
- 연구/마력

나머지는 1~2단계만 두고, 게임성이 확인되면 확장한다.

## 최종 감각

플레이어가 실제로 하는 일:

```text
방을 나눈다.
벽과 문을 만든다.
테이블, 의자, 장식, 조리대, 화장실, 세면대, 훈련 기물을 배치한다.
어떤 손님층을 끌어들일지 운영한다.
직원을 배치하고 보충/청소/수리/경비를 관리한다.
범죄, 소란, 위생 실패, 소문을 방치하거나 통제한다.
그 결과 열린 시설 계보 중 하나를 선택해 성급을 올린다.
```

게임이 읽는 것:

```text
배치
-> 운영 형태
-> 반복된 기록
-> 방의 정체성
-> 현재 시설의 다음 계보
```

이 시스템의 매력은 `정답 조합식 암기`가 아니라, 플레이어가 실제로 운영한 방식이 시설의 이름과 기능으로 굳어진다는 점이다.

```text
빽빽하고 빠른 식당 -> 대중 식당
넓고 비싼 식당 -> 고급 식당
용병이 모이는 식당 -> 전투 식당
범죄 소문이 도는 식당 -> 무법자 주점
직원이 자주 쓰는 식당 -> 직원 식당
위생과 서비스가 뛰어난 식당 -> 귀족 식당
```

그리고 각 결과는 다시 다음 성급의 출발점이 된다.

```text
고기 식당
-> 전투 식당
-> 전장의 식당
-> 군단 연회장

고기 식당
-> 고급 식당
-> 귀족의 식당
-> 왕실 연회장

고기 식당
-> 대중 식당
-> 군중 급식소
-> 전시 보급 식당
```

따라서 최종 방향은 다음으로 고정한다.

```text
시설은 성장 대상이다.
방은 맥락 생성기다.
가구는 증거다.
운영 기록은 촉매다.
LLM은 해석자다.
게임 엔진은 검증자다.
플레이어는 운영 방식으로 계보를 만든다.
```
