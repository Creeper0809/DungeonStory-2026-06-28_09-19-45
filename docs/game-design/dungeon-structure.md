# 던전 구조

던전은 2D 횡스크롤 그리드 구조를 기본으로 한다.

현재 구현된 그리드 시스템을 기준으로 설계한다.

최신 방향:

- 초기 설계의 `방 = 하나의 주 시설 영역` 관점은 파츠 기반 방 시스템으로 확장한다.
- 방은 벽, 문, 이동 가능한 내부 셀, 가구, 환경 태그, 운영 기록으로 판정한다.
- 시설 성급과 계보 성장은 [시설 계보 진화](facility-lineage-evolution.md)가 담당한다.
- 즉 공간은 조건과 기록을 만들고, 현재 시설은 그 조건을 바탕으로 다음 계보 시설로 진화한다.

## 현재 구현 기준

현재 코드 구조:

```text
Grid
-> GridCell[,]
-> GridCell은 GridLayer별 BuildableObject를 보유
```

현재 레이어:

```text
Hallway
Building
```

현재 건설 데이터:

```text
BuildingSO
- width
- height
- layer
- category
- horizontalDraggable
- verticalDraggable
- OnBuildCondition
```

현재 이동 구조:

- 같은 층에서는 좌우로만 이동한다.
- `HasBuilding()`인 셀만 이동 가능한 셀로 본다.
- `BuildingCategory.Movement` 건물은 자신이 차지한 셀들을 `connectedFloor`로 연결한다.
- 계단 같은 Movement 시설은 층 사이 이동을 담당한다.
- 일반 건물을 설치하면 해당 셀에 Hallway 레이어가 자동으로 깔린다.

즉 현재 구조에서는 `건물이 있는 셀 = 이동 가능한 던전 내부 공간`에 가깝다.

## 용어

### 셀

셀은 던전 구조의 최소 단위다.

```text
1 GridCell = 배치, 이동, 연결성 판단의 기본 단위
```

각 셀은 여러 레이어를 가질 수 있다.

```text
Hallway 레이어: 이동 바닥/복도
Building 레이어: 상점, 식당, 휴식방, 훈련장, 연구실 같은 주 시설
```

초기 구현에서는 한 셀에 Building 레이어 시설은 하나만 들어간다.

### 층

그리드의 y 좌표 하나가 한 층이다.

현재 월드 좌표상 한 층은 3 유닛 간격으로 표현된다.

```text
grid y 0 = 1층
grid y 1 = 2층
grid y 2 = 3층
```

층 사이 이동은 일반 좌우 이동으로 처리하지 않고, 계단 같은 Movement 시설이 담당한다.

### 방

초기 문서에서는 방을 별도 컨테이너로 두지 않고, 하나의 주 시설이 차지하는 그리드 영역으로 시작한다고 정의했다.

```text
방 = 하나의 주 시설이 차지하는 그리드 영역
```

예시:

```text
저가 음식점 width 3 = 3셀짜리 식당 방
훈련장 width 4 = 4셀짜리 훈련 방
연구실 width 2 = 2셀짜리 연구 방
```

이 방식은 첫 프로토타입의 단순화 규칙이었다.

최신 방향에서는 방과 시설을 분리한다.

```text
방 = 벽/문/내부 셀/가구 조건으로 판정되는 공간
시설 = 방 안에서 운영되며 계보와 성급을 가진 대상
성장 = 현재 시설의 계보 진화
```

이유:

- 현재 `GridCell`은 Building 레이어 하나만 점유할 수 있다.
- `BuildingSO`가 width/height로 여러 셀을 차지할 수 있다.
- 캐릭터 AI는 도달 가능한 `BuildableObject`를 목적지로 삼는다.
- 방과 시설을 분리하면 방 컨테이너, 내부 슬롯, 내부 이동 규칙이 추가로 필요하다.

따라서 구현은 단계적으로 간다.

```text
1단계: 기존 Building 기반 시설 유지
2단계: 벽/문/가구로 닫힌 방 판정
3단계: 방 조건과 운영 기록을 RoomProfile로 압축
4단계: 현재 시설의 계보 진화와 RoomProfile을 연결
```

## 방 안 시설 개수

초기 규칙:

```text
한 방 = 주 시설 1개
```

한 방 안에 여러 개의 독립 시설을 넣지 않는다.

대신 다음 값으로 방의 규모와 기능 차이를 만든다.

- width/height
- 수용 인원
- 대기열 크기
- 필요 직원 수
- 재고 용량
- 환경 태그
- 운영 정책
- 합성 단계

예시:

```text
작은 잡화점
- width 2
- 직원 1
- 손님 수용 1-2

큰 잡화점
- width 4
- 직원 2
- 손님 수용 3-4
- 재고 용량 증가
```

후반 확장으로는 `부착물`이나 `장식 슬롯`을 둘 수 있다.

하지만 이것도 별도 시설 여러 개를 넣는 방식이 아니라, 하나의 방 시설에 보조 효과를 붙이는 방식으로 둔다.

## 복도

복도는 Hallway 레이어로 표현한다.

역할:

- 이동 가능 공간을 만든다.
- 같은 층의 좌우 연결성을 만든다.
- 손님, 직원, 침입자의 기본 동선이 된다.
- 시설이 없는 빈 구간을 연결한다.

현재 구현상 일반 건물을 설치하면 해당 셀에 Hallway가 자동으로 깔린다.

따라서 다음 두 가지 복도가 존재한다.

```text
시설 내부 복도:
시설이 차지한 셀 아래 자동으로 생기는 이동 레이어

전용 복도:
시설 없이 이동만 담당하는 Hallway 레이어
```

설계 방향:

- 손님과 직원은 복도와 시설 내부 복도를 통해 이동한다.
- 전용 복도는 구역 사이를 잇거나 병목을 만드는 데 사용한다.
- 복도를 너무 많이 깔면 동선은 편해지지만 침입자도 빨리 이동할 수 있다.
- 복도를 끊으면 동선이 사라지므로, 시설 접근성과 방어를 동시에 고려해야 한다.

## 계단과 층 연결

계단은 `BuildingCategory.Movement` 시설로 둔다.

역할:

- 층 사이 이동을 가능하게 한다.
- 수평 동선만으로는 갈 수 없는 층을 연결한다.
- 침입자의 우회/돌파 경로를 만든다.
- 직원의 보충/수리 동선에도 영향을 준다.

현재 구현상 Movement 시설은 자신이 차지한 셀들을 `connectedFloor`로 연결한다.

따라서 계단은 여러 층에 걸친 건물로 두고, 그 셀들을 하나의 이동 시설로 묶는 방식이 적합하다.

초기 규칙:

```text
계단은 위아래 층을 연결한다.
계단이 없으면 층 사이 이동은 불가능하다.
계단이 파손되거나 제거되면 해당 연결도 끊긴다.
```

배치 의미:

- 계단이 많으면 운영 동선은 좋아진다.
- 계단이 많으면 침입자도 빠르게 깊은 곳으로 들어온다.
- 계단이 적으면 병목과 방어는 쉬워진다.
- 계단이 적으면 직원 보충/수리 동선이 길어진다.

## 문과 입구

입구는 외부와 던전을 잇는 시작점이다.

현재 길찾기에는 `Vector2Int.zero`가 성문/입구 기준점처럼 사용되고 있다.

설계 방향:

- 손님, 직원, 침입자의 출발 기준이 되는 입구를 둔다.
- 일부 건물은 성문/입구와 연결되어야 건설 가능하다.
- `ConditionNeedToConnect`를 사용해 특정 시설이 입구 또는 특정 시설과 연결되어 있는지 검사할 수 있다.

문은 첫 구현에서 필수 시스템으로 깊게 만들지 않는다.

초기 문 역할:

- 시각적으로 구역의 경계를 표현한다.
- 특정 구역에 들어가는 느낌을 만든다.

후반 확장 문 역할:

- 직원 전용 구역
- 침입자 지연
- 반란 격리
- 특정 종족 전용 구역

## 건설 규칙

현재 구현 기준 기본 규칙:

```text
1. 그리드 밖에는 설치할 수 없다.
2. x = 0, x = grid.width - 1 위치에는 설치할 수 없다.
3. 같은 레이어에 이미 건물이 있으면 설치할 수 없다.
4. y > 0인 건물은 아래쪽에 받쳐주는 건물이 있어야 한다.
5. 건물별 추가 조건을 만족해야 한다.
```

현재 철거 규칙:

```text
위층에 건물이 받쳐져 있으면 아래 건물을 철거할 수 없다.
```

이 규칙은 던전이 공중에 떠 있는 구조가 되는 것을 막는다.

게임적으로는 다음 의미를 가진다.

- 아래층 구조가 위층 확장의 기반이 된다.
- 구조를 잘못 부수면 위층 접근성이 무너질 수 있다.
- 침입이나 반란으로 파손된 시설을 수리해야 하는 이유가 생긴다.

## 시설 배치 규칙

초기 배치 규칙:

```text
한 셀의 Building 레이어에는 하나의 주 시설만 배치한다.
Hallway 레이어와 Building 레이어는 같은 셀에 공존할 수 있다.
일반 시설을 설치하면 필요한 Hallway 레이어가 자동으로 생긴다.
넓은 시설은 width/height만큼 여러 셀을 점유한다.
```

시설 크기 방향:

```text
1셀: 문, 소형 장치, 단순 보조 시설
2-3셀: 기본 상점/식당/휴식방/연구실
4-5셀: 대형 식당, 훈련장, 병영, 고급 시설
```

초기에는 height가 1인 시설을 중심으로 설계한다.

height가 2 이상인 시설은 다음과 같은 특수 시설에 제한한다.

- 계단
- 대형 홀
- 보스방/사장방 확장
- 대형 방어 시설

## 방어 시설 배치 방식

방어 시설도 첫 구현에서는 주 시설로 취급한다.

즉 가시 함정, 독 웅덩이, 화염 분사구, 번개 기둥, 냉기 분사구, 경비실은 모두 Building 레이어에 놓이는 시설이다.

```text
가시 함정 = 1셀짜리 방어 시설
독 웅덩이 = 1셀짜리 방어 시설
화염 분사구 = 1셀짜리 방어 시설
번개 기둥 = 1셀짜리 방어 시설
냉기 분사구 = 1셀짜리 방어 시설
경비실 = 1-2셀짜리 경비 시설
```

복도 위에 별도 부착물처럼 함정을 얹는 방식은 사용하지 않는다.

사용하지 않는 방식:

```text
복도 한 칸에 바닥 함정 + 벽 장치 + 문 장치 겹치기
긴 복도 전체에 부착형 함정 깔기
숨겨진 함정/감지 장치/문 잠금 장치
경영 시설과 방어 시설을 같은 칸에 겹치기
```

이 결정을 하면 다음 규칙이 단순해진다.

- 한 셀의 Building 레이어에는 여전히 시설 하나만 존재한다.
- 방어 시설도 경영 시설처럼 건설, 파손, 수리, 합성 대상이 된다.
- 방어 시설을 설치하면 그 셀도 던전 내부 공간으로 취급된다.
- 침입자는 방어 시설이 있는 셀을 지나가거나 머무르면서 효과를 받는다.
- 별도 TrapLayer, 복도 부착 UI, 복도 장치 선택 규칙을 만들지 않는다.

이 방식은 던전메이커처럼 `방 자체가 함정/방어 기능을 가진다`는 감각에 가깝다.

후반에도 복도 부착형 함정은 기본 확장 방향으로 두지 않는다. 방어 확장은 주로 더 큰 방어 시설, 상위 합성 시설, 경비실/병영, 시설 간 시너지로 처리한다.

## 구역 태그

구역은 별도 건물 종류가 아니라, 셀 또는 시설 묶음에 붙는 운영 태그다.

구역 태그는 길찾기를 막기보다 AI 선택, 작업 우선순위, 만족도, 위험도를 조절하는 데 사용한다.

초기 구역 태그:

```text
상업 구역
휴식 구역
직원 구역
연구/마력 구역
방어 구역
물류 구역
공용 통로
```

역할:

```text
상업 구역:
식당, 상점 중심. 손님 매출과 혼잡이 집중된다.

휴식 구역:
휴식방 중심. 만족도 회복과 체류 시간 조절을 담당한다.

직원 구역:
직원 우선 휴식, 창고, 작업 대기와 연결된다.

연구/마력 구역:
연구실, 마력 저장소, 뱀파이어/데몬 선호와 연결된다.

방어 구역:
경비실, 함정, 병목, 침입자 지연과 연결된다.

물류 구역:
창고와 보충 동선 중심. 직원 이동 효율에 영향을 준다.

공용 통로:
구역 사이 연결. 별도 시설 효과보다 이동 효율이 중요하다.
```

초기 구현에서는 구역 태그가 강제 규칙이 아니어도 된다.

먼저 다음 용도로만 사용한다.

- 시설 정보창에 구역 표시
- AI가 선호/기피를 계산할 때 참고
- 운영일 정산에서 구역별 문제 표시
- 나중에 직원 작업 우선순위와 연결

## 구역 지정 방식

초기에는 자동 구역 추론을 우선한다.

```text
식당/상점 주변 -> 상업 구역
휴식방 주변 -> 휴식 구역
창고 주변 -> 물류 구역
연구실/마력 저장소 주변 -> 연구/마력 구역
함정/경비 시설 주변 -> 방어 구역
```

나중에는 플레이어가 직접 드래그해서 구역을 지정할 수 있게 확장한다.

수동 구역 지정은 첫 구현 필수는 아니다.

이유:

- 현재 코드는 건설과 길찾기가 먼저 중요하다.
- 수동 구역 편집은 UI와 저장 구조가 추가로 필요하다.
- 자동 구역 태그만 있어도 종족 선호와 정산 피드백은 만들 수 있다.

## 동선 설계

던전 동선은 세 종류를 동시에 고려해야 한다.

```text
손님 동선:
입구 -> 선호 시설 -> 추가 시설 -> 출구

직원 동선:
휴식방/대기 위치 -> 창고 -> 근무 시설 -> 수리/청소 대상

침입자 동선:
입구 -> 약탈/파괴 대상 -> 사장방/플레이어 캐릭터
```

좋은 배치는 세 동선이 완전히 같지도, 완전히 분리되지도 않는 구조다.

예시:

```text
상업 구역을 입구 근처에 둔다.
창고는 상업 구역과 너무 멀지 않게 둔다.
연구/마력 구역은 깊은 곳에 둔다.
방어 구역은 입구와 사장방 사이 병목에 둔다.
계단은 운영 편의와 침입자 접근성을 동시에 고려해 둔다.
```

## 길찾기 설계

현재 길찾기는 BFS 기반이다.

현재 구조에서는 다음 이유로 적합하다.

- 그리드가 작고 명확하다.
- 이동 비용이 대부분 동일하다.
- 좌우 이동과 계단 이동만 있으면 된다.
- 시설 접근 가능 여부를 빠르게 판단하는 것이 중요하다.

캐릭터는 매번 모든 목적지를 따로 탐색하기보다, 시작 위치에서 한 번 탐색한 결과를 캐시해 도달 가능한 시설과 경로를 재사용한다.

이미 구현된 방향:

```text
Grid.version:
건설/철거 시 증가

AIBrain pathSearchCache:
같은 위치와 같은 gridVersion이면 탐색 결과 재사용
```

구역 태그가 추가되어도 첫 구현에서는 길찾기 비용을 바꾸지 않는다.

초기에는 다음 순서가 좋다.

```text
1. BFS로 도달 가능 시설 목록을 구한다.
2. 후보 시설을 욕구/선호/재고/혼잡/구역 태그로 평가한다.
3. 선택한 시설까지의 경로를 가져온다.
```

나중에 침입자나 특수 직원에게 다른 이동 비용이 필요해지면 A*나 가중치 기반 탐색을 검토한다.

## 배치 예시

초기 던전 배치 예:

```text
입구
-> 저가 음식점 / 잡화점
-> 창고
-> 계단
-> 훈련장 / 무기점
-> 연구실 / 마력 저장소
-> 사장방
```

의도:

- 입구 근처는 대량 손님 회전율이 좋다.
- 창고는 상업 시설과 가까워 보충이 빠르다.
- 훈련장과 무기점은 오크 동선과 방어 준비를 겸한다.
- 연구/마력 시설은 깊은 곳에 둬 침입 피해를 줄인다.
- 사장방으로 가는 길목에 방어 구역을 만들 수 있다.

## 첫 구현 범위

첫 구현에서는 다음만 확정한다.

- 그리드 셀 기반 배치
- Hallway/Building 레이어
- 일반 시설 설치 시 Hallway 자동 생성
- 한 방은 하나의 주 시설로 처리
- width 기반 시설 크기
- 계단을 통한 층 연결
- 공중부양 건설 금지
- 위층을 받치는 건물 철거 금지
- 방어 시설도 Building 레이어의 주 시설로 처리
- 복도 부착형 함정 레이어는 만들지 않음
- 자동 구역 태그
- BFS 기반 도달 가능 시설 탐색

첫 구현에서 미뤄도 되는 것:

- 방 컨테이너와 내부 시설 슬롯
- 수동 구역 드래그 편집
- 문 잠금/권한 시스템
- 구역별 이동 비용
- 복잡한 방 형태 편집
- 여러 시설을 한 방 안에 배치하는 시스템
- 복도 부착형 함정/장치 시스템
- A* 기반 가중치 길찾기
