# DungeonStory 현재 시스템 정리

> 감사 기준일: 2026-07-17  
> 기준: 현재 런타임 C# 코드, Resources 에셋, UI 연결 감사, PlayMode 검증 기록  
> 목적: 기획 문서에 적힌 의도가 아니라 **지금 게임에 실제로 존재하고 연결된 시스템**을 한눈에 파악한다.

## 1. 현재 게임의 정체

DungeonStory는 현재 다음 세 축이 결합된 2D 던전 경영 시뮬레이션이다.

1. 벽, 문, 통로, 가구를 직접 배치해 방을 만드는 공간 설계
2. 몬스터 손님과 직원이 욕구, 기분, 업무, 재고에 따라 움직이는 운영 시뮬레이션
3. 운영으로 성장한 던전이 자동 침공과 원정을 버티는 런 기반 공방 구조

현재 가장 깊게 구현된 축은 `건축 -> 방 판정 -> 시설 이용 -> 캐릭터 AI`다. 연구, 합성, 진화, 침공, 원정, 메타 진행도 실제 상태 변화와 UI까지 연결되어 있지만 콘텐츠 폭과 장기 진행 완성도는 건축/방 시스템보다 얕다.

## 2. 상태 표기

| 등급 | 의미 |
|---|---|
| A · 실플레이 연결 | 일반 플레이 입력, 런타임 상태 변화, UI 피드백이 연결되어 있다. |
| B · 기능 연결 | 핵심 계산과 UI는 동작하지만 콘텐츠, 밸런스, 연출 또는 장기 루프가 얕다. |
| C · 기반만 존재 | 서비스나 직렬화 코드는 있으나 플레이어 흐름에서 호출되지 않거나 일부 상태만 다룬다. |
| D · 미발견 | 현재 소스에서 해당 제품 기능을 찾지 못했다. |

## 3. 전체 플레이 루프

### 3.1 순간 단위 루프

```text
건축 카탈로그에서 파츠 선택
-> 설치 가능 칸과 고스트 확인
-> 벽/문/가구 배치
-> 닫힌 방과 방 성향 자동 재계산
-> 손님/직원이 갈 수 있는 시설 후보 갱신
-> 이동, 예약, 이용 또는 작업
-> 욕구/기분/재고/매출/청결/시설 기록 변화
-> UI, 캐릭터 기록, 정산 데이터 갱신
```

### 3.2 하루 단위 루프

```text
Day 시작
-> 운영 런 변수 추첨 및 침공 위협 증가
-> 손님 입장, 소비, 직원 업무, 사건 발생
-> 180 게임초 경과
-> Day 종료 정산 생성
-> 직원 불만 평가
-> 시설 진화 기록 누적
-> 다음 날 시설 상점/배송 목록 갱신
-> 다음 Day 시작
```

### 3.3 런 단위 루프

```text
사장 선택 및 시작
-> 시설 풀/설계도/직원/런 변수 조합
-> 운영과 확장
-> 위협 상승
-> 침입자 자동 진입
-> 방어 시설과 경비가 대응
-> 침입자가 사장에게 도달하면 최종 교전
-> 사장 사망 시 런 종료
-> 성과에 따른 계승 화폐와 메타 강화
```

현재 명시적으로 연결된 런 종료 조건은 **사장 사망 하나**다. 별도의 승리, 엔딩, 목표 달성 종료 조건은 현재 런타임 코드에서 발견되지 않았다.

## 4. 런타임 구조

### 4.1 구성 방식

- VContainer의 `DungeonRuntimeLifetimeScope`가 그리드, 입력, 방, 캐릭터 AI, 시설 상점, 연구, 침공, 원정, 메타 진행 등 대부분의 서비스를 등록한다.
- 전역 사건 전달은 `EventObserver` 계열 이벤트를 사용한다.
- 읽기 전용 조회는 `Query`, `Provider`, `Catalog`, `RuntimeReader` 인터페이스로 분리되어 있다.
- 방 배치와 시설 후보는 `RoomLayoutCache`, `FacilityCandidateCache`로 캐시한다.
- 건물의 개별 기능은 `BuildingSO`의 다형성 `Ability Modules` 목록으로 조합한다.

### 4.2 주요 런타임 도메인

```text
Grid / Buildings / Rooms
Character / Work / Recruitment
Operation / FacilityShop / Research
Synthesis / FacilityEvolution
Defense / Invasion / Offense
Run / Meta / Codex
UI / Infrastructure
```

## 5. 시간, 카메라, 입력

**상태: A**

- 시작 상태는 `Day 1`, 속도 `x1`, 보유금 `5000`이다.
- 하루 길이는 실제 시간 기준 180초이며 `Time.timeScale`의 영향을 받는다.
- 속도는 `x1`부터 `x5`까지 순환한다.
- 일시 정지는 `Time.timeScale = 0`으로 동작한다.
- 시간대는 Night, Morning, Noon, Evening으로 나뉘며 하루 진행 초에 따라 자동 전환된다.
- 카메라는 WASD와 방향키로 이동하고 현재 그리드 경계 안으로 제한된다.
- 화면 가장자리 스크롤 코드는 있으나 기본값은 꺼져 있다.
- 월드 클릭은 UI 위에서 차단되며 건축/철거 모드에서는 일반 정보 선택을 받지 않는다.
- 같은 위치에 캐릭터와 건물이 겹치면 캐릭터를 먼저 선택한다. 건물끼리는 정렬 레이어, 정렬 순서, Z 위치로 앞쪽 대상을 고른다.

## 6. 그리드와 건축

**상태: A**

### 6.1 그리드 레이어

| 레이어 | 용도 |
|---|---|
| Hallway | 걸을 수 있는 바닥과 통로 |
| Building | 일반 시설, 벽, 문, 이동 시설 |
| Character | 캐릭터 점유 |
| WallFixture | 벽 부착형 파츠 |
| CeilingFixture | 천장 부착형 파츠 |
| FloorOverlay | 바닥 위 독립 장식/효과 |

벽과 문은 별도 `Wall` 점유 레이어가 아니라 `Building` 레이어의 구조물이다. 문 설치는 같은 칸의 구조 벽을 원자적으로 교체한다.

### 6.2 배치 판정

배치 시 다음 조건을 하나의 권위 판정으로 검사한다.

- 그리드 범위 안인가
- 목표 레이어가 비어 있는가
- 필요한 바닥 지지가 있는가
- 건물별 건설 조건을 만족하는가
- 현재 해금 단계에 도달했는가
- 보유금이 건설 비용 이상인가
- 문이라면 교체 가능한 구조 벽 위인가

설치 가능 영역 오버레이는 **칸 자체의 설치 가능/불가능**을 표시하고, 선택한 다중 칸 건물 전체가 들어가는지는 고스트의 유효/차단 상태로 표시한다.

### 6.3 건축 UX

- 하단 건축 탭은 카테고리와 시설 목록을 보여 주는 컴팩트 패널이다.
- 시설을 선택하면 카탈로그가 접히고 월드, 그리드, 고스트가 보인다.
- 정상 고스트는 원본 색과 읽을 수 있는 불투명도를 유지한다.
- 차단 고스트는 빨간색으로 표시한다.
- 설치 직후에는 건축 흐름을 유지하며 건물 정보창을 강제로 열지 않는다.
- 철거 시 건설비와 환급률을 기준으로 환급한다.
- 구조 지지가 깨지는 철거는 별도 안전 판정을 거친다.

### 6.4 해금 단계

모듈형 시설 해금 단계는 날짜로 계산한다.

| 단계 | 날짜 |
|---|---|
| Phase 1 | Day 1~3 |
| Phase 2 | Day 4~6 |
| Phase 3 | Day 7 이후 |

## 7. 건물과 능력 모듈

**상태: A**

### 7.1 데이터 모델

`BuildingSO`는 배치 크기, 레이어, 스프라이트 같은 정적 정의와 다형성 `Ability Modules` 목록을 가진다. 새 능력은 `BuildingAbility` 파생 클래스를 추가하고 건물 에셋의 목록에서 선택해 세부 값을 조정하는 구조다.

현재 능력 타입은 25개다.

| 그룹 | 능력 |
|---|---|
| 비용/기본 | 경제, 시설 운영, 시설 등급, 의미 태그, 종족 선호 |
| 재고/상업 | 내부 재고, 이용 시 재고 필요, 보관, 판매, 서비스, 생산 |
| 공간 | 정식 방 필요, 독립 공간, 시설 파츠, 좌석, 테이블 |
| 직원/이용 | 직원 서비스 필요, 욕구 회복, 훈련, 청소 |
| 전투/진행 | 방어 시설, 경비, 진화 기여, 범죄 위험 보정 |
| 시각 | 조명 |

### 7.2 능력 실행 지점

능력은 단순 데이터 표시가 아니라 다음 런타임 훅에 참여한다.

- 시설 비주얼 생성 시 조명과 특수 렌더링 적용
- 손님/직원의 시설 이용 완료 시 욕구, 훈련 등 적용
- 직원 작업 완료 시 생산, 청소 등 적용
- 저장 시 능력별 상태 모듈 캡처
- 재고 분류와 소비 신호 제공
- 절도/범죄 위험 계산 보정

### 7.3 건물 런타임 상태

- 시설 레벨
- 파손 여부
- 완료된 이용 횟수
- 완료된 작업 횟수
- 청결도 0~100
- 내부 재고 또는 창고 재고
- 생산량, 방어 충전 등 능력별 상태 모듈
- 현재 이용자와 예약 수

파손은 현재 연속 내구도 HP보다 `정상/파손` 상태에 가깝다. 침입자가 시설을 파손 상태로 만들고 직원의 수리 작업이 이를 정상으로 되돌린다.

## 8. 방 시스템

**상태: A**

### 8.1 방 생성 원칙

방은 통짜 건물 에셋이 아니다. 실제 그리드에서 다음 요소를 조합해 실시간으로 검출한다.

```text
통로 바닥 + 외벽/내벽 + 문 + 내부 시설 파츠
```

- 구조 벽으로 경계가 완전히 닫혀야 한다.
- 문이 한 개 이상 있어야 사용 가능한 정식 방이다.
- 열린 경계는 `OpenBoundary`다.
- 닫혔지만 문이 없으면 `MissingDoor`다.
- 기존 `SelfContained` 시설은 호환 판정은 유지하지만 방 UI와 환경 효과에서 제외된다.

### 8.2 방 성향

내부 시설이 기여한 역할을 합산해 방 성향을 만든다. 한 방은 여러 성향을 동시에 가질 수 있다.

```text
식사 / 구매 / 휴식 / 훈련 / 연구 / 마나
물류 / 화장실 / 위생 / 행정 / 경비
```

예를 들어 연구 파츠와 마나 파츠가 함께 있으면 `연구 + 마나` 복합방으로 표시한다. 가장 많은 파츠가 기여한 역할이 유일하면 그 역할의 대표색을 사용하고, 동률이면 복합방 중립색을 사용한다.

### 8.3 환경 수치

정식 방은 다음 네 수치를 0~100으로 계산한다.

| 수치 | 현재 계산 요소 |
|---|---|
| 넓이 | 방 면적 정규화 45% + 빈 칸 비율 55% |
| 미관 | 기본 50 + Luxury 기여 - 파손 비율 - 과밀 패널티 |
| 청결 | 기본 60 + Hygiene + 청소 기록 + 운영 청결 - 파손 - 과밀 |
| 인상도 | 미관 35% + 넓이 30% + 청결 20% + 방 품질 15% |

등급 표시는 `끔찍함 0~19`, `나쁨 20~39`, `평범함 40~59`, `좋음 60~79`, `훌륭함 80~100`이다. 가중치와 임계값은 `RoomEnvironmentSettingsSO`에서 조정한다.

### 8.4 기분 연동

정식이며 사용 가능한 방에서 시설 이용이나 작업을 마치면 방 경험이 180초 동안 기분 요인으로 적용된다.

| 조건 | 기분 변화 |
|---|---:|
| 인상도 <20 | -6 |
| 인상도 <40 | -3 |
| 인상도 40~59 | 0 |
| 인상도 60~79 | +3 |
| 인상도 >=80 | +6 |
| 청결 <20 | -4 |
| 청결 <40 | -2 |
| 청결 40~79 | 0 |
| 청결 >=80 | +2 |

같은 방의 같은 요인은 중첩하지 않고 갱신한다.

### 8.5 방 보기 UI

- 우측 상단 `방` 토글로 방 보기 모드를 켠다.
- 커서 아래의 정식 방만 반투명 채움과 외곽선으로 강조한다.
- 오버레이는 바닥 위, 벽/문/가구/캐릭터 아래에 표시한다.
- 패널은 방 이름, 사용 가능 상태, 환경 막대, 구조 수치, 성향 기여 시설을 표시한다.
- 열린 경계는 위험색, 문 없는 공간은 경고색, 사용 가능한 방은 성향색을 사용한다.
- 건축/철거 모드에 들어가면 방 보기를 자동 종료한다.

## 9. 캐릭터 모델

**상태: A**

### 9.1 캐릭터 구성

- 이름, 종족, 특성, 역할, 직원/손님/사장 여부
- 체력과 부상/사망
- 9개 능력치
- 5개 감소형 욕구
- 기본 기분과 욕구/상호작용 기분 요인
- AI 블랙보드와 현재 행동
- 활동 기록, 대화, 페르소나, 사회 기억
- 입장, 활동, 퇴장, 원정 등 생명주기 상태

### 9.2 능력치

```text
공격 / 판매 / 연구 / 이동 / 근력
맷집 / 민첩 / 청소 / 지구력
```

능력치는 안정 ID 카탈로그로 등록되어 있어 새 능력치를 기존 `CharacterStatBlock` 필드 확장 없이 추가할 수 있다.

### 9.3 욕구와 기분

욕구는 `포만감`, `휴식`, `재미`, `배변`, `위생` 다섯 가지다. `기분`은 욕구가 아니라 별도 파생 상태다.

```text
최종 기분 = 기본 기분 55 + 활성 욕구 요인 합 + 상호작용 요인 합
```

- 욕구가 매우 낮으면 큰 부정 요인이 생긴다.
- 욕구가 높으면 작은 긍정 요인이 생긴다.
- 방 경험, 사회적 사건, 활동 결과는 지속시간이 있는 상호작용 요인이 된다.
- 기분 라벨은 절망적, 불쾌함, 가라앉음, 평온함, 만족함, 아주 좋음으로 나뉜다.

### 9.4 기록과 대화

- 활동은 구조화된 이벤트로 먼저 기록된다.
- 플레이어에게 보일 문장은 로컬 LLM 큐가 짧은 한국어 기록으로 다시 쓴다.
- 변환 중인 원문은 기록 탭에 노출하지 않는다.
- LLM 실패, 큐 포화, 응답 검증 실패 시 결정론적 한국어 폴백을 사용한다.
- 게임 상태 확정은 LLM 응답과 무관하게 즉시 진행된다.

## 10. 캐릭터 AI와 이동

**상태: A**

### 10.1 의사결정

- 캐릭터 AI는 행동 후보를 만들고 효용 점수로 다음 행동을 선택한다.
- 행동 후보는 욕구, 시설 역할, 경로 존재, 방 유효성, 재고, 직원 배치, 혼잡, 가격, 종족 선호, 과거 방문, 기분을 본다.
- 현재 에셋은 행동 9개와 고려 요소 16개, 합계 25개 AI 에셋이다.
- 결정 스케줄러와 시설 후보 캐시가 모든 캐릭터의 무거운 검색이 같은 프레임에 몰리지 않게 한다.

### 10.2 이동

- 수평 보행과 계단/엘리베이터/텔레포트 연결을 `GridMoveStep`으로 통합한다.
- 경로 탐색은 현재 그리드 버전을 기준으로 BFS 탐색 결과를 만든다.
- 벽 변경 등으로 그리드 버전이 바뀌면 필요한 시점에 경로를 다시 계획한다.
- 실제 보행 스텝 직전에 막힘을 확인해 벽을 통과하지 않도록 한다.
- 시설 이용 종료 위치는 멀리 떨어진 같은 층 임의 칸이 아니라 시설이 점유한 유효 칸을 기준으로 복귀한다.

### 10.3 손님 행동

```text
던전 밖 스폰
-> 입구 진입
-> 식사/쇼핑/휴식/화장실/위생/구경/대기 선택
-> 시설 예약
-> 이동 및 이용
-> 욕구, 기분, 방문 기록, 재고, 매출 반영
-> 만족도와 상황에 따라 다음 행동 또는 퇴장
```

동일 고객의 재방문 기록은 단골과 영입 후보 판정에 사용된다.

## 11. 직원과 업무

**상태: A**

### 11.1 업무 종류

```text
운영 / 보충 / 수리 / 청소
연구 / 경비 / 구조 / 휴식
```

- 직원별 업무 우선순위는 `1`, `2`, `3`, `끔`으로 설정한다.
- 업무 대상 예약으로 여러 직원의 중복 작업을 방지한다.
- 보충은 창고에서 재고를 가져와 대상 시설에 채우는 실제 운반 흐름을 사용한다.
- 사장은 특정 시설 우선 작업, 반란 제압 같은 직접 명령을 내릴 수 있다.
- 비번 직원은 업무 대신 욕구와 휴식을 우선한다.

### 11.2 불만과 반란

일일 종료 시 직원 기분과 누적 저기분 일수를 평가한다.

| 단계 | 기본 기준 | 영향 |
|---|---|---|
| 안정 | 기분 >50 | 정상 효율 |
| 낮은 만족 | 기분 <=50 | 효율 95% |
| 효율 저하 | 기분 <=35 또는 저기분 2일 | 효율 80% |
| 태업/결근 | 기분 <=25 또는 저기분 3일 | 효율 60%, 업무 차단 |
| 이탈 | 기분 <=15 또는 저기분 4일 | 업무 불가, 영구 이탈 가능 |
| 지역 반란 | 기분 <=8 | 반란, 사장 위협으로 확대 가능 |

대응은 자동 제압, 제압 명령, 격리, 진정으로 나뉜다.

## 12. 상점, 재고, 경제

**상태: B**

### 12.1 재고

재고 분류는 `식량`, `잡화`, `무기`, `마나` 네 종류다.

- 상점은 내부 판매 재고를 가진다.
- 창고는 분류별 또는 전체 허용 용량을 가진다.
- 생산 시설은 작업 완료 시 지정 재고를 만든다.
- 보충 요청 기준 이하가 되면 직원 물류 대상이 된다.
- 일일 배송 상점에서 재고를 구매할 수 있다.

### 12.2 판매

- 손님은 상품을 카트에 담고 지불한다.
- 상품 가격, 종족/상점 특화, 런 변수 가격 배율이 최종 소비에 영향을 준다.
- 직원이 필요한 상점과 셀프서비스 상점을 구분한다.
- 재고가 실제로 차감되고 보유금이 증가한다.
- 돈이 부족한 손님과 내부 직원 이용은 별도 처리한다.
- 절도는 재고 손실, 범죄 기록, 알림, 정산 사건으로 연결된다.

### 12.3 일일 정산

정산은 다음 항목을 집계한다.

- 총매출, 방문 수, 평균 만족도
- 시설별 매출, 종족별 방문
- 소비 재고와 창고 재고
- 재고 부족과 보충 실패
- 파손 시설과 수리 비용 예상
- 사건, 직원 불만, 직원 근무 요약
- 다음 날 재고 배송과 시설 상점 목록

중요한 현재 한계:

- 건물의 `일일 유지비` 값은 파손 시설의 **수리 비용 예상**에 합산될 뿐, 하루 종료 시 보유금에서 실제 차감되지 않는다.
- 직원 급여/임금 시스템은 현재 런타임 코드에서 발견되지 않았다.
- 따라서 현재 경제는 건설, 재고 구매, 시설/설계도 구매가 주요 지출이고 정기 비용 압력은 약하다.

## 16. 시설 진화

**상태: B**

시설 진화는 단순 레벨업이 아니라 해당 시설과 방이 실제로 어떻게 운영되었는지를 기록해 후보를 만든다.

### 16.1 입력 기록

- 정식 방의 성향과 환경
- 이용, 매출, 재고, 청소, 파손, 방어 등 운영 사건
- 연구 완료
- 재료와 기록 토큰
- 시설 정체성과 기존 계보

### 16.2 후보 생성

- 작성된 진화 규칙을 먼저 평가한다.
- 로컬 LLM이 사용 가능하면 맥락 기반 후보를 제안할 수 있다.
- 제안은 허용된 결과, 비용, 계보 규칙으로 다시 검증한다.
- 실패하거나 부적합하면 결정론적 후보를 사용한다.
- 승인된 진화는 기존 시설을 결과 시설로 교체하고 필요한 상태를 이전한다.

현재 작성된 진화 레시피는 4개이며 식당 계보에 집중되어 있다. 엔진은 범용적이지만 실제 콘텐츠 분포는 아직 좁다.

## 17. 침공 위협

**상태: B**

### 17.1 위협 계산

위협은 다음 요소로 시간당 상승한다.

- 던전 가치
- 평판
- 마지막 침공 이후 시간
- 잔여 위험
- 운영 날짜 보너스
- 런 변수와 메타 경고 보정

단계는 `평화 -> 경고 -> 침입 후보 -> 침공 -> 안전시간` 흐름이다.

### 17.2 자동 시작

위협이 후보 임계값을 넘고 지연 시간이 끝나면 `InvasionCandidateEvent`가 발생한다. `InvasionDirectorRuntime`가 이를 직접 수신해 입구를 찾고 침입자를 자동 생성한다. 즉 현재 침공은 UI의 테스트 버튼만으로 시작되는 구조가 아니라 정상 런타임에서도 자동 연결되어 있다.

### 17.3 침입자 행동

- 외부 스폰 지점에서 던전 문으로 접근한다.
- 사장 위치를 향해 경로를 다시 계산한다.
- 근처 주요 시설을 파손 상태로 만든다.
- 방어 시설과 경비의 공격을 받는다.
- 사장에게 도달하면 정해진 최종 교전 피해를 준다.
- 사장이 생존하면 방어 성공, 사망하면 런 종료다.

## 18. 방어 시설

**상태: B**

- 발동 조건, 대상, 범위, 재사용 대기시간, 별 등급을 가진다.
- 현재 효과는 직접 피해, 부식, 연소, 충전, 감속, 경비 공격 등을 포함한다.
- 방어 상태 서비스가 활성 시설과 쿨다운을 추적한다.
- 발동과 침공 결과는 전투 리포트와 도감 사건에 남는다.
- 현재 구체 방어 효과 에셋은 24개다.

전투는 배치와 준비가 중심인 자동 전투다. 플레이어가 침공 중 캐릭터를 직접 조작하는 액션 전투 시스템은 아니다.

## 19. 원정과 오펜스

**상태: B**

### 19.1 월드맵

- 정찰 레벨은 0~3이며 레벨에 따라 탐색 거리가 늘어난다.
- 기본 목표는 5개다.

| 목표 | 거리 | 위험 | 시간 | 주요 보상 |
|---|---:|---:|---:|---|
| 외곽 식재료 농장 | 4 | 8 | 55초 | 식량, 돈 |
| 상단 길목 | 6 | 14 | 70초 | 잡화, 돈 |
| 낡은 무기고 | 12 | 28 | 100초 | 무기, 방어 설계도 |
| 마력 유적 | 18 | 38 | 130초 | 마나, 특수 설계도 |
| 경쟁 던전 전초기지 | 26 | 55 | 180초 | 희귀 시설, 세력 약화, 영입 후보, 특수 몬스터 |

### 19.2 원정

- 목표별 최소 인원과 요구 전투력이 있다.
- 직원을 편성하면 해당 캐릭터의 생명주기가 `OnExpedition`으로 바뀐다.
- 경과 시간이 끝나면 파티 전투력과 위험도로 결과를 계산한다.
- 실패/피해는 캐릭터 체력에 반영된다.
- 완료 보상은 돈, 재고, 시설, 설계도, 세력 약화, 영입 후보 등 실제 런타임 상태에 적용된다.

현재 시스템은 완결된 한 번의 원정 흐름을 제공하지만 목표 수가 5개로 작고 월드맵의 장기 변화는 제한적이다.

## 20. 단골과 영입

**상태: B**

- 방문 기록은 고객별 방문 횟수와 만족도를 누적한다.
- 기본적으로 3회 방문, 만족도 65 이상이면 단골이 된다.
- 4회 방문, 만족도 75 이상이면 영입 후보가 된다.
- 영입한 캐릭터는 이후 일반 손님 스폰 풀에서 제외된다.
- 영입 캐릭터는 직원, 방어 인력, 원정 인력으로 활용할 수 있다.

현재 종족과 특성 콘텐츠가 작아 반복 플레이에서 후보 다양성이 빨리 소진될 가능성이 있다.

| 사장 생존 | 시작 사장 특성 후보 +1, 사장 최대 체력 증가, 침입 경고 조기화 |
| 설계 보존 | 특수 조합식 기록 보존 슬롯 |
| 전략 강화 | 상업/물류 비용 절감, 요새/방어 구매 절감, 비전/연구 작업 증가 |

기존 운영 지식, 사장 생존, 설계 보존 계열에 상업/물류, 요새/방어, 비전/연구 전략 강화가 더해졌다. 각 업그레이드는 비용, 최대 레벨, 정수 보너스 또는 배율 효과를 가지며 권위 있는 런타임 계산 경로에서 소비된다.

메타 프로필은 런 저장 슬롯과 분리해 영구 보존한다. 승패 보상은 중복 지급되지 않으며 다음 런은 자동/빠른/수동 런 저장을 비우되 메타 화폐, 업그레이드, 보존 조합식을 유지한다.

## 23. 도감과 기록

**상태: B**

도감은 런타임 이벤트를 수신해 항목을 채운다.

- 관찰한 종족과 시설
- 연구한 설계도
- 발견한 합성/진화 결과
- 방어 발동
- 침공과 전투 리포트
- 출처와 해금 계기

카테고리, 항목 목록, 선택 상세, 기록 아카이브가 UI에 연결되어 있다. 다만 도감 수집이 별도의 보상이나 목표 달성 체계로 이어지는 연결은 현재 약하다.

## 24. UI 표면

**상태: A**

### 24.2 주요 연결

- 창고: 재고 현황과 배송 구매
- 운영: 일일 정산, 단골 영입, 런 변수, 메타 강화, 경제 상세
- 방어: 위협도, 활성 침입자, 방어 시설, 전투 리포트
- 원정: 월드맵, 목표 선택, 편성, 진행, 보상 기록
- 연구: 설계도 대기열과 진행

기존 UI 감사 기준으로 P0/P1/P2 기능 행은 모두 실제 버튼 입력과 상태 변화까지 연결되어 있다. 다만 일부 `+진행`, `위협 증가`, `즉시 발동`류 버튼은 기능 접근과 QA를 겸하는 가속 조작이므로 최종 게임의 자연스러운 진행 연출과 동일하다고 보기는 어렵다.

## 25. 저장과 불러오기

**상태: C**

### 25.1 구현된 직렬화

`ModularFacilityWorldSaveService` v2는 다음을 JSON으로 캡처하고 복원할 수 있다.

- 그리드 크기
- 게임 속도, 보유금, 날짜, 현재 시간, 시각, 시간대
- 모든 배치 건물의 ID, 코드, 레이어, 위치, 크기
- 파손 여부와 시설 레벨
- 시설별 상태 모듈
- v1 저장 데이터 마이그레이션과 경고 보고

### 25.2 현재 연결되지 않은 부분

- 프로덕션 코드에서 `CreateSnapshot`, `ToJson`, `TryRestoreJson`을 호출하는 플레이어 저장 흐름이 없다.
- 저장 슬롯, 파일 경로 소유자, 저장 버튼, 자동 저장, 시작 시 불러오기가 없다.
- 게임 시작 시 `GameManager`가 Day 1, 5000골드로 다시 초기화한다.

### 25.3 통합 저장 범위에서 빠진 상태

- 캐릭터와 직원의 체력, 욕구, 기분, 기록, 업무 상태
- 연구 대기열과 완료 상태
- 시설 상점 해금과 일일 목록
- 런 변수와 메타 진행
- 단골/영입 기록
- 도감
- 침공, 원정, 전투 리포트
- 정산 히스토리

따라서 현재 저장 서비스는 **건물 월드 상태 직렬화 기반**이며, 실제 런 전체를 이어 하는 저장 시스템은 아니다.

## 28. 현재 시스템의 강한 연결 고리

### 28.1 공간이 시뮬레이션에 영향을 준다

```text
벽/문 배치
-> 방 유효성
-> 방 성향과 환경
-> 시설 후보와 동선
-> 이용 결과와 기분
-> 직원 불만과 고객 만족
```

이 연결은 현재 게임의 가장 분명한 차별점이다.

### 28.2 운영이 침공 위험을 만든다

```text
시설과 운영 성장
-> 던전 가치/평판 상승
-> 침공 위협 상승
-> 침입자 자동 생성
-> 시설 파손/사장 최종 교전
-> 수리와 다음 방어 준비
```

### 28.3 손님이 장기 자원이 된다

```text
좋은 시설 이용
-> 만족도와 재방문
-> 단골
-> 영입 후보
-> 직원/방어/원정 인력
```

### 28.4 운영 기록이 시설 진화 재료가 된다

```text
방 성향 + 시설 이용 + 연구 + 사건
-> 기록 토큰과 진화 압력
-> 후보 생성과 검증
-> 시설 계보 변화
```

## 29. 현재 가장 큰 단절

1. **인간 기준 페이싱**: 자동화는 진행 가능성과 성능을 증명하지만 기다림, 클릭 부담, 보상 간격이 재미있는지는 판정하지 못한다.
2. **콘텐츠 폭**: 범용 엔진에 비해 종족, 캐릭터, 설계도, 진화 계보, 원정 목표, 침입자 종류가 적다.
3. **보상 연출**: 희귀 획득, 방 완성, 계보 발견, 위기 돌파를 더 강하게 축하할 여지가 있다.
4. **반복 동기**: 세 전략은 자연 승리했지만 실제 플레이어가 첫 승리 뒤 다른 빌드를 선택하는지는 인간 세션으로 확인해야 한다.

## 30. 현재 게임을 한 문장으로 평가하면

현재 DungeonStory는 **방을 직접 조립하고 그 공간 안에서 캐릭터가 실제로 먹고, 사고, 일하고, 불만을 품으며, 그 운영 결과가 침공과 시설 성장으로 이어지는 시뮬레이션 기반은 강하게 구축된 상태**다.

현재는 **새 게임부터 운영, 최종 침공, 승패, 저장 복원, 계승과 다음 런까지 공개 UI로 이어지는 기능 완성형 출시 후보**다. 남은 판정은 시스템 존재 여부가 아니라 실제 첫 플레이에서 이해되고 재미있게 반복되는지에 대한 인간 검증이다.
