# 재고와 물류

재고와 물류는 경영 시설을 실제로 굴러가게 만드는 운영 시스템이다.

이 문서는 다음 질문을 정리한다.

- 재고를 어디까지 나눌 것인가?
- 창고와 시설 재고는 어떻게 연결되는가?
- 직원은 어떤 규칙으로 재고를 옮기는가?
- 재고 부족은 손님/직원 불만과 어떻게 연결되는가?

## 현재 구현 기준

현재 상점 구현은 다음 구조를 가진다.

```text
Shop
-> StockInfo
-> SaleItem 목록과 수량
-> 손님이 구매하면 Shop 내부 재고 차감
```

즉 현재는 `상점 내부 재고`가 먼저 구현되어 있다.

앞으로의 설계는 이 구조를 유지하되, 창고와 보충 작업을 추가해 다음 구조로 확장한다.

```text
창고 재고
-> 직원 보충 작업
-> 시설 내부 재고
-> 손님 구매/이용
-> 시설 내부 재고 감소
```

## 재고 성장 흐름

재고 수급은 게임 진행에 따라 역할이 바뀐다.

핵심 흐름:

```text
초반:
돈 -> 운영일 납품 -> 창고 재고 -> 시설 운영 -> 돈

중반:
돈 + 내부 생산 + 침입 방어 보상 -> 재고 안정화 -> 더 높은 수익

후반:
내부 생산 + 합성 시설 + 종족 특화 운영 + 오펜스 보상
-> 고급 재고/고급 시설 운영
-> 고수익/강한 방어/상위 합성
```

초반에는 돈으로 재고를 사서 던전을 굴리는 구조가 중심이다.

중반부터는 내부 생산 시설과 침입 방어 보상이 기본 재고 의존도를 줄이고, 던전 운영을 안정화한다.

후반에는 내부 생산과 원정 보상으로 고급 재고를 확보하고, 이를 합성 시설과 종족 특화 운영에 투입해 돈, 방어력, 합성 효율을 함께 끌어올린다.

후반에도 돈은 계속 필요하다. 다만 초반처럼 재고 구매에만 쓰는 것이 아니라 시설 확장, 연구, 합성, 원정 준비에 더 크게 쓰인다.

## 재고 분류 원칙

초기에는 재고를 너무 세분화하지 않는다.

재고는 두 단계로 나눈다.

```text
재고 계열: 물류와 창고가 관리하는 큰 분류
판매 품목: 손님이 실제로 사거나 시설이 소비하는 세부 아이템
```

예시:

```text
재고 계열: 식재료
판매 품목: 저가 식사, 고기 요리, 피 식사

재고 계열: 잡화
판매 품목: 소모품, 장난감, 생활용품
```

물류 AI와 창고는 `재고 계열` 중심으로 움직이고, 상점 UI와 구매 연출은 `판매 품목`을 보여준다.

이렇게 하면 품목 다양성은 만들 수 있지만, 직원 물류가 너무 복잡해지지 않는다.

## 초기 재고 계열

첫 구현에서는 4개 계열만 사용한다.

```text
식재료
잡화
무기
마력
```

### 식재료

식당 계열 시설이 소비하는 재고다.

사용 시설:

- 저가 음식점
- 고기 식당
- 피 식당 확장
- 전투 식당 확장

운영 의미:

- 식재료가 부족하면 식당 매출이 멈춘다.
- 식사 욕구를 해결하지 못한 손님과 직원의 불만이 오른다.
- 오크는 고기 식당 재고 부족에 특히 민감하다.

초기에는 `기본 식재료`와 `고기`를 별도 물류 계열로 나누지 않는다.

대신 판매 품목과 시설 태그로 차이를 표현한다.

```text
고기 식당은 식재료 계열을 소비하지만, 오크 만족도 보정과 높은 가격을 가진다.
```

### 잡화

상점 계열의 범용 판매 재고다.

사용 시설:

- 잡화점
- 간식점 확장
- 놀이/오락 계열 확장

운영 의미:

- 잡화는 대량 손님과 수인 확장에 어울린다.
- 단가는 낮거나 중간이지만 회전율이 좋다.
- 잡화점 재고 부족은 손님 체류 시간 감소와 구매 실패로 이어진다.

### 무기

오크, 훈련, 경비와 연결되는 고단가 재고다.

사용 시설:

- 무기점
- 훈련장 확장
- 전투 보급소 확장
- 경비/병영 계열 확장

운영 의미:

- 무기 재고는 단가가 높다.
- 오크 만족도와 전투/훈련 시설 시너지에 영향을 준다.
- 침입자는 무기 관련 시설과 창고를 우선 파괴 대상으로 삼을 수 있다.

### 마력

연구, 마력 시설, 뱀파이어/데몬 고객층과 연결되는 재고다.

사용 시설:

- 마력 저장소
- 연구실 확장
- 마력 연구실
- 마력 상점
- 저주/공포 계열 확장

운영 의미:

- 마력 재고는 즉시 매출보다 연구와 합성 해금에 영향을 준다.
- 뱀파이어와 데몬 만족도에 영향을 준다.
- 마력 시설이 파손되면 장기 성장과 특정 고객층 운영이 흔들린다.

## 후반 확장 재고

첫 구현에서는 쓰지 않지만, 후반에 다음 계열을 추가할 수 있다.

```text
수리 자재
희귀 전리품
제물/피
장식품
연구 자료
```

주의할 점:

- 새 재고 계열은 새 직원 작업이나 새 의사결정을 만들어야 한다.
- 단순히 이름만 다른 재고는 추가하지 않는다.
- 재고 계열이 늘어나면 창고 UI와 보충 AI가 복잡해진다.

초기 수리는 별도 `수리 자재` 없이 돈과 작업 시간만 사용한다.

수리 자재는 시설 파손/침입 피해가 충분히 재미있어진 뒤 확장한다.

## 창고

창고는 던전 전체 재고의 기준점이다.

역할:

- 재고 계열별 보관량을 가진다.
- 보충 작업의 출발점이 된다.
- 창고 위치가 직원 동선에 영향을 준다.
- 침입이나 반란으로 파손되면 여러 시설 운영이 동시에 흔들린다.

초기 창고 규칙:

```text
창고는 모든 초기 재고 계열을 저장할 수 있다.
창고에는 계열별 보관량과 총 보관량이 있다.
시설은 창고에서 재고를 직접 쓰지 않고, 내부 재고로 보충받는다.
```

예시:

```text
창고
- 식재료 40
- 잡화 30
- 무기 10
- 마력 8
```

후반에는 전문 창고를 추가할 수 있다.

```text
식재료 창고
무기 창고
마력 보관고
전리품 창고
```

하지만 첫 구현에서는 범용 창고 하나로 충분하다.

## 시설 내부 재고

각 시설은 자체 내부 재고를 가진다.

역할:

- 손님이 이용할 때 바로 소비된다.
- 직원이 보충 작업으로 채운다.
- 시설 UI에서 부족 상태를 보여준다.

예시:

```text
저가 음식점
- 필요 재고 계열: 식재료
- 내부 재고 최대: 20
- 보충 요청 기준: 6 이하

무기점
- 필요 재고 계열: 무기
- 내부 재고 최대: 8
- 보충 요청 기준: 2 이하
```

시설 내부 재고가 너무 적으면 직원이 계속 보충하느라 바빠진다.

시설 내부 재고가 너무 많으면 창고와 재고 관리 의미가 약해진다.

## 보충 작업

보충 작업은 자동으로 생성된다.

작업 생성 조건:

```text
시설 내부 재고가 보충 요청 기준 이하
창고에 해당 재고가 있음
시설이 파손 상태가 아님
시설까지 도달 가능한 경로가 있음
```

작업 흐름:

```text
시설 재고 부족 감지
-> 보충 작업 생성
-> 보충 우선순위가 있는 직원이 작업 선택
-> 창고로 이동
-> 재고 운반량 예약
-> 시설로 이동
-> 시설 내부 재고 증가
-> 작업 완료
```

직원이 운반할 수 있는 양은 능력에 영향을 받는다.

```text
운반량 = 기본 운반량 + 힘 보정 + 특성/종족 보정
```

작업 속도는 이동속도와 동선에 영향을 받는다.

```text
보충 시간 = 창고까지 이동 + 시설까지 이동 + 적재/하역 시간
```

## 누가 재고를 옮기는가

재고 보충은 직원 작업이다.

작업 우선순위의 `보충` 항목을 사용한다.

예시:

```text
수인 직원
- 보충 1
- 근무 2
- 청소 3

오크 직원
- 경비 1
- 수리 2
- 보충 3

슬라임 직원
- 청소 1
- 보충 2
- 근무 3
```

종족/능력 의미:

- 이동속도가 높은 직원은 짧은 보충 작업에 강하다.
- 힘이 높은 직원은 한 번에 많은 재고를 옮긴다.
- 수인은 보충/정찰/상점 운영에 유리하다.
- 오크는 무거운 무기 재고나 수리 작업에 유리할 수 있다.
- 슬라임은 보충보다 청소/오염 처리에 강하다.

첫 구현에서는 재고 무게를 세분화하지 않는다.

대신 재고 계열별 기본 운반 난이도만 둘 수 있다.

```text
식재료: 보통
잡화: 가벼움
무기: 무거움
마력: 보통, 마력 적성 보정 가능
```

## 보충 우선순위

모든 재고 부족이 같은 긴급도를 갖지는 않는다.

보충 작업은 다음 순서로 긴급도를 계산한다.

```text
품절 시설
-> 식사 시설 부족
-> 직원 유지에 필요한 시설 부족
-> 고수익 시설 부족
-> 일반 상점 부족
```

초기 규칙:

- 품절 상태는 가장 높은 보충 긴급도를 가진다.
- 식당 재고 부족은 손님과 직원 불만에 모두 영향을 주므로 우선도가 높다.
- 무기점/잡화점은 매출 손실 중심이다.
- 마력 재고 부족은 연구/특정 종족 만족도 중심이다.

플레이어는 특정 시설을 우클릭 또는 시설 UI로 `보충 우선` 대상으로 지정할 수 있다.

이 명령은 수동 운반 조작이 아니라 자동 작업의 우선순위를 높이는 방식이다.

## 재고 부족 단계

재고 부족은 단계적으로 표시한다.

```text
정상
-> 낮음
-> 부족
-> 품절
```

### 정상

내부 재고가 충분하다.

효과:

- 시설 정상 운영
- 손님/직원 불만 없음

### 낮음

보충 요청 기준 아래로 내려간 상태다.

효과:

- 보충 작업 생성
- 시설 UI에 낮은 재고 표시
- 아직 손님 불만은 작거나 없음

### 부족

현재 수요를 따라가지 못하는 상태다.

효과:

- 대기 시간이 늘 수 있다.
- 일부 손님은 다른 시설을 찾는다.
- 캐릭터 로그에 `재고 부족` 기록
- 운영일 정산에 재고 부족 시설 표시

### 품절

더 이상 판매/이용할 수 없는 상태다.

효과:

- 해당 시설 이용 실패
- 손님 만족도 하락
- 직원 근무 효율 저하
- 상점 매출 중단
- 사고 위험 증가

## 손님 불만 연결

재고 부족은 손님 행동 실패의 중요한 원인이다.

흐름:

```text
손님이 시설 도착
-> 재고 없음
-> 구매/식사 실패
-> 다른 시설 재탐색 또는 퇴장
-> 만족도 하락
-> 캐릭터 로그 기록
```

로그 예시:

```text
재고 부족
식사 실패
원하는 상품 없음
대기 중 품절
```

종족별 영향:

```text
슬라임:
저가 음식점 품절 시 대량 불만 발생

오크:
고기 식당/무기점 품절 시 난동 위험 증가

뱀파이어:
마력 시설 부족 시 체류 가치 감소, 공포 사고 위험 증가

수인:
잡화점/놀이 시설 품절 시 소동 위험 증가

데몬:
마력/희귀품 부족 시 저주 위험 증가
```

## 직원 불만 연결

직원도 재고 부족의 영향을 받는다.

직원 불만 원인:

- 식사 재고 부족으로 비번 회복 실패
- 근무 시설의 재고가 없어 일을 못함
- 보충 작업이 계속 밀려 과로 발생
- 창고가 멀어 반복 이동으로 피로 증가

로그 예시:

```text
식사 부족
근무 재고 없음
보충 지연
창고 거리 멂
```

효과:

- 피로 회복 실패
- 직원 만족도 하락
- 근무 효율 저하
- 태업/결근/사고 위험 증가

## 창고 부족

시설 내부 재고가 부족할 때 창고에도 재고가 없으면 보충 작업을 완료할 수 없다.

흐름:

```text
시설 재고 부족
-> 보충 작업 생성
-> 창고 재고 없음
-> 보충 실패
-> 시설 품절 지속
-> 손님/직원 불만 증가
```

표시:

- 시설 UI: `창고 재고 없음`
- 직원 로그: `보충 실패`
- 정산 창: `창고 부족 계열`

정산 예시:

```text
식재료 부족: 저가 음식점, 고기 식당
창고 식재료: 0
보충 실패: 3회
```

## 원재료 수급

창고에 들어가는 재고는 단순히 돈으로만 사지 않는다.

기초 재고는 안정적으로 구할 수 있어야 하지만, 좋은 재고와 대량 재고는 경영, 침입, 오펜스 루프와 연결되어야 한다.

수급 원칙:

```text
기초 수급: 게임이 멈추지 않게 보장
운영 수급: 시설과 직원으로 만들어냄
전투 수급: 침입 방어와 원정 보상으로 얻음
보정 수급: 운이 나쁠 때 계약/상점으로 보완
```

### 1. 운영일 납품

가장 기본적인 수급 방식이다.

운영일 정산 후 상인이나 납품 계약을 통해 재고 묶음을 구매한다.

```text
식재료 묶음
잡화 묶음
무기 묶음
마력 묶음
```

역할:

- 초반 던전이 품절로 멈추는 것을 막는다.
- 돈을 재고로 바꾸는 기본 지출처가 된다.
- 첫 구현에서 가장 단순하게 만들 수 있다.

제한:

- 낮은 등급/기초 재고 중심이다.
- 가격은 운영일마다 조금 변동될 수 있다.
- 고급 재고와 희귀 재고는 납품만으로 충분히 해결되지 않아야 한다.

운영일 납품은 `안전망`이지, 최종 성장 루프가 아니다.

### 2. 내부 생산

중반 이후에는 던전 안에서 일부 재고를 직접 생산할 수 있어야 한다.

생산 시설은 돈을 바로 벌기보다 창고 재고를 안정화하는 역할을 한다.

예시:

```text
버섯 재배실
- 생산: 식재료
- 작업: 관리, 보충
- 의미: 식당 운영 안정화

잡동사니 공방
- 생산: 잡화
- 작업: 근무, 보충
- 의미: 잡화점과 수인 계열 확장 기반

무기 정비소
- 생산: 무기
- 작업: 손재주, 힘
- 의미: 무기점, 경비, 오크 계열 운영 보조

마력 응축기
- 생산: 마력
- 작업: 연구, 마력 관리
- 의미: 연구실, 마력 저장소, 뱀파이어/데몬 운영 보조
```

초기 프로토타입에서는 내부 생산을 필수로 넣지 않는다.

먼저 운영일 납품과 창고/보충 흐름을 검증한 뒤, 생산 시설을 연구로 해금한다.

해금 흐름:

```text
기본 경영 안정
-> 재고 부족 문제 경험
-> 생산 시설 연구 해금
-> 특정 재고 계열을 던전 안에서 생산
```

### 3. 침입 방어 보상

모험가 침입을 막으면 일부 재고를 얻는다.

침입자는 약탈자이기도 하지만 장비와 보급품을 들고 들어오는 존재이므로, 방어 성공은 재고 수급 루프와 연결된다.

예시:

```text
일반 모험가: 잡화, 식재료 소량
전사형 침입자: 무기
마법사형 침입자: 마력
정찰대: 잡화, 정보
```

역할:

- 침입 방어가 단순 피해 방지가 아니라 보상 루프가 된다.
- 무기와 마력 같은 전투/연구 재고 수급에 의미를 준다.
- 침입자 유형별 보상 차이를 만들 수 있다.

첫 구현에서는 침입자 유형이 하나뿐이어도 된다.

그 경우 보상은 다음처럼 단순화한다.

```text
방어 성공:
잡화 + 무기 소량

큰 피해 없이 방어 성공:
추가 마력 또는 식재료
```

### 4. 오펜스 원정 보상

오펜스는 원하는 재고를 목표로 노리는 수급 루프다.

운영일 납품이 안정 수급이고, 침입 방어가 수동 보상이라면, 오펜스는 플레이어가 직접 선택하는 능동 수급이다.

예시:

```text
농장 습격: 식재료
상단 습격: 잡화
무기고 습격: 무기
마탑 습격: 마력
유적 탐사: 희귀 전리품, 설계도
```

역할:

- 부족한 재고 계열을 목표로 선택할 수 있다.
- 월드맵 정찰/스캔 범위와 연결된다.
- 희귀 재고와 상위 합성 재료의 주요 수급처가 된다.

오펜스가 해금되기 전에는 운영일 납품, 내부 생산, 침입 방어 보상만으로 충분히 버틸 수 있어야 한다.

### 5. 이벤트와 방문 상인

이벤트는 재고 수급의 변동성을 만든다.

예시:

```text
방문 상인:
특정 재고를 비싸게 판매

몬스터 단골 선물:
선호도가 높은 종족이 작은 재고를 제공

암시장:
마력/무기/희귀 전리품을 비싸게 판매

사고 보상:
특정 이벤트 해결 후 재고 획득
```

이벤트 수급은 플레이어의 계획을 보조하는 장치로 둔다.

필수 재고를 이벤트 운에만 맡기지 않는다.

## 재고 계열별 수급 방향

각 재고 계열은 주 수급처가 달라야 한다.

```text
식재료:
운영일 납품, 버섯 재배실, 농장 원정

잡화:
운영일 납품, 잡동사니 공방, 상단 원정, 침입 방어 보상

무기:
운영일 납품 일부, 침입 방어 보상, 무기고 원정, 무기 정비소

마력:
마력 응축기, 마탑/유적 원정, 마법사 침입자 보상, 암시장
```

운영 감각:

- 식재료는 가장 안정적으로 구해야 한다.
- 잡화는 상점 매출과 대량 손님 운영을 지탱한다.
- 무기는 오크, 경비, 침입 방어와 연결된다.
- 마력은 연구, 합성, 뱀파이어/데몬 고객층과 연결된다.

## 재고 획득 처리

획득한 재고는 기본적으로 창고로 들어간다.

```text
재고 획득
-> 창고 용량 확인
-> 창고에 저장
-> 필요한 시설에서 보충 작업 생성
```

창고가 없거나 용량이 부족하면 다음 중 하나로 처리한다.

초기 구현:

```text
구매/획득 불가
```

후반 확장:

```text
일부만 보관
초과분 폐기
임시 보관으로 빠르게 소멸
```

첫 구현에서는 단순하게 `창고 용량이 부족하면 구매/획득 불가`로 둔다.

## 운영일 정산 연결

운영일 정산에는 재고와 물류 상태를 요약한다.

표시 항목:

- 재고 부족 시설
- 품절 발생 횟수
- 보충 실패 횟수
- 창고 재고 잔량
- 가장 많이 소비된 재고 계열
- 보충 작업을 가장 많이 한 직원

정산은 직접 처방하지 않는다.

좋은 예:

```text
품절 발생: 고기 식당 2회
부족 계열: 식재료
보충 실패: 3회
창고 식재료: 0
```

피해야 할 예:

```text
식재료를 더 구매하세요.
창고를 입구 근처에 지으세요.
```

## UI 피드백

재고 피드백은 짧고 명확해야 한다.

시설 위 표시:

```text
재고 낮음
재고 부족
품절
보충 중
```

시설 정보창:

```text
내부 재고
필요 재고 계열
보충 요청 기준
창고 재고 여부
최근 품절 횟수
```

직원 로그:

```text
식재료 보충 완료
창고 재고 없음
목적지 도달 불가
보충 작업 밀림
```

## 데이터 구조 방향

필드 후보:

```text
StockCategory
- 식재료
- 잡화
- 무기
- 마력

SaleItem
- 품목 ID
- 이름
- 가격
- 아이콘
- 재고 계열
- 구매 효과

FacilityStockSlot
- 필요 재고 계열
- 현재 내부 재고
- 최대 내부 재고
- 보충 요청 기준
- 1회 이용 소비량

WarehouseInventory
- 재고 계열
- 현재 보관량
- 최대 보관량

RefillJob
- 대상 시설
- 재고 계열
- 요청 수량
- 긴급도
- 예약 직원
```

현재 `SaleItem`과 `StockInfo`는 이미 판매 품목과 내부 재고를 표현하고 있다.

추가가 필요한 방향:

- `SaleItem`에 재고 계열 추가
- 창고 재고 데이터 추가
- 시설별 내부 재고 슬롯 일반화
- 보충 작업 데이터 추가

## 첫 구현 범위

첫 구현 목표:

- 재고 계열 4개: 식재료, 잡화, 무기, 마력
- 상점/식당/마력 시설은 내부 재고를 가진다.
- 창고는 재고 계열별 보관량을 가진다.
- 재고가 낮아지면 보충 작업이 생성된다.
- 직원은 보충 우선순위에 따라 창고에서 시설로 재고를 옮긴다.
- 재고 부족/품절은 손님 만족도와 직원 효율에 영향을 준다.
- 정산 창에 재고 부족 시설과 보충 실패를 표시한다.

첫 구현에서 생략해도 되는 것:

- 재고 품질
- 유통기한/부패
- 재고 무게 세부 계산
- 전문 창고
- 수리 자재
- 시설별 복잡한 제작 체인
- 품목별 물류 분리
- 자동 구매/자동 발주
