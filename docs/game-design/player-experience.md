# 플레이어 경험

## 플레이어 경험 원칙

모든 시스템은 따로 존재하는 메뉴가 아니라 하나의 원인과 결과 사슬로 이어져야 한다.

핵심 경험 흐름:

```text
손님이 온다
-> 돈을 쓴다
-> 종족 욕구/불만이 생긴다
-> 시설 배치와 운영 문제가 드러난다
-> 침입이 던전의 약점을 시험한다
-> 연구와 합성으로 보완한다
-> 강해진 몬스터와 시설로 오펜스를 보낸다
-> 보상으로 다시 던전을 바꾼다
```

플레이어가 좋은 경험을 하려면 다음 흐름이 반복되어야 한다.

```text
문제 발생
-> 화면 위 감정/발동 알림
-> 클릭 시 상세 기록 확인
-> 도감 정보 누적
-> 플레이어가 해결 방향을 추론
-> 다음 선택 실행
```

예시:

```text
슬라임 만족도 낮음
-> 화면: 불만 이모티콘 풍선
-> 캐릭터 로그: 선호 시설 없음, 화염 시설 근처 동선
-> 도감: 슬라임 선호 - 휴식방(습기)
-> 플레이어가 배치/시설/동선을 조정
```

```text
침입자가 너무 빨리 사장방 도달
-> 화면: 침입 결과 알림
-> 상세 기록: 침입자 도달 위치, 발동한 함정, 피해 시설
-> 침략 도감: 성향 - 사장방 돌파
-> 플레이어가 다음 배치와 합성 목표를 스스로 정함
```

### 시스템 역할

각 시스템은 다음 역할을 가져야 한다.

- 경영: 돈 버는 파트가 아니라 던전의 체질을 만드는 시스템이다.
- 몬스터 고객: 던전이 어떤 고객층을 대상으로 하는지 정하게 만든다.
- 배치/동선: 손님 만족도와 침입 방어를 동시에 바꾸는 핵심 선택이다.
- 침입: 현재 던전의 약점을 보여주는 시험이다.
- 연구/합성: 침입과 운영에서 드러난 문제를 해결하는 성장 수단이다.
- 오펜스: 내 던전에서 키운 몬스터와 시설을 외부로 확장하는 장기 루프다.
- 런 변수/이벤트: 매 런의 운영 우선순위와 빌드 방향을 흔드는 변주 장치다.

### 읽히는 피드백

플레이어는 실패했을 때 억울하기보다 다음 행동을 떠올릴 수 있어야 한다.

필수 피드백:

- 손님이 왜 불만족했는지 표시한다.
- 직원이 왜 일을 못 했는지 표시한다.
- 침입 중 어떤 함정, 직원, 시설 효과가 작동했는지 표시한다.
- 어떤 시설이 어떤 이유로 피해를 입었는지 표시한다.

손님/직원 개인 단위의 원인은 상시 UI에 모두 노출하지 않는다. 림월드의 캐릭터 로그처럼, 캐릭터를 클릭했을 때 최근 행동과 감정 변화 로그를 확인하게 한다.

화면 위에서는 이모티콘 풍선으로 현재 감정만 직관적으로 보여준다.

```text
기쁨: 원하는 시설 이용, 만족도 상승
불만: 대기 길음, 선호 시설 없음, 가격 불만
당황: 길 막힘, 목적지 사라짐, 재고 없음
분노: 불만 누적, 사고 직전
피로: 직원 휴식 필요, 근무 불가
```

캐릭터 로그는 짧은 원인 태그 중심으로 기록한다.

예시:

```text
대기 길음
선호 시설 없음
재고 부족
길 막힘
혼잡함
방어 시설 부족
사장방까지 최단 동선
```

### 공통 조작 원칙

던전의 모든 작업은 기본적으로 자동으로 진행된다.

플레이어는 캐릭터를 직접 조종하지 않고, 캐릭터를 선택한 상태에서 대상에 우선 지정 명령을 내린다.

예시:

```text
직원 선택 + 파손 시설 우클릭: 해당 시설 수리 우선
직원 선택 + 창고 우클릭: 재고 보충 우선
직원 선택 + 연구시설 우클릭: 설계도 분석 우선
경비 선택 + 반란 몬스터 우클릭: 제압 우선
직원 선택 + 불만 직원 우클릭: 진정 시도 우선
```

우선 지정은 즉시 순간이동이나 수동 조작이 아니다.

캐릭터는 기존 AI 규칙에 따라 이동하고, 작업 가능 여부와 상태를 확인한 뒤 행동한다.

우선 지정이 실패하면 캐릭터 로그에 이유를 남긴다.

예시:

```text
길 막힘
작업 불가
피로 한계
필요 능력 부족
대상 사라짐
```

### 이벤트 결과 알림

침입, 사고, 운영일 정산, 연구 완료, 합성 완료, 상점 리셋 같은 이벤트 결과는 화면 오른쪽 또는 우하단에 작은 알림 버튼으로 표시한다.

알림은 플레이를 강제로 멈추는 모달이 아니라, 림월드처럼 플레이어가 필요할 때 눌러서 상세 내용을 확인하는 방식으로 둔다.

런 변수와 운영 이벤트의 상세 설계는 [런 변수와 이벤트](run-variables-and-events.md) 문서에서 관리한다.

알림 버튼 구성:

```text
아이콘
이벤트 이름
중요도 색상
발생 시각 또는 방금 전 표시
```

알림을 누르면 상세 패널을 연다.

상세 패널은 관측된 사실만 보여준다.

예시:

```text
이벤트: 침입 종료
참여 침입자: 모험가 3명
발동한 시설: 끈끈이 함정, 가시 함정
교전한 직원: 오크 경비원
피해 시설: 저가 음식점
획득 정보: 성향 - 사장방 돌파
```

상세 패널은 `무엇을 지어라`, `어떤 연구를 하라`, `어디에 배치하라` 같은 직접 처방을 하지 않는다.

여러 이벤트가 겹치면 알림을 오른쪽에 쌓고, 오래된 알림은 이벤트 로그에서 다시 볼 수 있게 한다.

### 자동 전투 피드백

자동 전투의 재미는 플레이어가 직접 조작하는 데서 나오지 않고, 전투 전에 준비한 던전 구조와 시설 조합이 실제로 작동하는 것을 보는 데서 나온다.

초기 구현에서는 침입 경로 리플레이나 전체 경로 하이라이트를 필수로 두지 않는다. 이 기능은 구현 비용이 크므로, 첫 버전에서는 다음 두 가지에 집중한다.

1. 전투 중 발동 표시
2. 전투 결과 요약

전투 중 발동 표시는 짧고 명확해야 한다.

예시:

```text
끈끈이 함정 발동: 침입자 이동 속도 감소
오크 직원 합류: 침입자와 교전 시작
훈련장 시너지: 오크 공격력 증가
뱀파이어 공포: 침입자 명중률 감소
가시 함정 발동: 침입자 피해
```

전투 결과 요약은 플레이어가 다음 배치를 스스로 추론할 수 있게 해야 한다.

결과 화면은 추천 대응을 직접 제시하지 않는다. 전투 직후 화면은 `무슨 일이 일어났는가`를 보여주는 기록이어야 하고, `무엇을 지어라/무엇을 연구해라` 같은 처방은 피한다.

예시:

```text
가장 많은 피해를 준 시설: 가시 함정
가장 오래 지연시킨 시설: 끈끈이 함정
결정적 방어: 오크 직원의 교전
피해를 입은 시설: 저가 음식점
```

이 방식은 전체 리플레이 없이도 `내 함정과 직원 시너지가 침입자를 막았다`는 감각을 줄 수 있다.

### 도감과 정보 해금

대응 힌트가 필요하다면 전투 결과 화면이 아니라 도감/정보 메뉴에서 제공한다.

도감은 정답을 직접 알려주는 공략 메뉴가 아니라, 플레이어가 관찰한 정보를 정리해주는 시스템이다.

가능한 도감:

- 몬스터 도감: 종족별 선호 시설, 기피 시설, 사고 위험, 시너지
- 침략 도감: 침입자 특징, 자주 노리는 대상, 약했던 방해 효과
- 시설 도감: 시설 효과, 발동 조건, 시너지 대상, 합성 조합 정보

정보 해금 예시:

```text
약점: 감속
약점: 근접 교전
성향: 시설 파손 우선
주의: 사장방 돌파
저항: 공포 효과
```

이 정보는 대응을 강요하지 않고, 플레이어가 `그럼 다음에는 어떤 시설과 배치를 써볼까`를 생각하게 만드는 단서로만 작동한다.

### 복잡도 관리

시스템은 한 번에 모두 노출하지 않는다. 플레이어가 먼저 문제를 경험하고, 그 문제를 해결할 필요를 느낀 뒤 새 시스템을 열어야 한다.

원칙:

- 새 시스템은 이전 시스템의 문제를 해결하기 위해 등장한다.
- 하나의 문제에는 최소 2개 이상의 해결 방법을 둔다.
- 처음에는 수치보다 명확한 사례를 보여준다.
- 자동 전투는 관측 가능한 발동, 피해, 지연, 교전 기록을 남긴다.
- UI는 모든 내부 수치를 공개하기보다 짧은 원인 태그와 도감 단서를 보여준다.
- 직접 처방은 피하고, 플레이어가 관측 정보를 바탕으로 해결 방향을 추론하게 한다.

이 기준을 만족하면 시스템이 많아져도 플레이어는 `외워야 할 규칙`이 아니라 `내 던전에서 일어난 사건`으로 받아들일 가능성이 높다.
