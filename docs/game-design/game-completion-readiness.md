# DungeonStory 완전한 게임 완료 기준

> 기준일: 2026-07-17

## 제품 정의

DungeonStory가 하나의 완전한 게임으로 동작한다는 것은 아래 흐름을 플레이어가 QA 명령 없이 처음부터 끝까지 반복할 수 있다는 뜻이다.

```text
새 게임/이어하기
-> 사장 선택
-> 벽, 문, 시설로 방 구성
-> 손님, 직원, 재고, 기분, 경제 운영
-> 연구, 합성, 진화, 원정으로 빌드 선택
-> 침입에 대응하고 최종 침공 해결
-> 승리/패배 결과와 계승 보상
-> 달라진 조건으로 다음 런
```

## 현재 판정

| 영역 | 상태 | 판정 |
|---|---|---|
| 건설, 방, 시설 능력 | 성숙 | 실제 배치와 방 판정, 환경, 능력 모듈이 연결됨 |
| 캐릭터, AI, 업무, 욕구, 기분 | 성숙 | 운영 결과가 행동과 기분에 반영됨 |
| 경제, 정산, 파산 대응 | 성숙 | 유지비, 급여, 체납, 긴급 대출까지 닫힘 |
| 런 시작, 최종 침공, 결과, 다음 런 | 성숙 | 공개 UI와 저장 복원으로 한 바퀴가 이어짐 |
| 연구 | 연결 완료 | 네 모듈식 건축 경로와 공개 건축 UI가 연결됨 |
| 합성, 진화 | 연결 완료 | 세 전략에 합성 3개·진화 2개씩 연결되고 공개 UI와 저장 복원을 검증함 |
| 원정, 메타 | 연결 완료 | 세 전략 보상과 영구 프로필, 다음 런 보존까지 연결됨 |
| 콘텐츠와 밸런스 | 자동 검증 완료 | 세 전략 자연 승리와 무방비 패배 완료, 인간 페이싱 평가는 남음 |
| 오디오, 설정, 접근성, 제품 셸 | 완료 | 타이틀, 저장, 설정, 브랜딩, 제작 음향과 채널 분리까지 검증됨 |
| 장시간 안정성, 배포 빌드 | 완료 | soak, 성능, 해상도, 저장 호환, Development/Release 실행 검증 완료 |

## 반드시 닫아야 할 아홉 가지

### 1. 완전한 런 순환

- 새 게임, 이어하기, 승리, 패배, 결과, 다음 런이 모두 공개 UI로 이어진다.
- 저장 후 재실행해도 같은 운영 상태와 진행 선택을 복원한다.
- 막힌 동선, 자금 고갈, 직원 부재가 영구 진행 불능을 만들지 않는다.

현재 이 축은 구현됐으며 전체 회귀를 계속 유지해야 한다.

### 2. 최소 세 가지 유효한 빌드

1. 상업/물류: 손님 회전, 재고 효율, 현금 흐름으로 성장한다.
2. 요새/방어: 경비와 함정, 수리 효율로 침입을 버틴다.
3. 비전/연구: 연구 속도, 마나 시설, 고급 진화로 후반 가치를 만든다.

각 빌드는 연구 해금, 합성 두 단계 이상, 진화 결과 두 개 이상, 원정 보상, 메타 보너스, 약점과 대응 침입 패턴을 가져야 한다. 어느 하나도 다른 빌드의 필수 시설을 전부 요구해서는 안 된다.

### 3. 선택이 이어지는 진행망

- 연구 결과가 실제 건축 선택을 연다.
- 합성이 현재 방과 시설 조합을 다음 운영 단계로 바꾼다.
- 진화는 방 환경과 운영 기록에 따라 다른 결과를 낸다.
- 원정은 부족한 자원이나 빌드 핵심 보상을 얻는 위험한 우회로가 된다.
- 메타 진행은 다음 런의 선택지를 넓히되 첫날을 자동 승리로 만들지 않는다.

연구·합성·진화·원정·메타 보상이 세 전략 방향으로 연결됐고 저장 복원과 다음 런에서도 유지된다.

### 4. 읽을 수 있는 위기와 대응

- 침입 패턴마다 목표, 이동, 공격 방식과 대응 시설이 다르다.
- 위기가 오기 전에 경고와 준비 시간이 있다.
- 실패 원인이 결과, 알림, 시설/캐릭터 정보에서 짧고 명확하게 보인다.
- 같은 방 배치를 반복하는 것보다 구조를 바꾸는 편이 실제 이득이어야 한다.

### 5. 실패와 회복

- 적자, 재고 부족, 파손, 낮은 기분, 인력 부족에 각각 회복 수단이 있다.
- 회복에는 비용과 선택이 필요하지만 즉시 런을 버리게 만들지는 않는다.
- 패배는 사장 사망이나 명시된 최종 조건으로만 확정한다.
- 승패 직전 저장/불러오기가 결과를 복제하거나 보상을 중복 지급하지 않는다.

### 6. 플레이어가 이해하는 UI와 피드백

- 현재 목표, 자금 흐름, 재고 부족, 직원 문제, 방 환경, 침입 위험이 한눈에 읽힌다.
- UI 클릭이 월드 클릭으로 새지 않고, 캐릭터와 건물 정보는 하나만 열린다.
- 건축은 설치 가능 영역, 고스트, 점유 칸과 차단 이유를 보여준다.
- 중요한 완료, 해금, 파손, 침입, 승패에는 시각/음향 피드백이 있다.
- LLM 문장은 표현만 풍부하게 하고 진행과 판정을 막지 않는다.

### 7. 한 런을 채우는 콘텐츠 밀도

- Day 1~3, 4~6, 7~9, 최종 침공에 서로 다른 결정이 나타난다.
- 손님과 직원이 이름만 다른 복제가 아니라 종족, 특성, 선호, 사고 위험으로 달라진다.
- 같은 빌드에서도 최소 두 번은 분기 선택이 발생한다.
- 3회 연속 런에서 동일한 시설 순서와 동일한 사건만 반복되지 않는다.

현재 시설 수와 세 전략의 진행망은 첫 출시 후보 범위에 충분하다. 신규 시설 양산보다 실제 첫 플레이의 선택 간격과 반복 체감을 먼저 조정한다.

### 8. 제품 셸과 감각

- 타이틀/메인 메뉴, 새 게임, 이어하기, 설정, 종료가 있다.
- BGM, UI, 건설, 작업, 침입, 파손, 결과 음향과 볼륨 분리가 있다.
- 해상도, 창 모드, 음량, 키 재지정, 텍스트 가독성, 색상 외 상태 표기가 있다.
- 로딩, 장면 전환, 일시정지, 저장 중 표시가 자연스럽다.
- 회사명, 제품명, 버전, 아이콘, 애플리케이션 ID가 출시 값이다.

현재 타이틀/새 게임/이어하기/저장/설정/종료, 접근성, 최종 엠블럼, Windows 실행 파일 아이콘, 제작 음향 6개와 채널별 음량이 공개 UI와 Release 빌드에서 검증됐다.

### 9. 출시 검증

- 세 빌드로 각각 최소 한 번 자연 승리하고 의도적인 패배도 검증한다.
- 연속 여러 런 soak에서 AI 정지, 예약 누수, 길찾기 교착, 저장 팽창이 없다.
- 지원 해상도와 창/전체 화면에서 UI 잘림과 겹침이 없다.
- Development/Release 빌드가 새 프로필과 기존 저장 모두에서 실행된다.
- 최종 기준은 주요 PlayMode 회귀 통과, 캡처 직접 확인, Console Error/Warning 0건이다.

세 전략의 자연 승리와 의도적 패배, 5배속 soak, 저장 왕복, 해상도 행렬, 공개 UI 회귀, Development/Release 빌드, 25초 독립 실행 스모크까지 완료했다. 최신 자동화 범위의 Console과 빌드 Error/Warning은 `0/0`이다.

## 실행 순서

1. 새 프로필로 실제 사람이 QA 도구 없이 첫 런을 플레이한다.
2. 목표 이해, 대기 시간, 클릭 부담, 보상 간격, 최종 침공 이해, 재도전 의향을 기록한다.
3. 관찰된 문제만 우선순위화해 수치, 설명, 피드백을 조정한다.
4. 수정 후 같은 프로필 조건으로 다시 플레이해 개선 여부를 비교한다.

자동화 기준으로는 배포 가능한 출시 후보다. “처음부터 끝까지 재미와 선택이 있는 게임”이라는 최종 판정은 인간 첫 런에서 위 여섯 항목을 확인한 뒤 내린다.
